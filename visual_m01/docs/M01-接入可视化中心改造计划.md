# M01 接入可视化中心改造计划

## 1. 文档目的

本文档用于梳理 `visual_m01` 从“独立立体几何教学应用”改造为“EduMind 可视化中心模板运行时”的实施方案。

本次改造目标不是继续增强 `visual_m01` 自身产品能力，而是：

- 裁掉与主站重复的功能
- 保留 M01 的核心 3D 教学与编辑能力
- 对齐可视化中心当前的模板实例、snapshot、继续编辑和工作台职责
- 参考已上线模板 M05 的接入边界，完成上线适配

---

## 2. 当前判断结论

### 2.1 `visual_m01` 当前状态

`visual_m01` 当前并不是轻量模板页，而是一套完整独立应用，包含：

- 登录页
- 工作台页
- 编辑器页
- 作品管理
- 模板推荐
- 预置作品库
- 几何引擎
- scene snapshot 存取

从现状看，它更像：

> “立体几何产品线工作台 + 题目作品库 + 3D 编辑器引擎”

而不是：

> “主站模板系统中的一个可嵌入模板运行时”

### 2.2 `visual_m01` 的优势

它已经具备主站模板运行时最重要的底层能力：

- 完整 3D 编辑器
- 实体 / 命令 / 工具 / 渲染器体系
- `getSnapshot()`
- `loadSnapshot()`
- `initEditorWithSnapshot()`
- 丰富的 M01 预置作品元数据与 scene 数据

所以它不是从零改造，而是：

> 底座已经很完整，缺的是“宿主适配层”和“职责收口”

### 2.3 当前最大的接入问题

当前 `visual_m01` 和主站可视化中心存在明显职责重叠：

`visual_m01` 当前负责：

- 用户登录
- 内部工作台
- 内部项目管理
- 内部作品持久化
- 内部模板推荐

而主站可视化中心已经负责：

- 用户体系
- 教学模板库
- 我的作品
- 模板实例创建与复制
- snapshot 持久化
- 继续编辑

所以改造重点不是“加功能”，而是“做减法 + 建协议”。

---

## 3. 改造目标

将 `visual_m01` 改造成以下形态：

### 3.1 目标形态

`visual_m01` 最终应作为：

> “M01 立体几何展示台模板运行时”

由主站模板工作台通过 iframe 承载，并由主站负责：

- 登录
- 模板实例
- 作品列表
- 复制
- 重命名
- snapshot 保存与恢复触发
- 继续编辑入口

### 3.2 运行时职责边界

`visual_m01` 保留：

- 3D 编辑器
- 立体几何实体 / 场景 / 计算引擎
- M01 预置 scene 数据
- 内部快照读写能力
- 缩略图捕获能力（可选保留，后续用于作品封面）

`visual_m01` 移除或下沉：

- 登录体系
- 注册页
- 内部 workspace
- 内部项目列表
- 内部项目 CRUD
- 内部“我的作品”
- 内部模板选择工作台
- 与主站重复的持久化入口

---

## 4. 参考对象：M05 的接入原则

M05 当前已在可视化中心上线，可以作为职责边界参考。

M05 给我们的启发不是“功能长得一样”，而是：

### 4.1 主站负责

- 模板展示
- 模板实例创建
- snapshot 持久化
- 继续编辑
- 作品归档
- 外层工作台入口与导航

### 4.2 模板运行时负责

- 自己的核心交互体验
- 自己的内部场景状态
- 向宿主暴露标准快照接口

### 4.3 M01 改造也应遵守同样原则

换句话说，M01 不再是“自己的一套产品”，而是主站里的一个“运行时能力模块”。

---

## 5. 改造总原则

### 原则 1：主站唯一用户入口

M01 不再自行登录，不再跳转登录页，不再持有自己的登录态。

### 原则 2：主站唯一作品入口

M01 不再持有自己的“作品工作台”和“我的作品”页面。

### 原则 3：主站唯一持久化入口

M01 不直接主导项目保存，而是由主站通过标准 bridge 拉取 snapshot，并写入后端模板实例。

### 原则 4：M01 专注编辑器能力

M01 的核心价值在于：

- 几何引擎
- 交互编辑
- 场景表达
- scene 数据模板化

所以改造后应尽量让它只做这部分。

### 6.4 Store 级功能

- 独立 AuthStore 的页面级控制
- ProjectStore 中与“内置作品系统”强绑定的行为

## 7. 需要保留的核心能力

### 7.1 编辑器内核

## 8. 必须新增的适配能力

这是本次改造最关键的一层。

### 8.1 新增“嵌入模式”

建议为 M01 增加独立运行模式，例如：

- `?embed=1`
或
- `/embed`

在嵌入模式下：

- 不显示内部 workspace
- 不显示内部登录
- 不显示与主站重复的导航
- 页面直接进入编辑器

### 8.2 新增模板桥接协议

主站当前模板工作台使用的 bridge 协议为：

- namespace: `edumind.templateBridge`
- request type:
  - `getSnapshot`
  - `loadSnapshot`
  - `validateSnapshot`
- response type:
  - `response`

M01 需要实现这一套 `window.postMessage` 协议。

### 8.3 Bridge 最低实现要求

#### `getSnapshot`

返回当前编辑器快照，来源应直接对接：

- `useEntityStore.getState().getSnapshot()`

#### `loadSnapshot`

接收主站传入的 snapshot，并执行：

- `initEditorWithSnapshot(snapshot)`
或
- `useEntityStore.getState().loadSnapshot(snapshot)`

#### `validateSnapshot`

校验 snapshot 是否满足 M01 所需结构：

- `entities`
- `nextId`
- `activeGeometryId`

返回：

- `success: true/false`
- 可选错误信息

## 9. Snapshot 对齐方案

### 9.1 当前主站实例结构

主站模板实例会保存：

- `snapshot`
- `snapshot_schema_version`

### 9.2 当前 M01 内部快照结构

M01 当前快照结构核心为：

```ts
{
  entities: Record<string, Entity>,
  nextId: number,
  activeGeometryId: string | null
}
```

### 9.3 建议对齐方式

建议不要直接上传裸快照，而是在运行时中包装成统一 envelope：

```ts
{
  envelope: {
    templateKey: "m01",
    snapshotSchemaVersion: 1,
    runtimeVersion: "m01-runtime-v1"
  },
  scene: {
    entities: ...,
    nextId: ...,
    activeGeometryId: ...
  },
  ui: {
    // 可选，后续再加
  }
}
```

### 9.4 为什么推荐 envelope

优点：

- 方便未来版本升级
- 方便区分不同模板
- 便于主站统一识别 snapshot schema version

## 11. UI 裁剪建议

### 11.1 嵌入模式应保留

- 3D 视图
- 工具栏
- Inspector / 参数面板
- 场景操作
- 导入导出（可视情况保留）

### 11.2 嵌入模式应隐藏

- 顶部品牌导航
- 内部工作台入口
- 登录相关按钮
- 与主站重复的保存/作品列表入口

### 11.3 保存按钮处理

建议：

- 在嵌入模式中，不再由模板自己决定最终持久化
- 可保留“保存到主站”类按钮，但按钮动作应改成：
  - 向宿主发送“请保存”信号
  - 或由宿主主动拉取 snapshot

---

## 12. API 与数据层改造建议

### 12.1 当前内部项目 API

当前 `visual_m01` 会调用：

- `getProject`
- `createProject`
- `updateProject`
- `deleteProject`

这些 API 属于“模板自己管理项目”的体系。

### 12.2 改造建议

嵌入模式下应停用或绕开这些 API：

- 不主动创建内部项目
- 不主动读取内部项目
- 不主动更新内部项目

### 12.3 预置 scene 的保留方式

预置作品数据应保留，但用途改变：

- 从“内部作品库数据”
- 改为“模板内部可选预置场景数据”

后续可以通过以下方式接主站：

- 主站为 M01 增加“预置场景”入口
- 选择预置场景后，将 scene snapshot 写入模板实例

## 13. 建议实施阶段

### 阶段 1：裁掉重复产品壳

目标：

- 保留编辑器
- 隔离 workspace / auth / project CRUD

交付物：

- 嵌入模式基础可运行

### 阶段 2：补 bridge 协议

目标：

- 支持 `getSnapshot`
- 支持 `loadSnapshot`
- 支持 `validateSnapshot`

交付物：

- 主站模板工作台可打开 M01
- 主站可保存 / 恢复 M01 场景

### 阶段 3：对齐主站实例体验

目标：

- 继续编辑可恢复
- 新建副本可正常进入
- 开发中 / 已上线状态可联调

交付物：

- 可视化中心正式接入 M01

### 阶段 4：增强体验

可选项：

- 自动缩略图
- 预置场景注入
- “从真题创建 M01 场景”

---

## 14. 风险点

### 风险 1：双持久化体系冲突

如果不及时裁掉内部 project API，后续可能出现：

- 主站保存一份
- 模板内部再保存一份

导致状态混乱。

### 风险 2：嵌入模式 UI 仍然过重

如果只是简单 iframe 进来不做裁剪，用户会看到：

- 主站外壳
- 模板内部外壳

体验会非常割裂。

### 风险 3：snapshot schema 未来扩展困难

如果直接上传裸 scene 数据，不做 envelope，后续升级版本时兼容成本会升高。

---

## 15. 建议最终落地形态

对 M01 最合适的最终形态不是：

> 一整套独立产品继续嵌到主站里

而是：

> 一个以 3D 编辑器为核心、支持 snapshot 协议、由主站统一托管实例与作品的模板运行时

---

## 16. 下一步执行建议

建议后续按以下顺序实际改造：

1. 新增嵌入模式路由或 `embed=1` 模式
2. 在嵌入模式中隐藏登录 / workspace / 项目管理 UI
3. 实现 `edumind.templateBridge`
4. 将 `getSnapshot/loadSnapshot` 对接到 EntityStore
5. 为 snapshot 增加 envelope
6. 主站联调模板实例保存与恢复
7. 再决定是否把 M01 内置作品库作为“预置场景库”继续暴露

---

## 17. 文档结论

`visual_m01` 当前最适合作为：

- “M01 立体几何展示台运行时内核”

而不是：

- “继续独立维护一套登录 + 工作台 + 项目系统”

本次改造核心方向应为：

> 做减法，补协议，对齐主站模板实例体系。
