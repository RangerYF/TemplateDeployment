# P-01 受力分析器 — 模块分析卡片

## 1. 用户场景 & 产品价值

### 目标用户
高中/初中物理老师，不要求有任何编程基础。核心用户画像是高中物理教师，需要在课堂上讲解不同模型下的受力分析（水平面、斜面、悬挂、连接体、圆周运动、传送带等）。

### 解决什么教学痛点
- **受力分析不直观**：黑板上画力的示意图是静态的，无法动态调整，学生搞不清不同约束条件下物体受哪些力
- **正交分解难以演示**：坐标系选择、分力箭头、分量数值需要逐步展示，板书无法动态呈现分解过程
- **条件变化影响不直观**：改变摩擦系数、斜面角度、外力方向后，各力如何变化，需要重新画图计算
- **模型种类多、组合复杂**：13 类受力模型 × 多种约束条件组合，人工穷举费时易错
- **临界条件分析困难**：恰好滑动、恰好脱轨等临界点，需要精确数值支撑教学

### 核心产品价值
**一句话**：老师选一个受力模型、勾选约束条件，系统自动完成受力分析并以力的示意图实时展示，支持正交分解动画和合力计算，覆盖高中力学全部典型受力场景。

## 2. 用户实际使用方式

### 教师典型操作流程

| 场景 | 操作流程 |
|------|---------|
| **课前准备** | 打开浏览器 → 选择受力模型（如"斜面"）→ 设置约束条件（粗糙/静止/无外力）→ 确认受力结果正确 → 可选设置参数（质量、角度、摩擦因数）|
| **课中演示** | 投影仪打开 → 展示默认受力图 → 逐步切换条件（光滑→粗糙、静止→加速）让学生观察力的变化 → 点击"正交分解"演示分解动画 → 显示合力 |
| **讲题** | 选择题目对应的模型和条件 → 动态调节参数（如改变斜面角度）→ 观察各力数值实时变化 → 展示临界条件（如 tan θ = μ）|

### 怎么用才是"好用"的
- **即选即用**：选模型 + 勾条件，立刻出受力图，不需要自己画
- **实时响应**：修改任何约束条件或参数，受力图立即更新，无需点击"确认"
- **条件穷举**：系统内置全部约束条件组合（如 FM-001-A 到 FM-001-H），不需要老师手动推导
- **投影清晰**：力的箭头、标注文字在 1080p 投影仪上清晰可辨
- **矛盾检测**：选择了互相矛盾的条件时，系统能正确提示

### 交互频率和深度
- **频率**：每节力学课使用 2-5 次，每次 3-10 分钟
- **深度**：中等 — 主要是模型选择 + 条件勾选 + 参数微调，不涉及自由构建场景

## 3. 核心功能点

### 功能清单（按重要性排序）

| 优先级 | 功能 | 类型 |
|--------|------|------|
| P0 | 受力模型选择（13 种模型：水平面/斜面/悬挂/连接体/圆周/弹簧/传送带/半球/凹槽/浮力/滑轮/杠杆/杆） | 参数驱动 |
| P0 | 约束条件设置（接触面性质/运动状态/外力有无及方向） | 参数驱动 |
| P0 | 自动受力分析与力的示意图实时展示 | 计算 + 渲染 |
| P0 | 参数调节（质量/摩擦因数/角度/外力等）实时更新 | 参数驱动 |
| P0 | 各力大小数值列表（G/N/f/F/T 等） | 计算 |
| P1 | 正交分解动画（坐标系选择 + 分量箭头 + 数值） | 动画驱动 |
| P1 | 合力显示与计算 | 计算 + 渲染 |
| P1 | 条件矛盾检测与提示 | 逻辑 |
| P2 | 物体类型选择（物块/小球/滑环/木板）与数量组合 | 参数驱动 |
| P2 | 导出图片功能 | 工具 |

### 哪些功能需要实体管理
**不需要传统的实体系统**。与立体几何展示台（M-01）不同，受力分析器中的"物体"和"力"不需要自由创建/删除/选中等编辑器操作。其核心交互模式是**"选择预设模型 + 设置约束条件 → 系统自动分析出力"**，即：
- 物体由模型预设决定（不是用户自由创建）
- 力由约束条件和物理规律自动推导（不是用户自由添加）
- 用户的操作是"选择"和"调参"，不是"构建"

但需要一个**模型注册表/配置系统**来管理 13 种模型及其约束条件组合（数据文档定义了约 60+ 种约束条件组合）。

### 哪些功能需要 undo/redo
**不需要 undo/redo**。因为用户操作是"选择模型→勾选条件→调参数"，每次操作都是独立的状态设定，不存在需要撤销的创建/编辑链。

### 哪些功能是参数驱动的（调参数 → 画面实时变化）
- 质量 m → 影响 G、N、f 等所有力的大小
- 斜面角度 θ → 影响 N、f 和正交分解的分量
- 摩擦因数 μ → 影响摩擦力大小
- 外力 F 大小和方向角 → 影响整体受力平衡
- 弹簧劲度系数 k → 影响弹力大小
- 圆周运动角速度 ω / 半径 r → 影响向心力
- 传送带速度 v_belt → 影响摩擦力方向

### 哪些功能是模拟/动画驱动的
- **正交分解动画**：力的箭头逐步分解为分量的动画过程（一次性动画，非持续模拟）
- 注意：受力分析器本身不涉及运动模拟（运动模拟是 P-02 的职责），它是**静态分析 + 参数驱动实时更新**的模式

## 4. 核心依赖识别

### 技术依赖

#### 渲染方式：2D Canvas 或 SVG
- 受力分析器的核心可视化是 **2D 力的示意图**（力的箭头、物体轮廓、斜面/绳索等几何结构）
- 不需要 3D 渲染 — PRD 中的所有受力模型（水平面、斜面、悬挂、圆周等）都是经典的 2D 力学图
- **推荐 Canvas**：力的箭头需要频繁重绘（参数变化时实时更新）；正交分解动画需要帧级控制；Canvas 在处理大量矢量图形的实时重绘时性能优于 SVG
- SVG 也可行（力的数量有限，通常 3-6 个力），适合需要 DOM 事件绑定的场景

#### 数学/物理计算需求
- **中等复杂度**：核心是牛顿力学公式计算（G=mg、f=μN、F=ma、向心力等）
- 所有公式在数据文档中已穷举（10 条核心公式 + 各模型专用公式）
- 不需要数值积分或微分方程求解（与 P-02 运动模拟器不同）
- 需要三角函数计算（正交分解涉及 sin/cos）
- 需要联立方程组求解能力（连接体整体法/隔离法、双绳悬挂的平衡方程）
- **Math.js** 足以覆盖，或自研轻量计算模块即可

#### 公式渲染需求
- **需要**：各力数值列表、平衡方程、正交分解公式需要以数学公式形式展示
- 推荐 **KaTeX**（PRD 技术栈已选定）

### 交互依赖

#### 交互模式：**参数驱动型**（非编辑器型）
这是本模块最关键的架构判断。P-01 的交互模式与 M-01（立体几何展示台）截然不同：

| 维度 | M-01 立体几何展示台 | P-01 受力分析器 |
|------|-------------------|----------------|
| 交互模式 | 编辑器型（创建/选中/编辑/删除实体） | 参数驱动型（选模型→勾条件→调参数） |
| 用户构建 | 自由组合几何体、标注、坐标系 | 不构建，只选择预设模型 |
| 状态来源 | 用户操作积累 | 模型配置 + 约束条件 + 参数值 |
| 核心数据流 | 命令 → 实体Store → 渲染 | 配置选择 → 物理求解器 → 渲染 |

#### 关键交互
- **点选/切换**：模型选择按钮、约束条件勾选框
- **滑块 + 数值输入框**：参数调节（质量/角度/摩擦因数等）
- **无拖拽需求**：不需要拖拽物体或力箭头（PRD 未提及拖拽交互）
- **正交分解按钮**：触发分解动画
- **合力显示按钮**：切换合力箭头的显示/隐藏

#### 是否需要实体系统
**不需要**。理由见上文"哪些功能需要实体管理"部分。

#### 是否需要命令系统（undo/redo）
**不需要**。理由见上文。

### 竞品 1：PhET — Forces and Motion: Basics / Forces in 1 Dimension

| 维度 | 分析 |
|------|------|

| **核心能力** | 施加力 → 观察物体运动，显示自由体受力图（FBD）、力/位置/速度/加速度 vs 时间图 |
| **技术实现** | HTML5 Canvas，PhET 自研框架（Scenery 渲染引擎 + Axon 响应式模型），开源 |

### 竞品总结

| 竞品 | 形态 | 模型覆盖 | 自动分析 | 约束条件系统 | 正交分解 | 易用性 |
|------|------|---------|---------|-------------|---------|-------|
| PhET | Web | 低（仅水平面） | 无（用户手动施力） | 无 | 无 | 极高 |

## 6. 初步技术判断

### 推荐的渲染技术
**2D Canvas**（首选）或 **SVG**（备选）

理由：
- 全部受力模型都是 2D 力学图，不需要 3D
- Canvas 适合频繁重绘（参数变化时实时更新力的箭头）
- Canvas 适合正交分解动画的帧级控制
- SVG 在力的数量较少时也完全可行（每个模型通常 3-6 个力），且 SVG 的 DOM 特性有利于事件绑定和可访问性
- 如果与 P-02（运动模拟器）融合，Canvas 更合适（运动轨迹绘制需要 Canvas）

### 推荐的架构模式
**参数驱动型响应式架构**（非编辑器型）

```
┌─────────────┐    ┌──────────────────┐    ┌──────────────┐
│  模型注册表   │    │  约束条件引擎     │    │  物理求解器   │
│ (13种模型     │───▶│ (条件组合校验     │───▶│ (力学公式    │
│  参数/条件    │    │  矛盾检测        │    │  联立求解)   │
│  配置数据)    │    │  映射到受力集)    │    │              │
└─────────────┘    └──────────────────┘    └──────┬───────┘
                                                    │
                                                    ▼
┌─────────────┐    ┌──────────────────┐    ┌──────────────┐
│  UI 控件层   │    │  正交分解引擎     │    │  2D 渲染器   │
│ (模型选择    │◀───│ (坐标系选择      │◀───│ (力箭头/     │
│  条件勾选    │    │  分量计算        │    │  物体/场景   │
│  参数滑块)   │    │  分解动画)       │    │  绘制)       │
└─────────────┘    └──────────────────┘    └──────────────┘
```

核心组件：
1. **模型注册表（Model Registry）**：数据驱动，管理 13 种模型的参数定义、约束条件集合、可能的力集合。对应数据文档中的 FM-001 ~ FM-063 全部配置
2. **约束条件引擎（Constraint Engine）**：根据用户勾选的条件，确定当前受力集合（如"粗糙+静止+水平推力" → G, N, f, F），并进行矛盾检测
3. **物理求解器（Physics Solver）**：根据受力集合和参数值，计算各力的大小和方向
4. **正交分解引擎**：选择坐标系策略、计算分量、生成分解动画数据
5. **2D 渲染器**：绘制物体轮廓、场景（斜面/绳索等）、力的箭头、标注文字
6. **UI 控件层**：模型选择按钮组、条件勾选面板、参数滑块/输入框

### 关键技术风险或难点

1. **约束条件组合的穷举与管理**：数据文档定义了 60+ 种约束条件组合，如何以数据驱动的方式管理这些组合（而非硬编码 if-else），是架构设计的关键。建议采用**规则引擎**或**条件-结果映射表**
2. **连接体模型的复杂性**：连接体涉及整体法/隔离法切换，需要对多物体系统进行联立求解（FM-031 绳连水平面两物体、FM-032 绳连斜面-悬挂），这比单物体分析复杂度显著提升
3. **与 P-02 运动模拟器的融合**：PRD 明确指出"P-01 与 P-02 在产品中深度融合"。受力分析的结果需要能无缝传递给运动模拟器，架构设计需预留融合接口。融合点可能包括：受力结果 → 加速度计算 → 运动模拟
4. **力的示意图自动布局**：不同模型的几何场景差异很大（水平面 vs 悬挂 vs 圆周），力箭头需要自动避免重叠和遮挡，自动布局算法有一定难度
5. **PRD 提到的"意图识别 Agent"**：约束条件设置涉及一个 AI Agent，但本分析按任务要求不深入 AI 部分，仅记录这一融合点
6. **初中模型与高中模型的差异**：浮力/滑轮/杠杆（初中）的受力分析逻辑与高中模型不同，需要在求解器中做好分层

### 与 P-02 的融合点记录
- PRD 说明："P-02 运动模拟器与 P-01 受力分析器在产品中深度融合"
