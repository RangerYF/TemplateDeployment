# M-01 立体几何展示台 — 模块分析卡片

## 1. 用户场景 & 产品价值

### 目标用户
小学/初中/高中数学老师，不要求有任何编程或软件操作基础。核心用户画像是高中数学教师，需要在课堂上讲解立体几何（棱柱、棱锥、旋转体、截面、外接球等）。

### 解决什么教学痛点
- **空间想象困难**：立体几何需要学生在脑中旋转三维模型，黑板上的静态图"感觉差点意思"
- **截面/展开图理解**：正方体截面可以是三角形到正六边形，黑板上画不清楚
- **精确计算验证**：外接球半径、体积、表面积等公式，老师需要工具辅助演示和验证
- **建系与坐标**：空间直角坐标系的建立和各点坐标求解是高考核心考点，需要直观可视化

### 核心产品价值
**一句话**：老师选一个几何体、设几个参数，就能得到可交互旋转的 3D 模型，所有标注、计算、辅助功能一键完成，覆盖高中 80% 立体几何教学需求。

## 2. 用户实际使用方式

### 教师典型操作流程

| 场景 | 操作流程 |
|------|---------|
| **课前准备** | 打开浏览器 → 选几何体类型（如正四棱锥）→ 设参数（底面边长3，高4）→ 改顶点名（P/A/B/C/D）→ 加坐标系/外接球 → 保存场景 |
| **课中演示** | 投影仪打开 → 拖动旋转让学生从各角度观察 → 点击截面工具演示正方体截面 → 展示展开图 |
| **讲题** | 边讲边指出"这条边叫 AB，这个面叫侧面 ABC" → 选两条异面直线度量角度 → 建坐标系求坐标 → 点击体积/表面积看计算步骤 |

### 怎么用才是"好用"的
- **5 分钟上手**：从没用过的老师，5 分钟内能独立做出一个标注完整的 3D 模型（验收标准原文）
- **操作极简**：点几下就搞定，不写代码，不学复杂操作
- **数据准确**：外接球半径、体积、表面积与手算结果一致
- **投影清晰**：在 1080p 教室投影仪上文字和模型清晰可辨，字体 ≥ 14px

### 交互频率和深度
- **频率**：每节课使用 1-3 次，每次 5-15 分钟
- **深度**：中等 — 主要是参数调整 + 标注 + 辅助功能，不涉及复杂建模

### 需要实体管理的功能
几乎所有功能都涉及实体管理 — 几何体、点、线段、面、坐标系、外接球/圆、角度度量、距离度量共 **9 种实体类型**，均需要创建/选中/删除/编辑。

### 需要 Undo/Redo 的功能
所有对实体的变更操作（创建/删除/修改属性/拖动点/重命名/参数修改/截面创建等），共 **11 种 Command**。

### 参数驱动的功能
- 几何体参数（边长/高/半径等）→ 3D 模型实时更新
- 动点参数（棱上 t∈[0,1]）→ 关联度量实时变化
- 坐标系 → 所有点坐标实时计算显示

### 模拟/动画驱动的功能
无 — M-01 是纯静态几何展示，没有物理模拟或时间轴动画。

## 4. 核心依赖识别

### 技术依赖

#### 渲染方式
- **主渲染**：3D WebGL — Three.js + @react-three/fiber + @react-three/drei
- **辅助渲染**：2D SVG — 展开图（SVGSvg 渲染，缩放/平移/导出 PNG）
- **辅助渲染**：2D Canvas/SVG — 三视图投影
- **公式渲染**：LaTeX 精确值展示（体积/表面积/角度/距离）

#### 数学计算需求
- **重度**：13 种几何体的顶点/棱/面构建（Builder 模式）
- **重度**：13 种几何体的体积/表面积/侧面积公式计算
- **重度**：角度计算（二面角/线面角/线线角，含精确值识别 — 23 种已知精确角度）
- **重度**：距离计算（5 种类型，含 √n 简化和分数有理化）
- **中度**：外接球/外接圆的球心/圆心和半径计算
- **中度**：截面交点计算（平面与几何体各面的交线）
- **中度**：展开图 2D 坐标计算（11 种几何体）
- **中度**：三视图投影（正交投影到三个方向）
- **轻度**：坐标系变换（用户自定义原点/轴方向 → 各点在新坐标系下的坐标）

#### 公式渲染需求
- 体积/表面积的 LaTeX 精确值（如 `V = \frac{\sqrt{2}}{12} a^3`）
- 角度精确值（如 `\arctan\sqrt{2}`）
- 距离精确值（如 `\frac{a\sqrt{6}}{3}`）
- 分步计算过程展示

### 交互依赖

#### 交互模式
**编辑器型**（主导）+ **参数驱动型**（辅助）

这是一个典型的编辑器型应用：
- 有实体选中/高亮/多选
- 有属性面板（Inspector）
- 有工具栏（7 种工具，多步交互）
- 有撤销/重做
- 有实体列表/层级管理

同时有参数驱动特征：
- 几何体参数滑块 → 模型实时更新
- 动点拖拽 → 关联度量实时变化

#### 关键交互
| 交互类型 | 具体场景 |
|---------|---------|
| 点选 | 选中实体（点/线段/面）、工具多步选点 |
| 拖拽 | 3D 旋转/缩放/平移（OrbitControls）、动点在棱上滑动 |
| Hover 高亮 | 线段/面鼠标悬停高亮 |
| 滑块 | 几何体参数调整 |
| 数字输入框 | 精确参数设置 |
| 双击 | 顶点标签重命名 |
| 右键菜单 | 删除/重命名/取点 |
| 快捷键 | Ctrl+Z/Y（撤销/重做）、Escape（退出工具）、Delete（删除） |
| Ctrl+点击 | 穿透选中被遮挡元素 |

#### 是否需要实体系统
**是** — 这是架构的核心。9 种实体类型，统一的 Entity 接口，Registry 模式注册渲染器和 Inspector。

#### 是否需要命令系统
**是** — 11 种 Command + BatchCommand，完整的 Undo/Redo 栈（50 步），级联删除。

## 5. 竞品分析

### 5.1 国际产品

#### GeoGebra 3D Calculator
- **形态**：Web / Desktop / Mobile（全平台覆盖）
- **技术**：Java 核心 + GWT 编译为 JS；3D 自研 WebGL 管线；CAS 引擎 Giac 通过 Emscripten → WebAssembly
- **优势**：免费开源、全球 1 亿+ 用户生态、社区资源丰富、跨平台、AR 支持、代数-几何联动
- **不足**：3D 界面简陋、截面/展开图/外接球需手动构建（非一键内建）、GWT 技术栈陈旧包体积大、复杂场景性能瓶颈
- **交互模式**：编辑器型 + 代数驱动混合
- **启示**：社区生态的重要性；但 M-01 的差异化在于"一键"截面/展开图/外接球，降低操作门槛

#### Cabri 3D
- **形态**：Desktop（Windows 为主）
- **技术**：OpenGL 原生渲染
- **优势**：截面和展开图是**原生内建功能**（一键展开多面体）、流畅 OpenGL 动画、多投影视图、可嵌入 PPT
- **不足**：仅 Desktop、收费贵（站点许可 £589+）、技术停滞多年未更新、仅 Windows
- **交互模式**：编辑器型
- **启示**：Cabri 3D 的原生截面/展开图功能是行业标杆，M-01 在 Web 端实现了类似能力

#### Desmos 3D Calculator
- **形态**：Web（纯浏览器）
- **技术**：TypeScript；Canvas + WebGL 混合；自研 GLSL 着色器；LaTeX → AST → IR → JS/GLSL 代码生成
- **优势**：完全免费、界面极简美观、被 AP 考试和多州评估采用、零安装
- **不足**：3D 功能仍为 Beta；纯公式驱动（无法直接"画"几何体）；不支持截面/展开图/外接球
- **交互模式**：表达式驱动型
- **启示**：Desmos 证明了"极简 + 免费 + Web"的商业模式可行；但定位不同，M-01 面向几何操作而非方程可视化

#### Wolfram Mathematica
- **形态**：Desktop / Cloud
- **技术**：Wolfram Language 自有引擎，支持 OpenGL/WebGL、USD/GLTF 输出
- **优势**：数学计算能力无可匹敌、符号计算引擎、500+ 交互案例
- **不足**：昂贵、学习曲线极陡（需编程）、不适合中学教学场景
- **交互模式**：编程驱动型
- **启示**：Wolfram 的精确值符号计算是标杆；M-01 的精确角度库（23 种）和 √n 化简参考了类似思路

### 5.4 市场空白与 M-01 定位

**核心发现**：**Web 端立体几何专业工具存在空白**。

- GeoGebra 有 Web 端但截面/展开图/外接球需手动构建
- Cabri 3D、几何图霸、玲珑画板功能全面但**全部是 Desktop Only**
- Desmos 是纯方程驱动，无几何操作能力
- 国内工具贴合高中教材但技术陈旧

M-01 的差异化定位：**唯一在 Web 端提供截面+展开图+外接球+坐标系+精确度量一体化内建功能的立体几何教学工具**，且贴合中国高中数学教材。

## 6. 初步技术判断

### 推荐的渲染技术
**已采用且合适**：Three.js + @react-three/fiber + @react-three/drei

- Three.js 是 Web 3D 的事实标准，生态成熟
- @react-three/fiber 将 Three.js 集成到 React 声明式模型，开发效率高
- 对于教学级别的 3D 渲染（非游戏/影视级），性能充裕
- SVG 用于展开图渲染（2D 精确矢量图），选择合理

### 推荐的架构模式
**已采用且成熟**：七系统编辑器架构

```
用户操作 → Tool 捕获 → Command 封装 → EntityStore 变更 → Signal 广播 → Renderer 更新 / Inspector 刷新
```

| 系统 | 设计模式 | 评价 |
|------|---------|------|
| Entity（9 种类型） | Registry + 联合类型 | 类型安全，扩展性好 |
| Command（11 种 + Batch） | Command 模式 | Undo/Redo 完备 |
| Signal（7 个信号） | 发布-订阅 | 解耦良好 |
| Tool（7 种工具） | 策略模式 | 多步交互状态管理清晰 |
| Selection | 单例 Store | 简洁实用 |
| Property/Inspector（9 种） | Registry + 自注册 | 与 Entity 类型一一对应，可扩展 |
| Renderer（9 种） | Registry + 自注册 | 与 Entity 类型一一对应，可扩展 |

### 现有架构的优势（作为后续模块参考基准）

1. **严格的关注点分离**：Engine（纯函数计算）/ Editor（状态管理与命令）/ Components（React UI 渲染）三层分离，Engine 无 UI 依赖
2. **自注册 Registry 模式**：Renderer 和 Inspector 通过 Registry 注册，新增实体类型只需添加对应文件，无需修改核心代码
3. **完整的 Command/Undo 系统**：所有变更通过 Command 执行，天然支持 Undo/Redo 和操作日志
4. **Builder 缓存**：参数不变时复用构建结果，避免重复计算
5. **TypeScript 类型安全**：Entity 使用联合类型 + 泛型约束，编译期捕获类型错误
6. **级联删除**：删除点时自动清理关联线段/截面/度量，保持数据一致性

### 现有架构的不足/局限

1. **强耦合于"几何体编辑器"场景**：Entity 的 9 种类型、Tool 的多步交互、Builder 模式均针对立体几何设计，不适合直接复用到函数图像（M-02）、解析几何（M-03）等 2D 为主的模块
2. **Engine 层的复用性有限**：`builders/`、`math/calculators/`、`unfolding/`、`projection/` 都是立体几何专用，其他模块需要完全不同的计算引擎
3. **交互模式单一**：当前架构面向"编辑器型"交互（选中→操作→属性面板），不支持"模拟演示型"（播放/暂停/时间轴）和"参数探索型"（函数参数联动滑块）的交互范式
4. **无动画/时间轴系统**：物理模拟（P 系列模块）和函数动画需要帧循环、时间控制、状态插值，当前架构没有这些基础设施
5. **Store 设计面向单几何体**：EntityStore 管理一个场景内的所有实体，但没有"多场景"或"多图层"的概念，函数图像叠加等场景需要扩展

### 关键技术风险或难点

1. **截面计算精确性**：平面与多面体各面的交线计算涉及大量边界条件（平面过顶点、平面与棱平行等），是 Bug 高发区
2. **精确值输出**：√n 化简、分数有理化、精确角度识别（arctan√2 等）需要符号计算能力，当前用查表+模式匹配实现，扩展性有限
3. **展开图算法**：曲面体（圆锥、圆台）的展开涉及扇形/扇环参数计算，棱柱/棱锥的展开涉及面的 2D 铺排，算法复杂度中等
4. **3D Raycasting 精度**：Hover 高亮和点选依赖 Three.js Raycasting，在细线段和小角度面上可能存在精度问题
5. **投影仪适配**：1080p 教室投影仪的色彩和分辨率与开发环境差异大，需要实际测试

| 资产 | 可复用于 | 说明 |
|------|---------|------|
| Command + HistoryStore | 任何需要 Undo/Redo 的模块 | 通用模式，可直接迁移 |
