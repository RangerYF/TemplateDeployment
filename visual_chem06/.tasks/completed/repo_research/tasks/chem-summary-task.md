## 分析要求

按照 `PROGRESSIVE-PLAN.md` 中的**综合分析模板**产出 `outputs/chem-summary.md`。

### 第一步：模块间相似度评估

从以下维度构建对比矩阵，识别相似聚类：

| 维度 | 说明 |
|------|------|
| 交互模式 | 编辑器型 / 参数驱动型 / 模拟演示型 / 数据查看型 / 混合型 |
| 渲染技术 | 3D WebGL (Three.js) / 2D Canvas / SVG / DOM / 混合 |
| 实体系统需求 | 完整 / 轻量 / 无 |
| undo/redo 需求 | 完整命令系统 / 不需要 |
| 核心计算/算法引擎 | 3D分子坐标(Z-matrix) / 晶体学坐标转换 / 化学式解析+配平 / 粒子碰撞模拟 / pH曲线计算 / 动画引擎 / 图论算法 |
| 数据特征 | 大规模预置数据(80+分子/118元素) / 中等预置(10-15模型) / 小数据内置 / 图结构数据 |
| 动画/时间轴需求 | 播放/暂停/逐步/调速 / 简单过渡动画 / 无动画 |

### 第二步：暂分组并分析

基于对比矩阵将相似模块暂分组。已知的初步线索（从分析卡片得出）：

- **C-02 + C-05 显著相似**：两者都是 3D Three.js 查看器型模块，共享 OrbitControls、原子球体/键渲染、Raycasting、CPK 配色等基础设施。C-02 聚焦分子，C-05 聚焦晶体，但 3D 渲染管线高度可复用
- **C-06 + C-07 可能一组**：都是动画/模拟驱动型，都需要粒子动画渲染（电子/离子流向 vs 分子碰撞），都需要播放/暂停/逐步/调速控制，但 C-06 是 SVG+Canvas 而 C-07 是 Canvas+图表
- **C-03 独立性极强**：唯一的纯 DOM 文本处理型模块，核心是化学式解析和配平算法，与所有其他模块的交互模式和渲染技术完全不同
- **C-04 独立性强**：数据密集型周期表应用，DOM+CSS Grid 为主，与 3D/Canvas/SVG 模块无技术交集
- **C-08 轻量独立**：DOM+Chart 库的小品模块，可能归入某个组也可能独立
- **C-09 独立性强**：唯一的网络图可视化模块，SVG+D3.js，交互模式与所有其他模块不同

**但以上仅是初步线索，需要在分析中验证或推翻。**

### 第三步：分组验证（核心步骤）

对每个暂定分组回答三个问题：

1. **共享核心层的接口能否清晰定义？**
   - 能用一两句话说清楚这个核心层做什么吗？
   - 接口边界在哪里？

2. **核心层能支撑各模块的哪些功能？**
   - 逐模块列出：哪些功能走核心层，哪些功能是模块专属
   - 核心层覆盖率高 → 分组合理
   - 核心层覆盖率低（大部分功能是模块专属）→ 分组可能勉强

3. **各模块的专属能力是什么？**
   - 专属能力的实现复杂度如何
   - 专属能力之间是否有冲突

### 第四步：产出仓库划分与架构方案

对每个最终确定的仓库：

- 包含哪些模块，划分依据
- 核心架构模式
- 核心技术依赖
- 通用层 vs 模块专属层的边界定义
- 项目结构概览

### 特别注意

- **C-02 与 C-05 的 3D 共享层**是化学产品线中最明显的共享机会，需要精确定义共享边界：场景管理/相机控制/射线拾取/CPK配色可共享，但坐标计算逻辑（Z-matrix vs 分数坐标转换）完全不同
- **C-06 与 C-07 的动画引擎共享可行性**：两者都需要粒子动画+播放控制，但 C-06 是预置模型配置驱动的精确动画，C-07 是物理模拟驱动的随机粒子，需评估动画引擎的统一抽象是否可行
- **化学模块普遍不需要实体系统和 undo/redo**：与数学/物理产品线不同，8 个化学模块全部为查看器/工具型，无一需要编辑器架构，这对仓库架构设计有重大影响
- **渲染技术异构性极高**：Three.js WebGL（C-02/C-05）、Canvas 2D（C-06/C-07）、SVG/D3.js（C-09）、DOM+CSS（C-03/C-04）、DOM+Chart（C-08），这意味着渲染层共享空间有限，分组更应基于业务逻辑和数据特征而非渲染技术
- **化学式渲染的跨模块复用**：C-03（方程式配平）、C-06（电极反应方程式）、C-07（反应方程式）都需要化学式/方程式的渲染能力，这是一个潜在的跨组共享组件
- **数据管理模式差异**：C-02（80+分子JSON）、C-04（118元素JSON）是大规模预置数据，C-05（15晶体）、C-06（13模型）是中等预置数据，C-08（4+4内置）是小数据——数据管理策略可能不同
- 不要为了归组而归组：如果某个模块确实独立性强，单独成仓是合理的
- 架构设计要考虑实际开发效率：共享层太薄不如不共享，共享层太厚则模块灵活性受损
- 化学模块间的技术异构性远高于物理模块（物理12个全用2D Canvas），分组粒度可能更细
