### 1.2 暂分组与依据

基于对比矩阵，识别出两个聚类：

**聚类 A — P-04 独立**：编辑器型电路搭建器
- **唯一**的编辑器型模块，需要完整实体系统（元件 + 导线 + 端口） + 完整命令系统（undo/redo）
- 核心引擎是 MNA 电路分析（改进节点电压法），与其他 11 个模块的物理计算完全不同
- 交互模式是"拖拽放置 + 连线 + 参数调节"，架构复杂度远超其他模块
- 双渲染模式（电路图 + 实物图）是独有需求
- 10 种不同类型元件，每种有独特参数和行为模型

**聚类 B — 其余 11 个模块**：参数驱动 + 模拟演示型物理可视化
- **共同特征**：
  - 全部使用 2D Canvas 渲染
  - 全部采用"参数驱动 + 模拟演示"的架构范式（选模型 → 调参数 → 播放/观察）
  - 9/11 需要时间轴控制（播放/暂停/逐步/调速）
  - 6+ 个模块需要实时图表联动（v-t/s-t/x-t/p-V 等）
  - 5+ 个模块需要数值积分器（RK4/Euler）
  - 全部不需要完整实体系统和命令系统
  - 全部使用 KaTeX 公式渲染 + 参数滑块面板

**子聚类初步线索**：
- **静态分析组（P-01, P-03）**：纯参数驱动，无时间轴，"调参→看结果"
- **时间轴动画组（P-02, P-05, P-06, P-09, P-12, P-13, P-14）**：核心是动画 + 图表联动
- **混合组（P-07, P-08）**：部分子模块有粒子模拟/场计算的独特需求

## 2. 分组验证

### 2.1 聚类 A 验证：P-04 独立

#### 共享核心层接口定义

**不适用**——P-04 作为独立模块，不存在"与谁共享核心层"的问题。

#### P-04 独立的理由

1. **架构范式完全不同**：P-04 是编辑器架构（Entity-Command-Tool-View），其他 11 个模块是参数驱动/模拟演示架构（State → Compute → Animate → Render）。强行合并意味着核心层必须同时支撑两种完全不同的架构范式。

2. **实体系统和命令系统是独占需求**：10 种元件实体 + 导线实体 + 端口模型 + 电路拓扑图 + 8 种 Command + undo/redo 历史栈——这些在其他 11 个模块中完全不需要。

3. **MNA 电路分析引擎是独有核心技术**：改进节点电压法（G 矩阵构建 + 线性方程组求解 + 故障注入 + 非线性迭代）是 P-04 特有的算法核心，与力学/光学/电磁学的物理公式计算毫无重叠。

4. **交互复杂度最高**：拖拽放置 + 接线柱吸附 + 导线路由 + 碰撞检测 + 参数面板弹出——这些编辑器交互在其他模块中不存在。

5. **双渲染模式独有**：电路图 ↔ 实物图两套渲染系统是 P-04 专属需求。

#### P-13 与 P-04 的电路复用评估

P-13（电磁感应）的 P2 优先级功能"自由组装模式"可能需要电路分析能力，但：
- P-13 的核心功能（预设模型）使用硬编码的 ODE 方程，**不需要通用电路求解器**
- 自由组装模式是 P2 优先级，且其电路复杂度远低于 P-04（仅导轨 + 棒 + R/C/E 简单拓扑）
- 如需复用，可将 P-04 的 MNA 求解器提取为独立 npm 包
- **结论：P-13 与 P-04 无需合仓**

**结论**：P-04 独立成仓，理由充分。✅

#### P-01/P-03 "无时间轴"模块是否应单独成组？

**不应该**。理由：
1. P-01 和 P-03 的物理域完全不同（力学 vs 光学），专属计算引擎零共享
2. P-01 与 P-02 有"深度融合"关系（PRD 明确指出），需要在同一仓库内方便协作
3. P-03 虽然无时间轴，但与其他模块共享"实验模块选择 + 参数面板 + Canvas 渲染"的基础框架，核心层覆盖率约 50%，足以受益于同仓
4. 单独成组后核心层退化为"参数面板 + Canvas + KaTeX"，实质价值过低
5. 它们只是"不需要时间轴"，并非架构范式不同——仍然是参数驱动型

#### P-07/P-08 的特殊性评估

**P-07（热力学）**：
- 粒子碰撞引擎是独特需求（500 粒子 @ 60fps），但技术上只是 Canvas 粒子渲染 + 简单碰撞检测，不构成架构差异
- 液柱/气缸模型是参数驱动计算，完全符合聚类 B 的架构范式
- **结论：并入聚类 B**

**P-08（电磁场）**：
- 场线追迹和 Marching Squares 等势面是计算密集型专属能力
- 静电场模块有轻量电荷实体管理（放置/移动/删除），但复杂度远低于 P-04
- 粒子运动场景与 P-02 高度相似（时间轴 + 动画 + 解析/数值积分）
- **结论：并入聚类 B**——轻量实体管理不足以独立成仓，场线追迹是模块专属能力

#### 11 模块同仓的可行性评估

**支持同仓的论点**：
1. **架构范式高度统一**：全部是"State → Compute → Animate → Render"的参数驱动/模拟演示架构
2. **渲染技术完全统一**：12/12 使用 2D Canvas（无 3D 需求）
3. **核心层覆盖率高**：时间轴控制器、图表组件、参数面板、坐标系渲染、KaTeX、数值积分器——这些核心能力被 9-11 个模块共享
4. **跨模块复用明确**：
   - P-01 ↔ P-02：受力分析→运动模拟的数据接口
   - P-05 ↔ P-06：SHM 数学模型共享
   - P-05 ↔ P-14：弹簧振子物理模型共享
   - P-12 ↔ P-14：阿特伍德机模型共享
   - P-02/P-09/P-13：数值积分器共享
5. **构建输出独立**：每个模块独立打包为单 HTML 文件，tree-shaking 确保不引入无关代码

**反对同仓的疑虑**：
- 11 个模块数量较多——但物理模块的架构同质性远高于数学（数学的 M-01 是 3D 编辑器、M-05 是模拟演示型，差异大），物理的 11 个模块都是同一架构范式的不同实例化
- 代码量可能较大——但每个模块的专属代码量有限（主要是物理计算引擎 + 场景渲染，不涉及复杂架构），且各模块目录独立、互不干扰

**结论**：11 个模块同仓，分组合理。✅

## 4. 每个仓库的架构与技术方案

### 4.1 edu-physics-circuit（P-04 电路搭建器）

#### 核心架构模式

**编辑器架构**：Entity-Command-Tool-View + MNA 电路分析引擎

```
用户操作 → Tool 捕获（选择/放置/连线/删除）
  → Command 封装 → EntityStore 变更 → 拓扑图更新
    → MNA 引擎求解 → 渲染器更新 + 数值面板刷新
```

| 系统 | 设计模式 | 说明 |
|------|---------|------|

interface PhysicsModule {

interface TimelineController {
  play(): void;
  pause(): void;
  stepForward(dt: number): void;
  stepBackward(dt: number): void;

  setSpeed(multiplier: number): void;  // 0.1x ~ 5x

#### 开发优先级建议

1. **Phase 1：P-02 运动模拟器**（优先级最高）
   - 理由：P-02 是核心引擎的"试金石"——时间轴控制器、动画帧循环、Canvas 坐标系、实时图表联动、参数面板全部在 P-02 中首次验证和稳定化
   - 产出：核心共享引擎 + P-02 完整功能
   - P-02 的时间轴控制器是后续 8 个模块的基础

2. **Phase 2：P-01 受力分析器 + P-05 简谐运动**
   - P-01：与 P-02 深度融合，复用 Phase 1 的核心引擎
   - P-05：验证三图联动 + 数值积分器，新增弹簧渲染能力

3. **Phase 3：P-06 波动 + P-14 机械能守恒 + P-12 动量守恒**
   - P-06：复用 P-05 的 SHM 模型，主要增量是波的叠加和多普勒
   - P-14：验证能量条形图组件，复用 P-05 的弹簧振子模型
   - P-12：碰撞公式较简单，复用运动动画框架

4. **Phase 4：P-09 天体运动 + P-13 电磁感应**
   - P-09：复用数值积分器 + 时间轴，新增开普勒方程和轨道渲染
   - P-13：复用数值积分器 + 图表联动，核心工作在 ODE 耦合引擎

5. **Phase 5：P-03 光学实验台 + P-07 热力学 + P-08 电磁场**
   - 这三个模块的专属能力复杂度最高（像素级渲染/粒子系统/场线追迹），放在最后确保核心层已完全稳定
   - P-03 的像素级渲染是独立技术栈
   - P-07 的粒子碰撞引擎独立性强
   - P-08 的场线追迹算法鲁棒性需要充分验证
