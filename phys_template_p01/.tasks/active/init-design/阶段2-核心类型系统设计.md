# 阶段2：核心类型系统设计

> 目标：定义 Entity System 的完整 TypeScript 类型体系，使后续所有模块（注册机制、预设schema、求解器、渲染器）都基于这套类型工作。

---

## 设计约束与前置条件

### 已确定的技术栈（阶段1产出）
- TypeScript strict 模式
- React 18 + Zustand（UI 面板部分）
- 原生 Canvas 2D API（渲染部分）
- 手写数学工具函数（`core/physics/`）

### 核心设计原则（来自产品方案）
1. **数据模型通用，求解逻辑枚举注册**：Entity/Relation/State 的类型是结构化通用的，不针对特定物理域
2. **Phase 1 & Phase 2 兼容**：类型设计必须同时支持"预设加载"和未来的"动态搭建"
3. **只加不改**：新增实体类型/关系类型不需要修改 `core/types.ts` 中已有代码，通过注册机制扩展
4. **参数面板 schema 驱动**：参数类型定义需支持自动渲染 slider/input/toggle/select

---

## 需要设计的类型清单

### 1. 基础向量与几何类型

**场景需求**：
- 所有实体都有位置信息
- 力、速度、加速度等物理量都是向量
- 需要在物理坐标和画布像素坐标之间转换

**需要定义**：
- `Vec2`：二维向量 `{ x: number, y: number }`
- `Rect`：矩形区域（用于场区域、命中检测）
- `Transform`：位置 + 旋转（用于实体的空间属性）

### 2. Entity 基础接口

**场景需求**（产品方案 1.1）：
- 场景中的一切都是实体，每个实体有统一的 id、type、properties
- 实体类别：物体、表面、连接件、场、约束源、观测工具
- 每类实体有不同的属性集（质量/电荷/角度/场强等）

**需要定义**：
- `EntityId`：实体唯一标识
- `EntityType`：实体类型标识（字符串，如 `'block'`、`'slope'`、`'bfield'`）
- `Entity`：实体基础接口（id、type、position、rotation、properties）
- `EntityCategory`：实体类别枚举（`'object'` | `'surface'` | `'connector'` | `'field'` | `'constraint'` | `'instrument'`）

**设计关注点**：
- `properties` 如何设计？用 `Record<string, unknown>` 还是用泛型 `Entity<T>`？
- 需要支持各域在自己的目录中定义具体实体类型（如 `BlockEntity extends Entity`），但 `core/types.ts` 中只定义基础接口
- 每个实体类型的属性集不同，类型系统要灵活但不丧失类型安全

### 3. Relation 接口

**场景需求**（产品方案 1.2）：
- 物理场景的关键是实体之间的关系
- 关系类型：接触、连接、场作用、包含
- 关系连接两个实体（from → to），附带约束属性

**需要定义**：
- `RelationId`：关系唯一标识
- `RelationType`：关系类型标识（字符串，如 `'contact'`、`'connection'`、`'field-effect'`、`'containment'`）
- `Relation`：关系基础接口（id、type、fromEntityId、toEntityId、properties）

**设计关注点**：
- 关系是否有方向性？（"物块在斜面上"，from=物块，to=斜面）
- 关系的 properties 如何设计？（接触关系可能有摩擦系数，连接关系可能有连接件引用）
- 求解器需要通过关系类型+实体类型组合来匹配，类型设计要便于匹配查询

### 4. SimulationState 接口

**场景需求**：
- Simulator 每帧需要读取/更新所有实体的状态
- 时间轴拖拽需要跳到任意时刻的状态快照
- 多个视角层需要从同一份状态数据中提取不同物理量

**需要定义**：
- `SimulationState`：某一时刻所有实体的完整状态快照
- `EntityState`：单个实体在某一时刻的状态（位置、速度、加速度、受力列表等）
- `SimulationStatus`：模拟状态枚举（`'idle'` | `'running'` | `'paused'` | `'finished'`）
- `TimelineState`：时间轴状态（当前时刻、总时长、播放速率）

**设计关注点**：
- 解析解场景（给定 t 直接算）和数值积分场景（逐帧迭代，三种积分器：Semi-implicit Euler / Velocity Verlet / RK4）的状态管理方式不同，类型设计要兼容两种
- 状态快照是否需要包含"历史轨迹"数据？（运动视角需要轨迹线绘制）
- 性能考虑：状态更新频率 60fps，数据结构要轻量

### 5. PhysicsResult 接口

**场景需求**（产品讨论记录第五条）：
- 求解器输出格式：每个物体受的所有力、合力、加速度
- 不同求解器输出的数据格式应统一，视角层和渲染器才能通用处理
- 需要支持分步展示（先单个力→再合力→再正交分解）

**需要定义**：
- `Force`：力的描述（类型、大小、方向、颜色、标签）
- `ForceAnalysis`：单个实体的受力分析结果（力列表、合力、正交分解结果）
- `MotionState`：运动状态（位置、速度、加速度、轨迹点）
- `EnergyState`：能量状态（动能、各势能、总能量、做功）
- `PhysicsResult`：求解器的完整输出（所有实体的受力+运动+能量结果）

**设计关注点**：
- 力的类型需要一个枚举或标识系统，用于颜色映射（产品方案 3.1 的视觉语言表）
- PhysicsResult 是否应该按视角拆分？还是提供完整数据，由视角层自行筛选？
- 求解器输出需要支持"解析解全量输出"（一次计算出所有时刻的结果）和"逐帧输出"两种模式
- 逐帧输出模式下，积分器选择影响结果精度：辛积分器（Semi-implicit Euler / Velocity Verlet）保能量适合守恒系统，RK4 仅用于非守恒 ODE（如电磁感应阻尼）

### 6. 参数 Schema 类型

**场景需求**（产品方案 4.3、7.5）：
- 参数面板由 schema 驱动自动渲染
- 支持的控件类型：slider（滑块）、input（数值输入）、toggle（开关）、select（下拉选择）
- 每个参数有 min/max/step/default/label/unit
- 参数改变时实时更新物理计算和渲染

**需要定义**：
- `ParamType`：参数控件类型（`'slider'` | `'input'` | `'toggle'` | `'select'`）
- `ParamSchema`：单个参数的完整 schema（name、label、type、min、max、step、default、unit、options 等）
- `ParamGroupSchema`：参数分组（便于面板分区展示）
- `ParamValues`：参数值的运行时容器

**设计关注点**：
- slider 和 input 的数值范围如何约束？需要支持整数和浮点
- toggle 对应的 boolean 参数和 select 对应的枚举参数的 schema 差异
- 参数之间的联动关系如何表达？（如"匀速"toggle 打开时自动反算摩擦因数）
- 参数变更需要触发求解器重新计算，schema 中是否需要标记"影响哪些实体的哪个属性"

### 7. 视角层类型

**场景需求**（产品方案 1.4、3.2）：
- 6种视角：受力、运动、能量、动量、场、电路
- 视角可叠加，有主视角和叠加层
- 信息密度：简洁/标准/详细
- 视角切换时清除叠加层

**需要定义**：
- `ViewportType`：视角枚举（`'force'` | `'motion'` | `'energy'` | `'momentum'` | `'field'` | `'circuit'`）
- `InfoDensity`：信息密度枚举（`'compact'` | `'standard'` | `'detailed'`）
- `ViewportState`：视角层状态（当前主视角、叠加层列表、信息密度）
- `ViewportData`：视角层需要的数据接口（不同视角从 PhysicsResult 提取不同数据）

**设计关注点**：
- 不同视角需要不同的数据格式：受力→力列表，运动→轨迹点+v-t数据，能量→能量条形图数据
- 视角层渲染器和实体渲染器是否共用接口？
- 预设如何声明支持的视角？（并非所有预设都支持全部6种视角）

### 8. 渲染相关类型

**场景需求**：
- 每种实体类型有自己的渲染方式（物块→矩形、斜面→三角形等）
- 物理量有统一的视觉语言（颜色、箭头样式）
- 坐标系变换（物理坐标 ↔ 画布像素坐标）
- 命中检测（hit test）

**需要定义**：
- `RenderContext`：渲染上下文（Canvas context + 坐标变换参数 + 视角状态）
- `HitTestResult`：命中检测结果（命中的实体ID、命中位置）
- `VisualStyle`：力/物理量的视觉样式（颜色、线宽、箭头样式、透明度）
- `ForceVisual`：力的视觉语言映射表类型（产品方案 3.1 的颜色-形状-标注映射）

## 子任务执行顺序

```
1. 设计基础类型（Vec2、Rect、Transform）→ 其他所有类型都依赖
2. 设计 Entity 接口 → 是整个系统的核心数据结构
3. 设计 Relation 接口 → 与 Entity 紧密关联
4. 设计参数 Schema 类型 → Entity 的属性面板驱动
5. 设计 PhysicsResult 接口 → 求解器输出格式
6. 设计 SimulationState 接口 → 组合 Entity + PhysicsResult 的状态管理
7. 设计视角层类型 → 从 PhysicsResult 提取数据
8. 设计渲染相关类型 → 最终消费数据的环节
9. 验证：用"斜面物块受力"和"带电粒子在磁场中"两个场景验证类型表达力
10. 输出完整的类型设计文档
```

## 验收标准

✅ Entity 类型能表达产品方案中列出的所有实体类别（物体、表面、连接件、场、约束源、观测工具）
✅ Relation 类型能表达所有关系类型（接触、连接、场作用、包含）
✅ 参数 Schema 类型能驱动参数面板自动渲染 slider/input/toggle/select
✅ PhysicsResult 能统一表达解析解和 ODE 数值积分两种求解器的输出
✅ 类型设计兼顾 Phase 1（预设加载）和 Phase 2（动态搭建）的需求
✅ 用"斜面物块受力"验证力学域的类型表达力
✅ 用"带电粒子在磁场中"验证电磁域的类型表达力
✅ 各域可以在自己的目录中扩展具体类型，无需修改 `core/types.ts`
