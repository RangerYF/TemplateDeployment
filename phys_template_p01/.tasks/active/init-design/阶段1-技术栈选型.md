# 阶段1：技术栈选型与验证

> 目标：确定项目使用的全部技术栈，记录每个选择的理由，确认关键组合可行。

## 需要决策的技术领域

### 1. 前端框架

**场景需求**：
- 三栏布局（左参数面板260px + 中画布自适应 + 右信息面板300px + 顶栏48px + 底时间轴56px）
- 参数面板由 JSON schema 驱动自动渲染（slider/input/toggle/select）
- 实体列表、选中联动、视角切换等标准 UI 交互
- 不需要 SSR，纯客户端应用

**候选**：
| 方案 | 优势 | 劣势 |
|------|------|------|
| React + Zustand | 生态成熟，Zustand 轻量简单，适合 Canvas 应用 | — |
| React + Redux | 生态最全 | 样板代码多，本项目状态不复杂，用 Redux 过重 |
| Vue 3 + Pinia | 模板语法对 schema 驱动的表单渲染友好 | 团队（Claude Code）对 React 生态更熟悉 |
| Solid.js | 细粒度响应式，性能好 | 生态较小，遇到问题参考资料少 |

**决策关注点**：
- 状态管理复杂度不高（当前选中实体、当前视角、模拟状态、参数值），不需要重量级方案
- Canvas 渲染不走 React 虚拟 DOM，框架主要管 UI 面板部分
- 两位开发者都通过 Claude Code 开发，框架的 AI 辅助开发友好度是考量因素

### 2. Canvas 渲染方案

**场景需求**：
- 2D 物理场景渲染（物块、斜面、弹簧、场区域等几何图形）
- 力箭头、速度箭头、轨迹线等标注绘制
- 实体点击选中（hit test）
- 60fps 动画（物体运动、箭头长度变化）
- 场景复杂度上限：几个物体 + 若干箭头 + 轨迹线 + 场区域标记

**候选**：
| 方案 | 优势 | 劣势 |
|------|------|------|
| 原生 Canvas 2D API | 无依赖、完全可控、性能最好、学习曲线低 | hit test 要自己实现，无内置场景图 |
| Konva.js | 内置场景图和事件系统，hit test 开箱即用 | 引入额外依赖，API 抽象可能与物理渲染需求不完全匹配 |
| Pixi.js | WebGL 加速，性能极好 | 本项目场景简单，WebGL 是杀鸡用牛刀 |
| Fabric.js | 适合图形编辑器 | 更偏向静态图形编辑，不太适合物理动画 |

**决策关注点**：
- 场景复杂度很低（不超过10个实体），原生 Canvas 性能绰绰有余
- hit test 实现不复杂（产品讨论记录第十三条已列出所有实体的命中检测方式，都是基础几何判断）
- 物理箭头、轨迹线等需要精确的像素级控制，原生 API 反而更灵活
- 要支持坐标系变换（物理坐标 ↔ 画布像素坐标）

### 3. 图表库

**场景需求**：
- v-t 图、s-t 图（运动视角）：折线图，随模拟时间实时更新
- 能量条形图（能量视角）：多个柱状图实时变化 + 总能量参考线
- a-t 图、x-t 图（振动视角）：正弦曲线
- 图表需要与模拟动画同步：当前时刻用竖线标出
- 图表区域在右侧信息面板（300px宽度内）

**候选**：
| 方案 | 优势 | 劣势 |
|------|------|------|
| Chart.js | 简单易用，内置动画，轻量 | 实时逐帧更新可能有性能问题 |
| D3.js | 灵活度极高，可做任意自定义 | 学习曲线陡峭，API 偏底层 |
| uPlot | 超轻量（<40KB），专为时序数据优化，性能极好 | 功能较少，样式自定义不如 D3 |
| 自绘（Canvas） | 无依赖，和主画布技术统一 | 要自己实现坐标轴、刻度、标签等 |

**决策关注点**：
- 图表需要和模拟引擎帧同步更新（每帧追加数据点），对性能有要求
- 需要在图表上画"当前时刻"竖线，图表库是否支持自定义叠加层
- 能量条形图是实时变化的柱状图，不是静态图表
- 300px宽度约束，图表要紧凑可读

### 5. 数学计算库

**场景需求**：
- 向量运算（加减、旋转、模长、点积）
- 三角函数（sin/cos/tan 及反函数）
- 数值积分（RK4）
- 一元二次方程求解、2x2线性方程组求解
- 不需要矩阵运算（无3D变换）、不需要符号计算

**候选**：
| 方案 | 优势 | 劣势 |
|------|------|------|
| 手写（core/physics/） | 无依赖、体积小、完全可控 | 需要自己写和测 |
| Math.js | 功能全面 | 包体积大（>600KB），本项目只用到其中1% |
| gl-matrix | 向量/矩阵运算，轻量 | 偏3D（vec3/mat4），2D向量要自己适配 |

**决策关注点**：
- 所需计算都很基础（高中物理公式），不需要复杂数学库
- RK4 实现只有20行代码
- 引入大型数学库的价值不大，反而增加包体积和依赖复杂度

### 6. 公式渲染

**场景需求**：
- 产品方案维度三（3.2）提到"详细"信息密度下显示公式（如 F=ma, v²=v₀²+2ax）
- 力箭头旁标注 "G=19.6N"（简单文字，不需要公式渲染）
- 信息面板可能显示推导过程（Phase 3 方向）

**候选**：
| 方案 | 说明 |
|------|------|
| KaTeX | 轻量 LaTeX 渲染，PRD 中已提及 |
| 纯文本 + Unicode | G=mg、F₁=μmg 等简单公式用 Unicode 下标即可 |
| 暂不引入 | Phase 1 的公式展示需求很弱，可以先用文字标注 |

**决策关注点**：
- Phase 1 的核心是可视化，不是公式推导
- 箭头旁的标注（"G=19.6N"）用 Canvas fillText 就够了
- KaTeX 可以后续需要时再引入

## 子任务执行顺序

```
1. 确定前端框架 + 状态管理 → 影响后续所有 UI 代码
2. 确定 Canvas 渲染方案 → 影响 renderer/ 整体设计
3. 确定图表库 → 影响信息面板的实现方式
4. 确定构建工具 + 项目配置 → 需要前端框架先定
5. 确定数学计算策略 → 影响 core/physics/ 的实现方式
6. 确定公式渲染策略 → 依赖最少，最后决定
7. 输出技术栈选型结论文档
```
