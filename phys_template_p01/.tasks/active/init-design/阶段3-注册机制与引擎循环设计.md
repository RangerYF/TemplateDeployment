# 阶段3：注册机制与引擎循环设计

> 目标：设计三大注册表（实体、求解器、渲染器）的 API，以及 Simulator 运行循环的接口，使两个域能通过注册机制独立挂载功能。

---

## 设计约束与前置条件

### 已确定的类型基础（阶段2产出）

阶段3的所有 API 签名基于阶段2定义的 51 个 TypeScript 接口。关键类型：

- **输入侧**：`Entity<T>`、`Relation<T>`、`SceneDefinition`、`ParamSchema`、`ParamValues`
- **输出侧**：`PhysicsResult`、`ForceAnalysis`、`MotionState`、`EnergyState`
- **渲染侧**：`RenderContext`、`ViewportData`、`HitTestResult`、`VisualStyle`
- **状态侧**：`SimulationState`、`TimelineState`、`SimulationStatus`、`SolveMode`、`IntegratorType`

阶段2已给出的 API 签名草案（本阶段需正式设计并细化）：
```typescript
registerEntity(type: EntityType, config: EntityRegistration): void;
registerSolver(pattern: SolverPattern, solver: SolverFunction): void;
registerRenderer(entityType: EntityType, renderer: EntityRenderer): void;
registerViewportRenderer(viewportType: ViewportType, renderer: ViewportRenderer): void;
```

### 核心设计原则
1. **只加不改**：新域注册不需要修改已有注册代码或公共文件
2. **插件式挂载**：`domains/mechanics/index.ts` 和 `domains/em/index.ts` 各自调用注册 API，在 `app/main.ts` 中导入即完成集成
3. **输入/输出分离**：`SceneDefinition`（静态输入）→ 求解器 → `PhysicsResult`（动态输出），职责清晰

## 需要设计的 API 清单

### 1. 实体注册 API

**场景需求**：
- 各域需要注册自己的实体类型（力学域注册 block/slope/spring，电磁域注册 charged-particle/bfield/efield）
- 注册时需声明：类型标识、类别、默认属性、参数 schema、渲染形状信息
- Phase 2 搭建器需要从注册表中读取"可用实体类型列表"和默认属性

**需要设计**：
- `EntityRegistration`：实体类型注册信息的完整结构
  - `type: EntityType` — 类型标识
  - `category: EntityCategory` — 类别
  - `defaultProperties` — 默认属性值（创建新实体时使用）
  - `paramSchemas: ParamSchema[]` — 该实体类型的可调参数定义
  - `label: string` — 显示名称（如"物块"、"斜面"）
  - `hitTest` — 命中检测函数签名（各实体形状不同）
- `EntityRegistry` 的 API：
  - `register(config: EntityRegistration): void`
  - `get(type: EntityType): EntityRegistration | undefined`
  - `getByCategory(category: EntityCategory): EntityRegistration[]`
  - `getAll(): EntityRegistration[]`

**设计关注点**：
- 注册表是单例还是可实例化？（单例更简单，但可测试性弱）
- `hitTest` 函数签名：`(entity: Entity, point: Vec2, transform: CoordinateTransform) => HitTestResult | null`
- 是否需要注册时提供"工厂函数"来创建该类型的新实体实例？（Phase 2 搭建器需要）

### 2. 求解器注册 API

**场景需求**（产品方案 7.3、讨论记录第五条/第二十一条）：
- 求解器按"实体类型组合 + 关系类型"枚举注册
- 运行时根据场景中的实体和关系组合，从注册表中匹配求解器
- 同一场景可能匹配多个求解器（如斜面上的物块同时有重力求解 + 接触力求解），需要确定组合执行方式
- 解析解和数值积分两种模式的求解器需要统一 API

**需要设计**：
- `SolverPattern`：匹配模式的表达方式
  - 方案A：字符串模式 `"block + slope + contact"` — 简单直观
  - 方案B：结构化对象 `{ entityTypes: ['block', 'slope'], relationType: 'contact' }` — 类型安全
  - 方案C：谓词函数 `(scene: SceneDefinition) => boolean` — 最灵活但失去了声明式匹配能力
- `SolverFunction`：求解器函数签名
  - 解析解：`(scene: SceneDefinition, t: number) => PhysicsResult` — 给定时刻 t 直接计算
  - 数值积分：`(scene: SceneDefinition, prevResult: PhysicsResult, dt: number) => PhysicsResult` — 基于前一帧迭代
  - 统一签名如何设计？
- `SolverRegistration`：完整注册信息
  - `id: string` — 求解器标识（如 `'slope-contact-solver'`）
  - `pattern: SolverPattern` — 匹配模式
  - `solveMode: SolveMode` — 解析解 / 数值积分
  - `integrator?: IntegratorType` — 数值模式下推荐的积分器
  - `solve: SolverFunction` — 求解函数
  - `priority?: number` — 优先级（多个求解器匹配时的执行顺序）
- `SolverRegistry` 的 API：
  - `register(config: SolverRegistration): void`
  - `match(scene: SceneDefinition): SolverRegistration[]` — 返回所有匹配的求解器（按优先级排序）
  - `get(id: string): SolverRegistration | undefined`

**设计关注点**：
- 匹配模式的表达力：需要区分"物块在斜面上"和"物块在水平面上"，它们的实体类型不同（slope vs horizontal-surface），但关系类型都是 contact
- 多求解器组合策略：如果一个场景中物块在斜面上且有弹簧连接，重力/接触力/弹簧力是分别用三个求解器还是一个组合求解器？
  - 方案1：每个求解器只计算自己负责的力，最终合并 → 求解器粒度细，但需要合并逻辑
  - 方案2：每个预设绑定一个"完整求解器"，负责该场景的全部力 → 求解器粒度粗，但实现简单（符合"每个预设硬编码方程"的设计）
- 解析解求解器能否一次输出整条时间线的结果？（`PhysicsResult[]` 而非单个 `PhysicsResult`）→ 对时间轴拖拽体验有重大影响

### 3. 渲染器注册 API

**场景需求**：
- 每种实体类型有自己的 Canvas 渲染方式（物块→矩形、斜面→三角形、场区域→半透明填充+符号阵列）
- 渲染器需要读取 `PhysicsResult` 中的数据来绘制箭头、标注等
- 视角层渲染器与实体渲染器是独立的：实体渲染器画物理对象，视角渲染器画物理量（力箭头、v-t 图等）

**需要设计**：
- `EntityRenderer`：实体渲染器函数签名
  - `(entity: Entity, result: PhysicsResult, ctx: RenderContext) => void`
  - 负责绘制实体本身（形状、填充、边框、选中高亮）
- `ViewportRenderer`：视角层渲染器函数签名
  - `(data: ViewportData, entities: Map<EntityId, Entity>, ctx: RenderContext) => void`
  - 负责绘制物理量叠加层（力箭头、轨迹线、能量条形图等）
- `RendererRegistry` 的 API：
  - `registerEntity(entityType: EntityType, renderer: EntityRenderer): void`
  - `registerViewport(viewportType: ViewportType, renderer: ViewportRenderer): void`
  - `getEntityRenderer(entityType: EntityType): EntityRenderer | undefined`
  - `getViewportRenderer(viewportType: ViewportType): ViewportRenderer | undefined`

**设计关注点**：
- 渲染顺序：先画表面（斜面、水平面）→ 再画场区域 → 再画物体 → 最后画视角叠加层。注册表是否需要记录 z-order / 渲染层级？
- 选中状态渲染：选中实体需要高亮边框，这个逻辑在每个渲染器中各自实现？还是由渲染循环统一处理？
- 叠加层透明度：视角叠加层用 0.3 透明度渲染，是在 `RenderContext` 中传入还是视角渲染器自行处理？

### 4. Simulator 运行循环

**场景需求**（产品方案 1.3）：
```
实体 + 关系 + 参数 → 受力分析 → 运动求解 → 事件检测 → 状态更新 → 渲染
```
- 60fps 动画循环
- 解析解场景和数值积分场景的循环策略不同
- 参数变更需要立即触发重算
- 暂停状态下仍需渲染（只是不推进时间）

**需要设计**：
- `Simulator` 的公开接口：
  - `loadPreset(preset: PresetData): void` — 加载预设，初始化场景
  - `start(): void` — 开始模拟
  - `pause(): void` — 暂停
  - `resume(): void` — 恢复
  - `seekTo(time: number): void` — 跳到指定时刻
  - `setPlaybackRate(rate: number): void` — 设置播放速率
  - `updateParam(key: string, value: ParamValues[string]): void` — 参数变更
  - `reset(): void` — 重置到初始状态
  - `getState(): SimulationState` — 获取当前状态
- 内部循环流程：
  - `requestAnimationFrame` 驱动
  - 每帧：计算 dt → 匹配求解器 → 执行求解 → 检测事件 → 更新状态 → 触发渲染
- 与 Zustand store 的集成方式

**设计关注点**：
- 解析解场景的 `seekTo` 实现：直接代入 t 重算，O(1) 复杂度
- 数值积分场景的 `seekTo` 实现：
  - 向前跳：从当前时刻逐帧积分到目标时刻？太慢
  - 向后跳：需要从 `resultHistory` 中查找？
  - 混合方案：对数值积分场景，首次播放时缓存 `resultHistory`，后续 seekTo 从缓存读取
- 参数变更时的处理流程：参数改变 → 更新 `SceneDefinition` 中对应实体的属性 → 重置模拟状态 → 从 t=0 重新求解
- 事件检测机制如何设计？
  - 每个求解器是否需要注册自己的事件检测函数？
  - 事件类型：物体离开表面（N<0）、物体进入/离开场区域、到达终态、碰撞
  - 事件触发后的动作：切换求解器（多段运动）、停止模拟、状态标记

### 5. 解析解与数值积分的统一求解器接口

**场景需求**（讨论记录第五条）：
- 解析解场景（斜面、平抛、圆周等）：给定 t 直接用公式算
- 数值积分场景（电磁感应单棒等）：逐帧 ODE 积分
- 预设 JSON 声明使用哪种模式

**需要设计**：
- 统一的求解器调用接口，Simulator 不需要关心底层是解析解还是数值积分
- 解析解求解器是否可以预计算整条时间线？
  - 优势：`seekTo` O(1)、参数变更后一次性算完、时间轴拖拽极流畅
  - 劣势：如果参数面板频繁调整，每次都全量重算有浪费
- 数值积分求解器的积分器选择：
  - 默认 Semi-implicit Euler（大多数场景）
  - 周期运动用 Velocity Verlet（弹簧振子、天体）
  - 非守恒 ODE 用 RK4（电磁感应阻尼）
  - 由预设 JSON 声明还是求解器注册时指定？

**设计关注点**：
- 统一接口的抽象层次：是在 `SolverFunction` 签名层面统一，还是在 `Simulator` 调用层面统一？
- 解析解求解器的"预计算"结果如何缓存？放在 `SimulationState.resultHistory` 中？

### 6. 时间轴控制接口

**场景需求**（产品方案 3.3 页面布局、使用场景 S4）：
- 播放 / 暂停 / 拖拽到任意时刻 / 变速（0.5x / 1x / 2x）
- 底部控制栏 56px：`◀│ ▶ ⏸ ⏩│ ──●──────── │ t=0.0s │ [0.5x][1x][2x]`
- 拖拽时间轴时，画布实时更新到对应时刻的状态

**需要设计**：
- `TimelineController` 的接口（或直接是 Simulator 的一部分？）
- 时间轴 UI 组件需要的数据：当前时刻、总时长、播放状态、进度百分比
- 拖拽时的交互行为：
  - 解析解场景：拖拽时实时计算并渲染
  - 数值积分场景：拖拽时从 `resultHistory` 读取最近帧渲染
- 与 Zustand store 的绑定方式

**设计关注点**：
- 时间轴是连续值还是离散帧？（UI 显示连续，但底层可以按帧对齐）
- 播放结束（到达 `duration`）后的行为：自动暂停？循环？

### 7. 预设加载流程

**场景需求**（产品方案 7.5、讨论记录第十八条）：
- 预设 JSON 文件 → 加载 → 注入 Entity System → 匹配求解器 → 就绪
- "加一个新功能点 = 加一个新 JSON 文件"，不需要改已有代码

**需要设计**：
- `PresetData`：预设 JSON 的 TypeScript 类型（从 SceneDefinition 扩展，加元信息）
  - 元信息：id、name、description、category、supportedViewports
  - 场景数据：entities、relations、paramGroups、paramValues
  - 求解配置：solveMode、integrator、duration
- `PresetLoader` 的接口：
  - `load(preset: PresetData): SceneDefinition` — 解析 JSON → 构造 SceneDefinition
  - `validate(preset: PresetData): ValidationResult` — 校验预设完整性（实体类型已注册？关系引用的实体存在？）
- 加载流程的每一步：
  1. 读取 JSON 文件
  2. 校验 schema 合法性
  3. 创建实体实例（从注册表获取默认属性，用预设值覆盖）
  4. 创建关系实例
  5. 匹配求解器（从 SolverRegistry 查找）
  6. 初始化 SimulationState
  7. 触发首帧渲染

**设计关注点**：
- 预设 JSON 中的实体描述与 `Entity` 接口的差异：JSON 中可能只有 type + properties（id 由加载器生成），而 `Entity` 需要完整的 id + type + category + transform + properties
- 预设校验失败时的处理策略：静默降级？报错提示？
- 预设切换时的清理逻辑：销毁当前场景、清理 Canvas、重置状态

## 子任务执行顺序

```
1. 设计 SolverPattern 匹配模式 → 核心难点，决定注册机制的表达力
2. 设计 SolverFunction 统一签名 → 需要先确定解析解/数值积分的统一方案
3. 设计三大注册表 API（EntityRegistry + SolverRegistry + RendererRegistry）→ 都依赖上面两步
4. 设计 Simulator 运行循环 → 消费注册表的核心消费者
5. 设计事件检测机制 → Simulator 内部子系统
6. 设计时间轴控制接口 → Simulator 的外部控制接口
7. 设计预设加载流程 → 整个管道的入口
8. 验证：用"斜面物块受力"走通 注册→加载→求解→渲染 全流程
9. 验证：用"电磁感应单棒"走通 数值积分场景的全流程
10. 输出完整 API 设计文档
```

## 验收标准

✅ 求解器注册的 pattern 表达方式已确定，能区分"物块在斜面上"和"带电粒子在磁场中"
✅ 注册 API 支持"只加不改"——新域注册不需要修改已有注册代码
✅ Simulator 运行循环能同时处理解析解场景（直接公式）和 ODE 场景（Semi-implicit Euler / Velocity Verlet / RK4）
✅ 时间轴控制能支持播放/暂停/拖拽到任意时刻/变速
✅ 预设加载流程从 JSON 到渲染就绪的每一步都有明确的接口
✅ 事件检测机制能处理"物体离开表面""进入/离开场区域""到达终态"三类事件
✅ 用"斜面物块受力"走通注册→加载→求解→渲染全流程（伪代码级验证）
✅ 用"电磁感应单棒"走通数值积分全流程（伪代码级验证）
