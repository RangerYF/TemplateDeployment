# 架构设计任务 · 背景文档

> 任务目标：完成物理教具统一模拟器的技术架构设计，产出可直接用于项目初始化和并行开发的架构方案。

---

## 一、项目概述

### 产品定位

为中国初高中物理教师构建一个**统一的物理模拟器平台**，而非按物理域分散的多个独立工具。老师从预设场景库加载场景、调整参数、切换视角，3分钟内复现一道物理题的完整过程。

### 核心设计理念

- **一个统一模拟器**：所有物理域（力学、电磁、振动、热学、天体）共存于同一画布，支持跨域综合题（斜面+磁场）
- **场景可穷举**：高中物理题型高度模式化，不需要通用物理引擎，用枚举法覆盖
- **预设优先**：Phase 1 靠预设场景+参数微调覆盖85%教学场景，Phase 2 再开放结构化搭建

### 分阶段实施

| Phase | 产品形态 | 覆盖率 |
|-------|---------|--------|
| Phase 1 | 预设场景库 + 参数微调，老师不能自由添加实体 | ≈85% |
| Phase 2 | 结构化搭建（从菜单选组件拖入画布） | ≈95% |
| Phase 3 | 深化教学体验（更多预设、专项模板、导出增强） | ≈98% |

## 二、架构设计的目标

### 必须产出

1. **技术栈选型**：前端框架、渲染方案、构建工具、包管理等
2. **目录结构定义**：完整的 `src/` 目录树，明确每个目录的职责和归属
3. **core/types.ts 基础接口**：Entity、Relation、State 等核心类型定义
4. **注册机制设计**：实体注册、求解器注册、渲染器注册的具体实现方案
5. **Preset JSON Schema**：预设文件的完整字段定义
6. **参考实现规范**：一个最简预设（如"水平面物块受力"）的完整代码示例，作为两位开发者的开发范本

### 设计约束

- 必须支持两人并行开发（按物理域纵切，详见第五节）
- 必须为 Phase 2 预留扩展空间（Entity System 数据模型通用化）
- Phase 1 不做通用求解器、不做通用碰撞检测、不做手动添加实体
- 全部 2D 渲染，不引入 Three.js
- 目标设备：1920x1080 投影仪为主，兼容 1280x720 和 iPad

## 三、已确定的关键决策

以下决策已在产品方案讨论中敲定，架构设计时直接采用，不需要重新讨论。

### 3.1 仓库策略：单一 Monorepo

所有模块在同一个代码仓库，通过目录结构隔离边界。P-04 电路搭建器作为 monorepo 内的独立目录存在。

### 3.2 并行开发分工：按物理域纵切

| | 开发者 A：力学域 | 开发者 B：电磁域 |
|--|-----------------|-----------------|
| **P0 模块** | P-01 受力分析、P-02 运动模拟、P-12 动量守恒、P-14 机械能守恒 | P-08 电磁场可视化、P-13 电磁感应、P-04 电路搭建器 |
| **P1 模块** | P-05 简谐运动、P-06 波动、P-09 天体运动 | P-07 热力学 |

每人负责完整的功能域（引擎+渲染+UI+预设），独立交付可运行的功能，互不阻塞。

**选择纵切而非横切（引擎/UI分离）的原因**：vibe coding 模式下，每个人必须能独立跑起来验证功能，横切导致任何功能都需要两人对接才能看到效果。

### 3.3 求解器架构：数据模型通用，求解逻辑枚举注册

- Entity System 的数据模型按通用方式设计（为 Phase 2 预留）
- 求解器按"实体类型+关系类型"的组合模式枚举注册
- 运行时通过注册表匹配求解器：场景中有什么实体和关系 → 查表找到对应求解器 → 执行
- Phase 1 和 Phase 2 的区别仅在于场景数据来源（JSON文件 vs 用户操作），匹配机制不变

### 3.4 目录结构与防冲突机制

```
src/
├── core/                    # 框架机制（极少改动）
│   ├── entity-registry.ts   #   实体注册
│   ├── solver-registry.ts   #   求解器注册
│   ├── renderer-registry.ts #   渲染器注册
│   ├── types.ts             #   基础接口
│   └── physics/             #   共享物理工具（纯数学，不含域知识）
│       ├── vector/          #     向量运算（一个函数一个文件）
│       ├── geometry/        #     几何判断（一个函数一个文件）
│       ├── integrators.ts   #     数值积分器（RK4、欧拉）
│       ├── algebra.ts       #     代数方程求解
│       └── constants.ts     #     物理常数
│
├── shell/                   # UI 骨架（极少改动）
│   ├── Layout.tsx           #   三栏布局
│   ├── Canvas.tsx           #   画布容器
│   ├── ParameterPanel.tsx   #   参数面板（schema驱动）
│   ├── InfoPanel.tsx        #   信息面板
│   └── Timeline.tsx         #   时间轴控制
│
├── renderer/                # 渲染基础设施
│   ├── primitives/          #   箭头、标注、轨迹线等基础绘制
│   └── registry.ts          #   渲染器注册
│
├── domains/                 # 各自的领地，完全不交叉
│   ├── mechanics/           # 开发者 A
│   │   ├── entities/
│   │   ├── solvers/
│   │   ├── renderers/
│   │   ├── viewports/
│   │   ├── presets/
│   │   └── index.ts
│   └── em/                  # 开发者 B
│       ├── entities/
│       ├── solvers/
│       ├── renderers/
│       ├── viewports/
│       ├── presets/
│       └── index.ts
│
└── app/                     # 应用入口
    └── main.ts
```

**防冲突核心原则**：
- core/ 和 shell/ 提供机制，不含域知识
- 具体物理域的一切都在 domains/ 各自目录下
- 扩展基础设施时只加文件不改已有文件，通过注册机制集成
- core/physics/ 的 vector/ 和 geometry/ 按"一个函数一个文件 + index.ts 导出"组织

### 3.5 共享物理工具库的收录标准

`core/physics/` 只放两个域都会用到的纯数学函数：

- 放 core/：`rotateVector`、`pointInRect`、`rk4Step`、`solveQuadratic`、物理常数
- 不放（留在域 solvers/）：`lorentzRadius`、`slopeAccel`、`elasticCollision`、`inducedEMF` 等特定物理公式

判断标准：函数是否不关心"这是什么物理场景"。开发中发现某函数两边都需要，再提到 core/ 即可。

### 3.6 预设是功能点的载体

- 每个功能点 = 一个预设 JSON 文件
- 预设完整自描述：实体 + 关系 + 参数schema + 视角 + 功能开关
- 参数面板由预设中的 schema 驱动自动渲染，不需要为每个预设写 UI 代码
- 加新功能点 = 加新 JSON 文件，不改已有代码

## 四、架构设计需要解决的问题

以下问题在产品方案讨论中已识别，但具体方案留给架构设计阶段定义。

### 4.1 技术栈选型

产品需求文档建议的技术栈：
- 前端框架：React
- 渲染：Canvas 2D
- 物理计算：原生 JS/TS
- 数学库：Math.js / numeric.js（矩阵运算、ODE求解）
- 图表：Chart.js / D3.js（v-t图、能量条形图）

架构设计时需确认：是否采用以上方案，或有更合适的选择。

### 4.2 注册机制的具体实现

需要设计三个注册表的 API：
- 实体注册：`registerEntity(type, definition)` — 注册实体类型和默认属性
- 求解器注册：`registerSolver(pattern, solver)` — 注册"实体+关系组合模式"到求解函数的映射
- 渲染器注册：`registerRenderer(type, renderer)` — 注册实体类型到渲染函数的映射

关键问题：求解器的 pattern 如何表达？简单字符串匹配还是结构化描述？

### 4.3 Preset JSON Schema

需要定义预设文件的完整字段，至少包含：
- 元信息（id、名称、所属模块）
- 实体列表（类型、属性、位置）
- 关系列表（类型、关联的实体）
- 参数 schema（哪些属性暴露给用户调节，各自的 min/max/step/default）
- 支持的视角列表
- 功能开关（正交分解、摩擦切换、多物体对比等）

### 4.4 视角层的实现方式

6个视角（受力/运动/能量/动量/场/电路）：
- 视角如何与预设绑定？预设声明支持哪些视角？
- 不支持的视角点击时显示什么？
- 视角叠加时的渲染顺序和透明度控制

### 4.5 模拟引擎的运行循环

```
实体 + 关系 + 参数
    ↓
受力分析引擎（遍历关系 → 计算每个物体的合力）
    ↓
运动求解器（合力 → 加速度 → 数值积分 → 位置/速度更新）
    ↓
事件检测（碰撞？离开表面？进入/离开场区域？到达终态？）
    ↓
渲染（当前视角层决定显示哪些物理量）
```

需要定义：
- 这个循环的代码组织方式（是一个 Simulator 类？还是函数式管道？）
- 解析解场景和数值积分场景的统一接口
- 时间轴控制（播放/暂停/拖拽/调速）如何与模拟引擎交互

### 4.6 参考实现的范围

框架初始化阶段需要完成一个最简参考实现，作为两位开发者的开发范本。建议用"水平面上的物块受力"：
- 包含：一个物块实体 + 一个水平面实体 + 接触关系
- 展示：受力视角（重力+支持力箭头）
- 可调参数：质量
- 覆盖全链路：预设加载 → Entity注入 → 求解器匹配 → 渲染 → 参数面板联动

## 五、项目推进路线

```
当前 → 架构设计
        产出：技术栈 + 目录结构 + 核心类型 + 注册机制 + Preset Schema + 参考实现规范
        ↓
      框架初始化（一人完成或一起完成）
        产出：项目骨架 + core/ 实现 + shell/ 实现 + renderer/primitives/ + 参考预设
        ↓
      并行开发（两人各自推进，互不等待）
        开发者 A：从 P-01"斜面下滑"开始，逐步扩展力学域
        开发者 B：从 P-08"电场可视化"开始，逐步扩展电磁域
        每做完一个预设 = 一个完整可用的功能
        ↓
      整合跨域场景
        "斜面+磁场"等跨域综合预设，两人合作完成
        ↓
      Phase 2 扩展
        新增搭建器 UI（工具栏 + 拖放 + 自动关联）
        已有求解器、渲染器、视角层全部复用
```

### 每位开发者的推进节奏

以开发者 A 为例：

```
1. P-01"斜面·下滑" → 驱动实现基础受力求解 + 物块渲染 + 受力视角 → 浏览器可跑 ✓
2. P-01 其他模型 → 复用框架，每个模型加一个预设 JSON → 逐个可跑 ✓
3. P-02 运动模拟 → 复用受力引擎，新增运动求解器 + 运动视角 + v-t图 → 可跑 ✓
4. P-12 动量守恒 → 复用运动求解器，新增碰撞求解 + 动量视角 → 可跑 ✓
5. P-14 机械能守恒 → 几乎不需新求解器，主要是能量视角 → 可跑 ✓
```

## 七、术语表

| 术语 | 含义 |
|------|------|
| Entity | 场景中的物理对象（物块、斜面、弹簧、电场区域等） |
| Relation | 实体间的物理关系（接触、连接、场作用、包含） |
| Solver | 求解器，针对特定"实体+关系"组合的物理方程计算函数 |
| Viewport / 视角层 | 控制显示哪些物理量的"滤镜"（受力/运动/能量/动量/场/电路） |
| Preset / 预设 | 一组预配置的实体+关系+参数的场景快照，加载即用 |
| Registry / 注册表 | 将实体类型、求解器、渲染器注册到系统中的机制 |
| Domain / 域 | 按物理属性划分的开发领域（mechanics = 力学域，em = 电磁域） |
| Vibe coding | 使用 Claude Code 进行开发的工作模式，特点是边做边验证、快速迭代 |
