## 三、架构层面差距分析：场景组织模式

### 3.0 核心问题：需求期望 vs 当前实现

**需求文档设想的交互流程**：

```
老师操作：选择「水平面」模型（1个统一场景）
  → 左侧面板出现约束条件：
      接触面：○ 光滑  ● 粗糙
      运动状态：● 静止  ○ 匀速  ○ 加速
      外力：○ 无  ● 有 → 方向：水平/斜向上/斜向下
  → 右侧受力图自动更新（力的种类、方向随条件实时变化）
  → 切换"静止→加速"：摩擦力方向和大小自动重算，无需换场景
```

**当前实现的交互流程**：

```
老师操作：从预设列表选择
  · 水平面·物块受力（静止，无外力）     ← 独立预设
  · 水平面·施加外力（有外力，可调角度）  ← 独立预设
  · 水平面·摩擦减速（有初速度）         ← 独立预设
  · 水平面·推力加速（有推力）           ← 独立预设
  → 每个预设绑定专用求解器，参数可调但场景结构不可变
  → 想从"静止"切到"加速"？必须退出当前预设，选另一个
```

**差距总结**：

| 维度 | 需求设想 | 当前实现 |
|------|---------|---------|
| 场景粒度 | 1个"水平面"场景，内部条件组合 | 4个独立预设 |
| 交互方式 | 选条件 → 系统自动推导受哪些力 | 选预设 → 参数微调 |
| 切换成本 | 同一场景内切换条件，零成本 | 需退出再选另一个预设 |
| 力的增减 | 动态——开/关摩擦自动增减摩擦力 | 固定——预设决定了力的集合 |
| 智能程度 | 条件驱动，系统"理解"物理 | 模式匹配，求解器按类型计算 |

### 3.0.1 现有架构的能力盘点

经过代码研究，发现**现有架构已经具备实现"统一场景"的基础设施**，只是没有按需求意图来组织：

| 基础设施 | 是否具备 | 现状 |
|----------|---------|------|
| `select` 参数类型（下拉选择） | ✅ 已有 | `ParamSchema` 支持 `type: 'select'`，有 `options[]` |
| `toggle` 参数类型（布尔开关） | ✅ 已有 | 已用于 `frictionToggle`、`showDecomposition` |
| 求解器内多分支逻辑 | ✅ 已有 | `block-on-slope.ts` 已在一个函数内处理 4 种场景 |
| 参数变化触发重新求解 | ✅ 已有 | `paramValues` 变化 → 重新计算 |
| 动态力列表 | ✅ 已有 | 求解器根据条件 `push/不push` 摩擦力等 |

**关键发现**：`block-on-slope.ts` 实际上已经是"统一求解器"模式——一个函数根据 `initialVelocity`、`mu` 等参数值走不同物理分支。但在预设层面，仍被拆成了 4 个独立预设 + 4 个独立 qualifier。

### 3.0.2 改进方案对比

#### 方案 A：统一场景 + 条件选择参数（推荐）

**核心思路**：将同类预设合并为 1 个"类别预设"，用 `select`/`toggle` 参数替代 qualifier 分发。

**改动范围**：
- 预设 JSON：合并同类为 1 个，增加 `select` 参数
- 求解器：合并同类为 1 个统一求解器（斜面已基本就绪）
- 参数面板：已支持 `select` 渲染，无需改动
- 公共代码：**无需改动**

**水平面示例**：

```
当前：4 个预设 × 4 个求解器（含 qualifier 变体）
改为：1 个预设「水平面受力」+ 1 个统一求解器

预设参数面板：
  ┌─────────────────────────┐
  │ 质量 m     [====●===] 2kg│
  │ 接触面     ○光滑 ●粗糙   │  ← select 或 toggle
  │ 摩擦因数 μ [==●=====] 0.3│  ← 选"光滑"时隐藏或禁用
  │ 外力       ○无 ●有       │  ← toggle
  │ 外力大小 F [===●====] 10N│  ← 选"无"时隐藏
  │ 外力方向 θ [==●=====] 30°│  ← 选"无"时隐藏
  │ 初速度 v₀  [●=======] 0  │  ← 0=静止，>0=运动
  └─────────────────────────┘

求解器根据参数组合自动推导：
  光滑+静止+无外力     → G, N
  粗糙+静止+水平推力   → G, N, f, F
  粗糙+有初速+无外力   → G, N, f（减速）
  ...任意组合均可处理
```

**斜面示例**：

```
当前：4 个预设（static/sliding-down/sliding-up/smooth）
改为：1 个预设「斜面受力」

参数面板新增：
  接触面     ○光滑 ●粗糙
  初速度 v₀  [slider]     ← 0=从静止开始，>0=冲上/初始速度

求解器逻辑（block-on-slope.ts 已基本具备）：
  v₀=0 + μ>0 + tanθ≤μ  → 静止平衡
  v₀=0 + μ>0 + tanθ>μ  → 加速下滑
  v₀>0                  → 冲上减速（可能反向下滑）
  μ=0                   → 光滑下滑
```

**工作量估算**：
- 水平面：合并 4→1 预设，重写 1 个统一求解器 — **1~2 天**
- 斜面：合并 4→1 预设，求解器已基本统一，微调 — **0.5~1 天**
- 悬挂：保持现状（单绳/双绳/绳杆是不同拓扑，不适合合并） — **0 天**
- 后续阶段：新模型直接按统一场景模式设计 — **无额外成本**

**优点**：
- 完全对齐需求文档的交互设想
- 预设数量大幅减少（水平面 4→1，斜面 4→1）
- 用户体验连贯（同一场景内切换条件）
- 公共代码无需改动
- 可渐进实施（先改水平面，验证后再改斜面）

**缺点**：
- 统一求解器逻辑更复杂（需处理所有条件组合）
- 需要处理"参数联动显隐"（如选"光滑"时μ参数隐藏）

#### 方案 B：保持多预设，但增加"场景分组"UI

**核心思路**：不改预设结构，只在 UI 层将同类预设分组显示。

```
预设面板：
  ┌─ 水平面 ─────────────────┐
  │  · 静止平衡              │
  │  · 施加外力              │
  │  · 摩擦减速              │
  │  · 推力加速              │
  ├─ 斜面 ───────────────────┤
  │  · 静止平衡              │
  │  · 自由下滑              │
  │  ...                     │
  └──────────────────────────┘
```

**工作量**：~1 天（仅 UI 分组）
**优点**：改动最小
**缺点**：没有解决核心问题——老师仍需切换预设才能改变条件

#### 方案 C：通用约束求解引擎

**核心思路**：实现真正的"选条件→自动推导受力"引擎。

**工作量**：2~3 周，接近重写求解层
**优点**：完美对齐需求
**缺点**：远超 Phase 1 范围，风险高

### 3.0.3 方案推荐

**推荐方案 A**，理由：
1. 已有基础设施支撑（`select` 参数 + 求解器多分支 + 参数联动重求解）
2. 工作量可控（~3 天改造已有的 8 个预设 + 2 个统一求解器）
3. 完全对齐需求文档的交互设想
4. 公共代码零改动，风险低
5. 后续阶段新模型直接按此模式设计，无额外负担

**但有一个前置问题需要解决**：参数联动显隐——当选择"光滑"时，μ 参数应该隐藏或禁用。当前 `ParamSchema` 没有 `visibleWhen` / `enabledWhen` 条件字段。这是方案 A 需要的唯一公共代码扩展。

## 五、已实现预设的质量审查

### 5.1 参数范围对比

| 参数 | 需求范围 | 实际范围 | 差异 |
|------|----------|----------|------|
