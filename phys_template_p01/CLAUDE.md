## 产品定位

为中国初高中物理教师构建的**统一物理模拟器**，所属 EduMind AI 互动课件平台 · 物理学科教具。

**核心价值**：老师从预设场景库加载场景、调整参数、切换视角，3 分钟内复现一道物理题的完整过程。

**核心理念**：物理世界没有"力学工具"和"电磁工具"的边界。一道高考综合题可能同时涉及斜面下滑 + 进入磁场做圆周运动。因此所有物理域共存于同一实体系统和渲染引擎，**一个模拟器覆盖全部物理域**。

**设计原则**：
- 统一实体系统：场景中的一切都是 Entity，通过 Relation 描述物理关系
- 视角即滤镜：受力/运动/能量/动量/场/电路 6 种视角，不改数据只改显示
- 预设驱动：高中物理题型高度模式化，用枚举法覆盖，不做通用物理引擎
- 目标设备：1920×1080 投影仪为主，兼容 1280×720 和 iPad，全部 2D

## 第一层：核心规则

### 核心规则A：规则优先级
- 触发条件：出现规则冲突或路径选择不确定。
- 动作：路径裁决按 `用户显式指令 > 风险分级自动流转 > 当前模式默认路径` 执行；安全与验证门禁仍受核心规则约束。
- 输出：明确说明当前采用的规则来源。
- 验收：不存在同轮反复索要模式口令的机械往返。

### 核心规则B：文档优先与任务归并
- 触发条件：开始新工作，或用户提出追加需求/bug反馈。
- 动作：先读取当前任务文档；同一任务续作默认并入当前任务，除非用户明确要求新任务。
- 输出：在任务文档记录关键用户原文与决策。
- 验收：无无依据新建任务；关键决策可追溯。

### 核心规则C：按计划执行与偏离处理
- 触发条件：进入执行模式后发现计划外改动需求。
- 动作：停止执行并返回计划模式补齐计划，不允许直接偏离实现。
- 输出：记录偏离原因与补充计划条目。
- 验收：执行内容与计划逐项可映射。

### 核心规则D：修改后检查门禁
- 触发条件：对项目文件产生修改。
- 动作：按受影响目录命中"附录F·最小回归门禁矩阵"中配置的检查命令执行回归检查；若命中数据链路，追加受影响模块的 dry-run 验证。
- 输出：在回复中给出检查命令与结果。
- 验收：本轮改动均有对应验证证据。

### 核心规则E：数据任务校验
- 触发条件：涉及数据读取、转换、映射、ID处理。
- 动作：先验证真实数据样本的字段名、类型、嵌套层级；禁止字段名推测。
- 输出：在任务文档记录数据结构样例与关键字段映射。
- 验收：出现数据结构不一致时立即停止执行并回到计划模式。

### 核心规则F：数据库安全
- 触发条件：涉及数据库清理、重建或破坏性语句。
- 动作：先备份，再二次确认清除范围；优先非破坏性方案。
- 输出：在任务文档记录备份位置与确认记录。
- 验收：无未备份的 `DROP/TRUNCATE` 等破坏性操作。

### 核心规则G：防重复确认
- 触发条件：用户已表达执行确认意图（如“确认执行/按计划修复/开始改/继续/执行”）。
- 动作：直接进入执行，不重复要求“进入执行模式”口令。
- 输出：简要确认将按既定计划执行。
- 验收：同一阶段不出现重复确认循环。

### 核心规则H：风险分级驱动流程
- 触发条件：MODE 0 任务评估完成后。
- 动作：必须先判定风险等级（L0-L3），再按等级映射的最小流程路径流转。
- 输出：在任务文档记录“风险等级 + 流程路径 + 强制门禁”。
- 验收：所有执行任务均可追溯到风险等级与对应路由。

### 核心规则I：交付DoD模板化
- 触发条件：进入 MODE 6，或需要对本轮改动给出交付结论。
- 动作：按统一DoD模板输出“变更摘要、验证结果、风险与回滚、任务完成质量模块”。
- 输出：给出可追溯命令证据与质量结论；L2-L3补充高风险检查结论。
- 验收：DoD字段完整，且与执行证据一致。

## 第二层：流程规则（RIPER-7）

### MODE 0：任务确认模式
- 用途：任务生命周期入口（创建、切换、暂停、恢复）。
- 必须行动：
  1. 生成任务ID（`月-日-时-分-任务关键词`）。
  2. 在 `.tasks/active/` 创建任务文档（复杂任务完整文档，简单任务轻量记录）。
  3. 读取 `.tasks/task_evaluation_framework.md` 完成"任务类型 + 风险等级（L0-L3）"评估。
  4. 简单任务先输出"任务理解+执行计划"，待用户确认后执行。
  5. 按风险等级路由到后续模式（L0可直达执行，L1-L3按矩阵执行）。
  6. 续作优先补充到现有任务文档。

### MODE 1：研究模式
- 用途：仅信息收集。
- 允许：读取文件、澄清问题、理解结构。
- 禁止：提出实现建议、写代码、做最终规划。
- 数据强制：查看真实样本、验证完整结构、记录字段和嵌套关系。

### MODE 2：创新模式
- 用途：头脑风暴可选方案。
- 允许：讨论想法与优缺点。
- 禁止：定案、具体实现、代码编写。

### MODE 3：计划模式
- 用途：输出可执行技术计划。
- 要求：编号清单；精确到文件路径与改动点。
- 数据前置：必须先分析真实数据结构，禁止基于命名假设。

### MODE 4：执行模式
- 用途：严格实现MODE 3计划。
- 进入条件：用户明确要求执行，或出现执行确认意图，或风险路由命中执行阶段。
- 禁止：计划外创造性添加。
- 数据前置：执行前再次核对数据字段名/类型/嵌套层级。

### MODE 5：审查模式
- 用途：验证实现是否与计划一致。
- 输出格式：
  - 偏离：`:warning: DEVIATION DETECTED: [描述]`
  - 结论：`:white_check_mark: 实现完全匹配计划` 或 `:cross_mark: 实现偏离计划`

### MODE 6：任务结束模式
- 用途：任务收尾与归档。
- 必须行动：
  1. 仅在对应项目文件改动时执行格式化。
  2. 仅用户明确要求时执行Git提交。
  3. 检查临时文件清理。
  4. 提供下一步建议。
  5. 按DoD模板输出交付结论（变更摘要、验证结果、风险回滚、质量模块）。

## 风险分级与最小流程路径（L0-L3）

| 等级        | 判定条件（命中任一即升级）                                | 最小流程路径                                             | 必须产物                         | 强制门禁                                           |
| ----------- | --------------------------------------------------------- | -------------------------------------------------------- | -------------------------------- | -------------------------------------------------- |
| L0 低风险   | 单文件微调；不改业务逻辑；不涉数据/数据库/部署            | MODE 0 -> MODE 4 -> MODE 6                               | 轻量任务记录、变更摘要           | 改代码时执行命中目录最小回归；纯文档可记录跳过理由 |
| L1 常规风险 | 1-3文件改动；常规功能/配置修复；无高风险数据链路          | MODE 0 -> MODE 3 -> MODE 4 -> MODE 5 -> MODE 6           | 执行计划、验证证据、审查结论     | 命中目录最小回归 + DoD模板化交付                   |
| L2 高风险   | 涉及数据读取/转换/映射/ID；跨模块联动；外部接口变更       | MODE 0 -> MODE 1 -> MODE 3 -> MODE 4 -> MODE 5 -> MODE 6 | 数据结构样本、字段映射、风险说明 | L1全部 + 数据契约校验 + 受影响模块dry-run验证      |
| L3 关键风险 | 涉及数据库破坏性操作、生产路径/环境切换、系统级高影响改动 | MODE 0 -> MODE 1 -> MODE 3 -> MODE 4 -> MODE 5 -> MODE 6 | 分阶段计划、回退方案、确认记录   | L2全部 + 数据库安全强制 + 交付自检强制             |

## 自动流转规则（风险路由版）
1. 新任务且目标明确：完成 MODE 0 后，按风险等级自动路由到下一模式。
2. L0 默认可直接执行；L1 默认进入计划；L2-L3 默认先研究再计划。
3. 研究完成后自动进入计划；计划确认后进入执行；执行完成后进入审查。
4. 续作/bug反馈默认并入当前任务，先给“问题理解 + 处理计划 + 执行确认请求”。
5. 用户出现执行确认意图（`确认执行/按计划修复/开始改/继续/执行`）时，直接进入执行，不重复索要口令。
6. 用户显式模式指令始终优先；若与自动流转冲突，以显式指令为准并记录原因。
7. 若显式指令试图绕过高风险门禁（数据校验/数据库安全），必须保留门禁并说明依据。
8. 禁止机械化往返：同一阶段已确认后，不回退索要模式切换口令。

### J. 并行开发约束

#### J1. 新增预设标准流程
新增预设时，必须阅读并遵循 `docs/开发者指引.md` 中的 5 步流程。

#### J2. 命名规范
- **文件命名**：实体 `{type}.ts`、求解器 `{scene-description}.ts`、渲染器 `{type}-renderer.ts`、视角 `{viewport}-viewport.ts`、预设 `{scene-description}.json`
- **预设 ID 格式**：`{模块号}-{模型ID}-{变体}`，如 `P01-FM001-horizontal-block`
- **求解器 ID 格式**：`{domain}-{description}`，如 `mech-block-on-horizontal-surface`

#### J3. 视觉一致性约束
- 力的颜色必须使用 `src/core/visual-constants.ts` 中的 `FORCE_COLORS`，禁止在渲染器中硬编码力颜色
- 力箭头从实体边缘出发（非中心），由 `force-viewport.ts` 中 `getEdgeStart()` 统一处理。新增实体类型时需确保 `properties` 中包含几何信息（矩形：`width`+`height`，圆形：`radius`），箭头起点计算自动适配

#### J4. 公共代码修改流程
- 域开发者可能需要修改公共代码的场景：新增 `EntityCategory`、新增 `ViewportType`、新增 `ForceType` 颜色
- 修改前：在任务文档记录需求和理由
- 原则：**只追加不修改**已有导出，不删除已有接口字段

#### J5. 类型扩展约定
- `core/types.ts` 只能追加，不能修改已有接口
