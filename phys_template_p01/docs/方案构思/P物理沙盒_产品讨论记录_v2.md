## 一、跨域题目的支持问题

### 关注点

如果一道题目同时涉及力学和电磁学（如带电粒子从斜面滑下后进入磁场做圆周运动），按v1方案分了力学沙盒和电磁学沙盒，能在同一个画布上呈现吗？

### 结论

v1方案不能支持。这直接推动了v2方案的核心转变：从"按物理域分沙盒"改为"一个统一模拟器"。

跨域综合题在高考中很常见，且恰恰是最难、最需要可视化辅助的题目类型：

| 题型                          | 涉及的域          |
| ----------------------------- | ----------------- |
| 斜面/平抛 + 进入磁场做圆周    | 力学 + 电磁       |
| 带电粒子在重力场+电场中运动   | 力学 + 电磁       |
| 电磁感应导体棒的运动+能量守恒 | 电磁 + 力学(能量) |
| 弹簧振子 + 电磁感应           | 振动 + 电磁       |

**结论：不应该按物理域分产品边界。物理世界本身没有"力学工具"和"电磁工具"的隔离，"力学/电磁/波动"只应该是渲染层的视角切换，不是产品层的隔离。**

## 二、约束求解器和碰撞检测的必要性

### 关注点

方案中Phase 3提到了"通用约束求解器"和"任意实体组合的碰撞检测"。这两个分别是什么场景下要用到？在初高中物理教学中，作用大吗，要经常用到吗？

### 约束求解器

**解决什么问题**：当场景中有多个物体通过绳/杆/弹簧/接触面相互连接时，需要同时求解所有物体的受力和运动。

**举例**：阿特伍德机——两个物块通过绳子绕过滑轮连接，需要联立方程组求解加速度和绳子张力。

**教学中的频率**：需要"约束求解"的题型（连接体、阿特伍德机、传送带、人船模型）确实常见，但这些题型的物理模型是有限的、可穷举的。每种模型都可以硬编码其方程，不需要通用求解器。

**通用约束求解器只在老师自由组合出系统预设里没有的连接方式时才真正需要，这种场景在教学中频率极低。**

### 碰撞检测

**解决什么问题**：模拟运行过程中，检测物体是否碰到了另一个物体、是否到达了表面边缘、是否进入了某个场区域。

**举例**：物块从斜面飞出后做平抛运动，需要检测抛物线轨迹与水平面的交点。

**教学中的频率**：需要"碰撞/事件检测"的物理过程确实常见（两物块碰撞、平抛落地、粒子进出磁场），但对于预设场景，这些检测点都可以用解析公式直接计算，不需要通用碰撞检测引擎。

**通用碰撞检测只在老师自由搭建任意几何形状时才需要，这种场景在教学中频率极低。**

### 总结

| 能力           | 什么时候需要                       | 教学中频率 |
| -------------- | ---------------------------------- | ---------- |
| 通用约束求解器 | 老师自由组合出预设里没有的连接方式 | 极低       |
| 通用碰撞检测   | 老师自由搭建任意几何形状的边界     | 极低       |

## 三、Phase 3 的方向调整

### 关注点

产品目标是"让老师更好地进行教学、学生更好理解"，不是做一个像Algodoo那样的自由搭建工具。在这个背景下，原来的Phase 3（通用约束求解器+碰撞检测+自由编辑器）还需要吗？

### 结论

**原来的Phase 3不需要。** 它追求的是"工具的通用性"，不是"教学效果更好"。

Phase 1（预设+调参）覆盖85%场景，Phase 2（结构化搭建）覆盖到95%。剩下的5%是极端复杂的题目组合，老师在课堂上本来就不太用工具演示。

**Phase 3应改为"深化教学体验"方向**，具体内容等Phase 1上线后根据老师真实使用反馈来确定。可能的方向包括：

| 方向         | 说明                                                   |
| ------------ | ------------------------------------------------------ |
| 更多预设覆盖 | 扩充预设库到200+，覆盖各省模考高频题                   |
| 教学辅助增强 | 关键公式自动推导展示、物理量因果关系标注               |
| 导出能力增强 | 导出动画GIF/视频、导出带标注的PNG用于试卷插图          |
| 专项难题模板 | 针对高频难题类型逐个做专项模板（板块模型、弹簧耦合等） |

## 四、高考压轴题的覆盖率

### 关注点

做完Phase 1、2和调整后的Phase 3，各种高考压轴题都能模拟出来吗？

### 覆盖率分析

#### 力学压轴

| 题型                  | Phase 1能覆盖？                   |
| --------------------- | --------------------------------- |
| 斜面+平面组合         | 能——两段运动解析解拼接            |
| 传送带                | 能——分段运动，每段解析可算        |
| 板块模型              | Phase 2——需要"包含"关系和相对运动 |
| 弹簧+物块飞出做平抛   | 能——两段拼接                      |
| 竖直圆周临界条件      | 能——圆周运动解析解                |
| 多过程串联（3段以上） | Phase 2——多段串联                 |

#### 电磁学压轴

| 题型                                   | Phase 1能覆盖？      |
| -------------------------------------- | -------------------- |
| 带电粒子电场偏转                       | 能——类平抛解析解     |
| 带电粒子磁场圆周                       | 能——圆弧几何解析可算 |
| 多区域运动（加速→偏转→再进入另一个场） | 能——多段拼接         |
| 电磁感应单棒                           | 能——ODE数值积分      |
| 电磁感应双棒                           | Phase 2——耦合ODE     |
| 回旋加速器                             | 能——每段解析拼接     |

#### 综合压轴

| 题型                                | Phase 1能覆盖？        |
| ----------------------------------- | ---------------------- |
| 斜面+磁场                           | 能——跨域预设           |
| 重力+电场                           | 能——等效重力场解析解   |
| 电磁感应+能量                       | 能——电磁感应预设       |
| 电磁感应+动量（碰撞后在磁场中运动） | Phase 2——碰撞+电磁耦合 |

#### 覆盖率统计

| 覆盖阶段      | 占比 | 说明                                 |
| ------------- | ---- | ------------------------------------ |
| Phase 1能覆盖 | ≈60% | 单域、单/双过程、解析解可直接算      |
| Phase 2能覆盖 | ≈35% | 多物体交互（板块、双棒）、多过程串联 |
| 仍然覆盖不了  | ≈5%  | 极少数极端复杂场景                   |

#### 覆盖不了的5%是什么

| 场景                     | 原因                                      |
| ------------------------ | ----------------------------------------- |
| 可滑动的弧形轨道+物块    | 轨道和物块运动互相耦合，不是简单分段拼接  |
| 弹簧连接两物块在粗糙面上 | 弹簧力+摩擦力同时作用，摩擦力方向可能翻转 |
| 三体以上的多约束耦合     | 复杂度过高                                |

这5%即使有通用求解器也不一定能完美模拟。如果需要覆盖，更好的方式是**逐个做专项模板**（Phase 3方向），而不是做通用引擎。

**Phase 1+2+调整后的Phase 3（补充专项模板），高考压轴题覆盖率可达98%以上。**

## 五、Phase 1 的受力分析引擎和运动求解器实现方式

### 关注点

Phase 1中的受力分析引擎和运动求解器，都是根据预设模板来的？不同的模板有不同的逻辑？

### 结论

对。Phase 1没有通用的受力分析引擎。每个预设模板内部硬编码了自己的物理方程。

| 预设           | 受力逻辑                             | 运动求解                            |
| -------------- | ------------------------------------ | ----------------------------------- |
| "斜面下滑"     | `G=mg, N=mgcosθ, f=μN, F合=mgsinθ-f` | `a=F合/m`，匀加速直线运动解析解     |
| "平抛运动"     | `G=mg`（只有重力）                   | 水平匀速+竖直自由落体，解析解       |
| "洛伦兹力圆周" | `qvB=mv²/r`                          | 匀速圆周运动，`r=mv/qB`             |
| "电磁感应单棒" | `F安=BIL, I=BLv/R`                   | `m·dv/dt = -B²L²v/R`，RK4数值积分   |
| "弹簧振子"     | `F=-kx`                              | `x=Acos(ωt+φ)`解析解，或大振幅时RK4 |

但底层仍然用Entity系统管理数据：

```
预设加载 → 往Entity系统注入实体和关系 → 同时注入这个预设专属的求解函数
```

实际工作量可控：很多预设的方程高度相似（所有斜面题的受力分析逻辑几乎一样，只是参数不同），按模型类型复用即可。

## 六、Phase 2 中的约束求解和碰撞检测

### 关注点

Phase 2允许老师拖动实体构造场景，那约束求解器和碰撞检测在Phase 2中需要吗？

### 结论

**需要这两个能力，但不需要通用版本。用枚举法就够了。**

Phase 2的搭建是结构化的——老师从菜单选择已知类型的组件，不是自由画任意形状。

#### 碰撞检测——Phase 2 只需支持的组合

| 物体 | 表面                | 检测方法                    |
| ---- | ------------------- | --------------------------- |
| 质点 | 直线（斜面/水平面） | 点到直线距离=0，解析公式    |
| 质点 | 圆弧（弧形轨道）    | 点到圆心距离=半径，解析公式 |
| 质点 | 矩形区域（场边界）  | 点是否在矩形内，坐标判断    |
| 物块 | 物块                | 矩形相交检测                |

总共几种组合，每种写一个函数。

#### 约束求解——Phase 2 只需支持的模式

| 约束类型             | 求解方式                                     |
| -------------------- | -------------------------------------------- |
| 绳连接（不可伸长）   | 两物体加速度大小相等，联立牛顿二定律         |
| 杆连接（刚体约束）   | 距离不变，速度沿杆方向分量相等               |
| 弹簧连接             | F=-kx，代入各物体运动方程做数值积分          |
| 接触面（法向约束）   | N≥0时保持接触，N<0时脱离                     |
| 滑轮（改变力的方向） | 绳两端张力相等（轻滑轮），位移关系由几何确定 |

每种约束硬编码一套求解逻辑。老师从菜单选"加一根绳子连接A和B"时，系统加载对应的约束方程模板。

#### Phase 1 和 Phase 2 的区别

|          | Phase 1                          | Phase 2                                 |
| -------- | -------------------------------- | --------------------------------------- |
| 场景来源 | 预设（整个场景的方程都预先写好） | 老师搭建（系统根据实体+关系组合出方程） |
| 碰撞检测 | 每个预设内硬编码临界条件         | 按实体类型组合查表，调用对应检测函数    |
| 约束求解 | 每个预设内硬编码物理方程         | 按约束类型查表，组合出联立方程          |

## 七、枚举法方案的权衡

### 关注点

按枚举法做（每种实体组合、每种约束类型各写一套逻辑），好处是复杂度可控，坏处是能支持的场景受限。能支持的场景就是所有实体、关系之间自由组合出来的有意义的组合数量，是吗？

### 结论

对。Phase 2能支持的场景 = 实体类型 × 关系类型 × 求解模板的有意义组合。

但不是所有组合都有物理意义（"弹簧连接两个磁场区域"没意义），有物理意义的组合是有限的：

| 组合类别         | 示例                           | 有意义的组合数 |
| ---------------- | ------------------------------ | -------------- |
| 物体+表面        | 物块在斜面上、小球在弧形轨道上 | ≈10种          |
| 物体+连接件+物体 | 两物块用绳连接、弹簧连两物块   | ≈10种          |
| 物体+场          | 带电粒子在磁场中               | ≈8种           |
| 物体+表面+场     | 导体棒在导轨上+磁场            | ≈5种           |
| 多段串联         | 斜面→平面→磁场区域             | ≈15种          |

**总计约50种有物理意义的组合**，每种写一套求解模板。

权衡总结：

- **好处**：复杂度可控；每种组合都能保证数值精确（方程是人写的，不是机器猜的）
- **坏处**：只能支持已实现的组合；老师搭出没有对应求解模板的组合时，系统只能提示"暂不支持这种场景"

从产品角度看，"暂不支持"比"通用求解器算出一个错误结果"更好——工具不支持某道题，老师画黑板就行了；工具算出错误数值，会误导学生。

## 八、Web端性能问题

### 关注点

产品是通过浏览器使用的，用Web端技术实现。当系统比较复杂（比如有电磁场和运动同时存在）时，物体的受力、运动、位置等都要实时计算，会有性能问题吗？

### 结论

**不会有性能问题。** 原因是我们的方案设计本身决定了计算量很小。

#### 计算量分析

大部分场景用解析解，给定时间t直接用公式算出位置，不需要逐帧迭代。这和游戏物理引擎（每帧模拟上千个碰撞）完全不同。

| 场景类型                          | 计算方式       | 每帧计算量                   |
| --------------------------------- | -------------- | ---------------------------- |
| 解析解场景（斜面、平抛、圆周等）  | 代入公式直接算 | 几次四则运算（微秒级）       |
| ODE数值积分场景（电磁感应单棒等） | RK4积分        | 几十次四则运算（微秒级）     |
| 最复杂场景（多段串联+场切换）     | 分段求解       | 几百次四则运算（仍是微秒级） |

浏览器每帧有16毫秒（60fps）的时间预算，我们的物理计算连0.1毫秒都用不到。

#### 性能瓶颈在渲染而非计算

| 环节                        | 我们的情况             | 性能压力 |
| --------------------------- | ---------------------- | -------- |
| 物理计算                    | 解析公式/简单ODE       | 极低     |
| 2D渲染（Canvas）            | 几个物体+力箭头+轨迹线 | 低       |
| 场可视化（电场线/磁场符号） | 静态网格标记           | 低       |
| 动画（60fps刷新）           | 更新几个物体位置+重绘  | 低       |

场景复杂度上限大概是：几个物体+几种力的箭头+轨迹线+场区域标记。这对现代浏览器的Canvas 2D毫无压力。PhET的模拟器就是Web端运行的（HTML5+Canvas），场景复杂度和我们类似，在各种设备上都流畅运行。

#### 需要注意的点

| 场景         | 原因                           | 应对                                               |
| ------------ | ------------------------------ | -------------------------------------------------- |
| 长时间轨迹线 | 运行时间很长时轨迹点会积累很多 | 设置轨迹点上限（如最多2000个点），超出后丢弃最早的 |
| iPad低端设备 | 老款iPad的Canvas性能较弱       | 适当降低渲染精度（场线密度等），不影响计算精度     |

## 九、与PhET的异同

### 关注点

我们现在的实现思路，跟PhET比较像，是吗？

### 结论

**核心思路相似，但有关键区别。**

#### 相似点

| 维度           | PhET                             | 我们                           |
| -------------- | -------------------------------- | ------------------------------ |
| 技术平台       | Web端（HTML5+Canvas）            | Web端                          |
| 产品定位       | 教学辅助工具                     | 教学辅助工具                   |
| 物理求解       | 每个模拟器内硬编码自己的物理方程 | 每个预设内硬编码自己的物理方程 |
| 不追求通用引擎 | 对                               | 对                             |
| 2D为主         | 绝大多数是2D                     | 全部2D                         |

核心相似点：**都是"每个场景自己写自己的物理方程"，而不是做通用物理引擎。**

#### 关键区别

| 维度        | PhET                                           | 我们                                       |
| ----------- | ---------------------------------------------- | ------------------------------------------ |
| 产品形态    | 每个模拟器是**独立应用**，各自独立运行         | **一个统一模拟器**，所有场景在同一个画布上 |
| 跨域能力    | 不支持——力学和电磁是独立的模拟器               | 支持——一个场景可以同时有斜面和磁场         |
| 底层架构    | 没有统一的数据模型，每个模拟器自己定义数据结构 | 统一的Entity系统管理所有实体和关系         |
| Phase 2搭建 | 不支持——用户只能调参                           | 支持——老师可以从菜单拖入实体组合新场景     |
| 目标用户    | 全球通用，覆盖初中到大学                       | 中国初高中物理教学，贴合课标和高考题型     |

**一句话概括：我们相当于PhET的升级版——保留了"每个场景硬编码物理方程"的务实做法，但解决了PhET的两个核心问题：场景之间是孤岛（不能跨域）、用户只能调参不能搭建。**

## 十、2D渲染与3D需求

### 关注点

方案全部是2D渲染。初高中物理教学中，涉及3D的题目或知识点多吗？原始需求中有涉及3D的吗？

### 结论

**初高中物理教学中几乎没有需要3D渲染的内容。物理沙盒用2D是正确的选择。**

#### 物理教学内容的维度分析

| 领域      | 典型内容                     | 维度         |
| --------- | ---------------------------- | ------------ |
| 力学      | 斜面、平抛、圆周、碰撞       | 2D           |
| 电磁学    | 电场偏转、磁场圆周、电磁感应 | 2D           |
| 能量/动量 | 功能关系、碰撞模型           | 2D           |
| 光学      | 折射、反射、透镜             | 2D（光路图） |
| 热学      | 气体定律、分子运动           | 2D示意即可   |

唯一涉及"空间方向"的是安培力/洛伦兹力的方向判断（左手定则、右手定则），但这些在考试中也是通过2D截面图来出题的（标注"×"表示向纸面内、"·"表示向纸面外）。

**高考物理没有真正需要3D渲染才能理解的题目。**

#### 原始需求中的3D内容

需要3D的是其他产品线，与物理沙盒无关：

| 产品                | 3D内容                   | 产品线 |
| ------------------- | ------------------------ | ------ |
| M-01 立体几何展示台 | 几何体3D渲染、旋转、截面 | 数学   |
| M-06 向量运算演示台 | 空间向量、叉积可视化     | 数学   |
| C-02 分子结构查看器 | 3D球棍模型               | 化学   |
| C-05 晶体结构查看器 | 3D晶胞模型               | 化学   |

物理产品线（P系列）的原始需求中没有任何3D需求。

## 十一、电磁感应导体棒场景的覆盖

### 关注点

物理中有一种题目，在两根导轨上放一根导体棒，把电、运动、力、电磁等知识点结合在一起。现在的设计能模拟这种场景吗？

### 结论

**能模拟。这就是"电磁感应单棒"预设，Phase 1就能覆盖。**

#### 物理过程

导体棒在两根平行导轨上，外接电阻，处于垂直磁场中：棒切割磁感线→产生感应电动势→回路中有电流→电流在磁场中受安培力→棒减速（或加速）→速度变化→电动势变化→电流变化→力变化。力、运动、电、磁全部耦合在一起。

#### 求解方式

| 项目     | 内容                 |
| -------- | -------------------- |
| 预设名   | "电磁感应单棒"       |
| 受力逻辑 | `F安=BIL`，`I=BLv/R` |
| 运动方程 | `m·dv/dt = -B²L²v/R` |
| 求解方式 | RK4数值积分          |
| 覆盖阶段 | Phase 1              |

老师可调参数：磁感应强度B、棒质量m、导轨间距L、外接电阻R、初速度v₀、摩擦系数等。画面上能同时展示：棒的运动、力的箭头、电流方向和大小、速度-时间图、电流-时间图。

#### 双棒变体

两根导体棒都能动（一根推动，另一根被感应电流驱动）的更复杂场景，是Phase 2覆盖的，因为涉及耦合ODE。

| 场景         | 阶段    |
| ------------ | ------- |
| 电磁感应单棒 | Phase 1 |
| 电磁感应双棒 | Phase 2 |

## 十二、Phase 1预设中实体的交互方式

### 关注点

Phase 1中实体是预设的，不能自由增加实体。但为什么不能选择实体？如果能选择的话，老师使用起来会更自然——直接在画面上点击想修改的实体，而不需要看完所有参数面板再建立参数与画面的对应关系。

### 结论

**Phase 1中的实体应该能点击选中。** 正确的交互方式是：画面上看到什么、点什么、改什么。

预设加载后，画面上的每个实体（导体棒、导轨、电阻、磁场区域）都能点击选中，选中后右侧显示这个实体的属性面板。

#### Phase 1的交互边界

| 能做                             | 不能做                 |
| -------------------------------- | ---------------------- |
| 点击选中任意实体                 | 新增实体               |
| 查看和修改选中实体的参数         | 删除实体               |
| 拖动实体调整位置（如果预设允许） | 改变实体之间的连接关系 |

Phase 1的限制只是"不能增删实体和改变连接关系"，不是"不能选择实体"。

## 十三、物理沙盒的核心模块设计

### 关注点

实体能选中，在设计上需要Selection模块。除了Entity和Selection，还需要其他模块吗？跟几何编辑器的七系统架构是什么关系？

### 结论

**物理沙盒需要4个核心模块，比几何编辑器简单得多。**

#### 需要的模块

| 模块      | 作用                       | 与几何编辑器对比                     |
| --------- | -------------------------- | ------------------------------------ |
| Entity    | 管理所有实体数据           | 类似，已在方案中                     |
| Selection | 点击选中实体，联动属性面板 | 更简单——不需要多选、框选             |
| Renderer  | 把实体画到Canvas上         | 更简单——形状种类少，不需要自由变换   |
| Simulator | 物理求解+动画循环          | **几何编辑器没有这个**，物理沙盒独有 |

#### 几何编辑器有但物理沙盒不需要的

| 模块                        | 为什么不需要                                                            |
| --------------------------- | ----------------------------------------------------------------------- |
| Command（撤销/重做）        | Phase 1只是调参和运行，不需要复杂操作历史                               |
| Tool（画点/画线等多种工具） | Phase 1没有绘制操作，Phase 2也只是从菜单拖入                            |
| Event（复杂事件分发）       | 交互简单，标准的点击/拖拽事件处理就够                                   |
| Property（属性计算引擎）    | 几何编辑器要算角度、面积等派生属性；物理沙盒的属性计算在Simulator里完成 |

#### 实体选中的实现方式

每个实体在Canvas上有自己的渲染区域，点击时做命中检测（hit test）：

```
用户点击画布坐标 (x, y)
→ 遍历所有实体，判断 (x, y) 是否落在该实体的区域内
→ 命中 → 标记为选中状态 → 右侧显示该实体的属性面板
```

不同实体类型的命中检测：

| 实体类型 | 渲染形状           | 命中判断          |
| -------- | ------------------ | ----------------- |
| 导体棒   | 线段               | 点到线段距离<阈值 |
| 导轨     | 两条平行线         | 点到线距离<阈值   |
| 物块     | 矩形               | 点在矩形内        |
| 磁场区域 | 矩形（带×或·标记） | 点在矩形内        |
| 弹簧     | 锯齿线             | 点到路径距离<阈值 |
| 斜面     | 三角形             | 点在三角形内      |

这和立体几何编辑器的Selection系统是同一个思路，在Web Canvas开发中是常规实现，没有技术难点。

## 十四、方案总结与选型依据

### 关注点

确认对整体方案的理解：我们做的是有场景限制的物理场景模拟器，每个场景的公式都是预定义好的，通过梳理所有实体、关系、求解模板的组合来覆盖教学场景。之所以用这种方案，是因为做通用物理模拟器太复杂了，对吗？

### 方案理解确认

理解完全正确。整体方案可以概括为：

1. 梳理清楚所有的实体类型、关系类型、求解模板的有意义组合
2. 为每种组合预先编写精确的物理计算公式
3. Phase 1：预设场景，场景和公式的关系是预先绑定的
4. Phase 2：老师自己搭建场景，系统根据使用的实体和创建的关系自动匹配对应的求解模板
5. 目标：覆盖中国大陆初中、高中物理学科的各种教学需要的可视化场景

### 选型依据

选择枚举方案而非通用物理引擎，**不只是因为通用引擎"太复杂做不出来"，更根本的原因是初高中物理教学场景本身就是可枚举的，枚举方案就是最合适的方案。**

具体原因：

| 维度       | 通用物理引擎                      | 枚举方案                             |
| ---------- | --------------------------------- | ------------------------------------ |
| 复杂度     | 极高，不一定能做出来              | 可控，按组合逐个实现                 |
| 计算精度   | 近似求解，可能不精确              | 每个场景的方程都是人写的，物理上精确 |
| 教学适用性 | 算出不精确的数值会误导学生        | 精确结果可用于教学                   |
| 性能       | 通用碰撞检测+约束求解每帧计算量大 | 解析公式/简单ODE，计算量极小         |
| 使用体验   | 自由度太高，老师不知道怎么用      | 预设+调参，老师上手简单              |

**教学场景可枚举的依据**：知识点是课标规定的（有限的）、题型是有规律的（高频题型可穷举）、老师需要的是精确的物理计算结果而不是"看起来像"的近似模拟。

**因此枚举方案不是"做不了通用引擎的妥协"，而是教学工具的正确选择。**

## 十五、仓库策略：单仓库还是多仓库

### 关注点

项目涉及14个物理模块（P-01到P-14），是否需要按模块拆成多个独立仓库？

### 结论

**一个 Monorepo，通过目录结构隔离边界。**

核心原因：产品方案V2的设计理念是"一个统一的物理模拟器"，P-01/P-02/P-08/P-12/P-13/P-14 六个 P0 模块共享 Entity System、物理求解器、渲染引擎和视角层。拆成多仓库会直接破坏统一架构——一道跨域综合题（斜面+磁场）需要力学和电磁两个域的实体同时存在于同一画布上。

**唯一可能独立的是 P-04 电路搭建器**，因为它的交互模式（拖拽连线搭电路）和计算引擎（基尔霍夫方程组）与统一模拟器完全不同。但考虑到共享 UI 组件库和类型定义基础设施，建议仍然在 monorepo 内作为独立目录存在，未来如需独立部署可零成本拆出。

P-13 电磁感应的电路视角与 P-04 的交叉点：统一模拟器内部需要一个轻量级电路求解模块（仅支持串并联+欧姆定律），与 P-04 完整电路搭建器是不同层次的实现。

## 十六、并行开发分工：为什么按物理域纵切而非按技术层横切

### 关注点

两位开发者使用各自的 Claude Code 并行开发，如何分工才能让边界清晰、互不阻塞？

### 最初的方案：按技术层横切

最初考虑的分工是"一人做引擎层（实体系统+物理求解器），一人做交互层（UI+渲染+视角）"。

### 问题

这种横切方案在 Claude Code vibe coding 场景下有三个致命问题：

1. **每个功能都需要两人对接才能跑起来**。vibe coding 的核心工作方式是"写完就跑，从功能角度看对不对"。横切后一人做引擎、一人做UI，单独都跑不起来，无法独立验证。
2. **接口契约在 vibe coding 中一定会频繁变动**。Vibe coding 是边做边发现"这里应该这样"，不是提前把接口设计到完美。横切时接口一变两人都要改，互相阻塞。
3. **进度不同步产生等待**。两人工作时间和进度不一定一致，"A做完引擎等B做UI"或"B做完UI等A的引擎"是常态。

### 最终方案：按物理域纵切

| | 开发者 A：力学域 | 开发者 B：电磁域 |
|--|-----------------|-----------------|
| 负责模块 | P-01、P-02、P-12、P-14 | P-08、P-13、P-04 |
| 交付形态 | 完整可用的功能（引擎+渲染+UI+预设） | 同左 |
| 独立验证 | 每个预设做完就能在浏览器里跑 | 同左 |
| 互不阻塞 | 力学域和电磁域在 Phase 1 中无交叉依赖 | 同左 |

每个人交付的是"完整的功能"而不是"半成品等对接"，完全契合 vibe coding 的工作方式。

## 十七、框架基础设施如何避免并行开发冲突

### 关注点

项目框架初始化后的基础设施（Entity System、UI骨架、渲染基类），在并行开发过程中一定会被两个人改到。如何避免冲突？

### 结论

**用插件式注册机制替代集中式定义，让基础设施"只加不改"。**

冲突的本质是两个人同时修改同一个文件的同一段代码。解决方案不是"不让改"，而是设计成不需要改已有代码就能扩展：

**类型定义——各自注册而非集中定义**：`core/` 只提供实体注册机制，不列举所有实体类型。开发者A在 `domains/mechanics/entities/` 注册物块、斜面，开发者B在 `domains/em/entities/` 注册电场、磁场，互不接触同一个文件。

**渲染能力——插件式而非基类继承**：`renderer/` 只提供注册机制和基础绘制 primitive。各域的具体渲染逻辑在各自 `domains/xxx/renderers/` 下注册，不需要修改任何公共文件。

**UI 面板——配置驱动而非硬编码**：参数面板读取预设 JSON 中的参数 schema 自动渲染（slider/input/toggle等），不需要为每个预设写 UI 代码。加新预设 = 加新 JSON 文件。

### 遇到基础设施不够用怎么办

| 情况 | 处理方式 | 冲突风险 |
|------|---------|---------|
| 需要新的绘制 primitive | 在 `renderer/primitives/` 下加新文件，不改已有文件 | 无 |
| 需要新的 UI 组件 | 在 `shell/components/` 下加新文件 | 无 |
| `core/types.ts` 需要加字段 | 简单沟通后加（加字段向后兼容） | 极低 |
| 注册机制本身需要修改 | 极少发生，走沟通 | 极低 |

**原则：只加文件不改已有文件、各自目录、通过注册机制集成。**

## 十八、预设 Schema 与功能点的组织方式

### 关注点

每个模块都有很多模型和功能点（如P-01有13种受力模型，P-02有6种运动场景），这些功能点在代码中如何组织？

### 结论

**每个功能点 = 一个预设 JSON 文件，预设是功能的最终载体。**

预设 JSON 完整描述一个场景需要的一切：包含哪些实体、实体间关系、可调参数范围、支持的视角、特殊功能开关。求解器和渲染器按实体类型和关系类型注册，是通用的——预设文件只告诉它们"这个场景有什么"。

**加一个新功能点 = 加一个新 JSON 文件**，不需要改已有代码。

预设 Schema 的具体字段定义属于架构设计阶段的产出。设计原则：

- 预设 = 实体 + 关系 + 参数 + 视角 + 功能开关，完整自描述
- 参数面板由 schema 驱动：UI 组件读取预设中的参数定义自动渲染
- 求解器通过实体类型和关系类型自动匹配，不需要在预设中指定用哪个求解器

### 开发节奏

不是"先把一个模块的所有功能做完再做下一个"，而是"先纵深打通一个预设，再横向扩展"。以力学域为例：

1. 先做 P-01 的"斜面·下滑"一个预设 → 驱动实现基础受力求解+物块渲染+受力视角 → 浏览器可跑
2. 扩展 P-01 其他模型 → 复用框架，每个模型一个新 JSON
3. 转做 P-02 → 复用受力引擎，新增运动求解器+运动视角
4. 转做 P-12 → 复用运动求解器，新增碰撞求解+动量视角
5. 最后 P-14 → 几乎不需新求解器，主要是能量视角

每一步都有可见的产出，完全契合 vibe coding 的迭代节奏。

## 十九、Phase 2 搭建能力在项目组织上的展望

### 关注点

Phase 1 的预设覆盖85%教学场景，Phase 2 让老师自己搭建场景覆盖到95%。Phase 2 在代码组织上怎么做？会不会需要重构？

### 结论

**Phase 2 是扩展，不是重构。** Phase 1 的 Entity System + 注册机制 + 预设 schema 已经为 Phase 2 铺好了路。

Phase 1 和 Phase 2 的底层是同一套东西，区别只是场景数据从哪来：

- Phase 1：开发者写好的 preset JSON 文件
- Phase 2：老师在运行时通过拖放操作动态生成的场景数据（格式与 preset JSON 完全一致）

Phase 2 真正新增的只有一个模块——搭建器 UI（工具栏 + 拖放 + 自动关联），负责把老师的操作翻译成 Entity + Relation 数据。已有的求解器、渲染器、视角层全部复用。

跨域综合场景（压轴题）的支持关键在于"多阶段求解"——系统检测物体从一个阶段进入下一个阶段的事件（如"离开斜面+进入磁场区域"），自动切换求解策略。这个事件检测机制在产品方案维度一（1.3模拟引擎流程）中已经定义。

## 二十、P1 优先级模块的分工归属

### 关注点

P0 模块的分工（力学域/电磁域）已确定，但 P1 优先级模块（P-05/P-06/P-07/P-09）归谁？

### 结论

按照"物理域纵切"的同一逻辑，按物理属性的亲近性分配：

| 模块 | 归属 | 理由 |
|------|------|------|
| P-05 简谐运动 | 开发者 A（力学域） | 弹簧振子是力学的 F=-kx，复用 A 已有的弹簧实体和运动求解器 |
| P-06 波动 | 开发者 A（力学域） | 本质是多个振子排列，是 P-05 的延伸 |
| P-09 天体运动 | 开发者 A（力学域） | F=GMm/r² 就是力，轨道运动复用运动求解器 |

## 二十一、求解器架构的统一——不存在"通用"vs"硬编码"的矛盾

### 关注点

产品方案维度一描述了通用的 Entity System（实体+关系+统一求解），但讨论记录第五条说"每个预设硬编码自己的物理方程"。这是否是两种不兼容的架构？

### 结论

**不矛盾。这是同一架构的两个视角。**

之前的描述把"通用Entity系统"和"预设硬编码方程"说成对立的路线A和路线B，称其"在代码层面不兼容"，这个判断是错误的。

实际的架构只有一条路线：

- **数据模型按通用方式设计**：Entity、Relation、State 的 schema 是结构化的、通用的，不针对特定物理域
- **求解逻辑按"实体+关系组合"枚举注册**：每个求解器处理一种特定的实体+关系组合模式（如 `block + slope + contact → slope-contact-solver`）
- **匹配机制是统一的查表**：运行时根据场景中的实体和关系组合，从注册表中找到对应的求解器

Phase 1 和 Phase 2 的区别仅在于"谁来提供场景数据"：
- Phase 1：预设 JSON 文件（开发者预先定义的）
- Phase 2：老师在运行时通过拖放操作动态生成的

底层的求解器是同一批代码，匹配逻辑也不变。所谓"预设硬编码"，只是说求解器的物理方程是人写的（而非通用引擎自动推导的），这和 Entity System 的数据模型是否通用完全不冲突。

## 二十二、碰撞检测与共享物理工具库的归属

### 关注点

碰撞检测、约束求解、数值积分等能力被力学域和电磁域共同需要。放在谁的目录下？

### 结论

**放在 `core/physics/` 中作为共享工具函数库。**

理由：
1. **两个域都需要**——力学要检测"物块离开斜面"，电磁要检测"粒子进入磁场区域"，用的是同一个"点在矩形内"几何判断
2. **纯数学工具，不含域知识**——`pointInRect(x, y, rect)` 不关心是物块还是粒子，`rk4Step(fn, state, dt)` 不关心是力学ODE还是电磁ODE
3. **"只加不改"原则仍适用**——每个函数是独立导出的，加新函数不改已有函数，两个开发者可以各自添加需要的工具函数而不冲突

