# 物理教具统一模拟器 · 产品方案

> 所属产品线：EduMind AI 互动课件平台 · 物理学科教具
> 日期：2026-03-16

---

## 设计理念

**一个统一的物理模拟器。** 场景中可以同时存在斜面、物体、弹簧、电场、磁场等任何物理元素。PRD中定义的各种模型（斜面受力、平抛运动、洛伦兹力圆周……）都是这个模拟器的场景预设。

```
物理模拟器
  ├── 场景预设库（按教材章节/物理模型组织，覆盖PRD全部11个模块的功能点）
  ├── 视角层（受力/运动/能量/动量/场/电路 —— 控制显示哪些物理量）
  └── 实体系统（物体/表面/连接件/场 —— 统一数据模型）
```

**为什么不按物理域分多个工具**：物理世界本身没有"力学工具"和"电磁工具"的边界。一道高考综合题可能同时涉及斜面下滑+进入磁场做圆周运动，按域隔离的产品无法在同一画布上完整呈现这类题目。

## 维度一：统一模拟器架构

### 1.1 实体系统设计

场景中的一切都是实体（Entity），每个实体有统一的 id、type、properties。

| 实体类别 | 具体类型 | 关键属性 |
|---------|---------|---------|
| **物体** | 物块、小球、小车、带电粒子、导体棒 | 质量、电荷量、初速度、位置、形状 |
| **表面** | 水平面、斜面、弧形轨道、导轨 | 角度、摩擦因数、长度、位置 |
| **连接件** | 弹簧、轻绳、轻杆、滑轮 | 劲度系数、自然长度、最大张力 |
| **场** | 重力场、匀强电场、匀强磁场 | 场强大小、方向、作用区域 |
| **约束源** | 铰接点、固定点、滑动轨道 | 约束类型、位置 |
| **观测工具** | 速度传感器、能量计、电压表 | 绑定的物体、显示选项 |

### 1.2 实体间关系

物理场景的关键不是实体本身，而是**实体之间的关系**：

| 关系类型 | 说明 | 示例 |
|---------|------|------|
| **接触** | 物体与表面之间 | 物块放在斜面上（法向约束+可能的摩擦） |
| **连接** | 物体与连接件之间 | 物块挂在弹簧上、两物块用轻绳相连 |
| **场作用** | 场对物体的作用 | 带电粒子在磁场区域内受洛伦兹力 |
| **包含** | 物体在另一个物体内/上 | 小球在小车内（小车是物体，不是表面） |

### 1.3 模拟引擎流程

```
实体 + 关系 + 参数
    ↓
受力分析引擎（遍历所有关系→计算每个物体的合力）
    ↓
运动求解器（合力→加速度→数值积分→位置/速度更新）
    ↓
事件检测（碰撞？离开表面？进入/离开场区域？到达终态？）
    ↓
渲染（当前视角层决定显示哪些物理量）
```

### 1.4 视角层

视角层控制**显示哪些物理量**，不改变底层数据。老师随时切换，就像给同一个场景戴上不同的"滤镜"。

| 视角 | 显示内容 |
|------|---------|
| **受力** | 各力箭头 + 力列表 + 合力 + 正交分解 |
| **运动** | 速度/加速度箭头 + 轨迹 + v-t/s-t图 |
| **能量** | 能量条形图 + 做功标注 + 总能线 |
| **动量** | 动量箭头 + 系统总动量 + 碰撞分析 |
| **场** | 场线 + 等势面 + 场区域高亮 |
| **电路** | EMF + 电流方向 + 电压标注 |

**视角可叠加**：默认只开一个，老师可以手动叠加多个（如受力+运动同时显示）。

## 维度二：使用场景与操作流程

### 2.1 四个具体场景

| # | 场景名 | 频率 | 老师要做什么 | 工具提供什么 | 结果 |
|---|--------|------|------------|------------|------|
| S1 | **讲概念——展示分析结果** | 高频(每节课) | 展示一个物理模型的受力/运动/能量等分析结果 | 从预设库加载场景→选择视角层→展示 | 学生看到清晰的物理量可视化 |
| S2 | **讲例题——复现题目条件** | 高频(每节课) | 把一道具体题的条件输入，看完整物理过程 | 从预设库加载→微调参数→播放模拟 | 题目条件可视化，数值与手算一致 |
| S3 | **动态探究——改变参数** | 中频 | 实时改变某个参数，观察物理过程的变化 | 参数实时响应，所有物理量即时更新 | 学生看到参数变化的物理效果 |
| S4 | **过程回放——关键帧讲解** | 低频 | 暂停在某个时刻，讲解当前所有物理量 | 时间轴拖拽 + 暂停态数值读取 | 学生理解过程中每个瞬间的状态 |

### 2.2 操作步骤流程

#### S1 讲概念（4步）

1. **打开模拟器**：点击首页"物理模拟器"
2. **加载预设**：从场景预设库选择（如"斜面受力分析"），画布出现完整场景+默认参数
3. **选视角**：顶部选"受力"视角→画布显示各力箭头和数值
4. **调条件（可选）**：左侧面板调角度/摩擦等→受力图实时更新

#### S2 讲例题（4步）

1. **加载预设**：场景库搜索或按章节浏览→选中最接近的预设
2. **调参数**：左侧面板修改为题目的精确条件（m=2kg, θ=30°, μ=0.3）
3. **查看受力**：选"受力"视角→核对各力大小→点击"正交分解"
4. **播放运动**：切到"运动"视角→点击播放→v-t图同步绘制→暂停讲解

#### S3 动态探究（3步）

1. **基于已有场景**（接S1或S2）
2. **拖动参数**：如角度滑块从30°拖到45°→画布实时更新
3. **观察变化**：数值面板实时更新，学生直观看到因果关系

#### S4 过程回放（3步）

1. **播放运动**：点击播放，物体运动完整播放一遍
2. **拖动时间轴**：拖到关键时刻（如进入磁场瞬间、碰撞瞬间）
3. **读取数值**：暂停态下数值面板显示所有物理量

### 2.3 跨域综合题的操作流程

**场景：带电小球从斜面顶端释放，滑下后进入斜面下方的匀强磁场区域做圆周运动**

| 步骤 | 老师操作 | 系统响应 |
|------|---------|---------|
| 1. 加载预设 | 场景库搜索"斜面+磁场"或从"跨域综合"分类找到 | 画布出现斜面+下方磁场区域+带电小球 |
| 2. 调参数 | 设斜面角度、摩擦因数、小球质量/电荷量、磁场强度 | 场景几何和参数更新 |
| 3. 受力视角 | 选"受力"视角 | 显示斜面上的受力（G、N、f） |
| 4. 播放 | 点击播放 | 小球沿斜面加速→离开斜面→进入磁场→做圆周运动，全过程连续 |
| 5. 切视角 | 切到"运动"视角 | 显示速度箭头+完整轨迹（斜面直线+磁场圆弧） |
| 6. 暂停在磁场入口 | 拖动时间轴到小球刚进入磁场的时刻 | 数值面板显示此刻的v（=斜面末速度=圆周切线速度），可验证r=mv/qB |

| **支持力 N** | 蓝色 #2980B9 | 实线箭头⊥面 | 箭头末端 |
| **摩擦力 f** | 橙色 #E67E22 | 虚线箭头沿面 | 箭头末端 |
| **弹力 T** | 绿色 #27AE60 | 实线箭头沿绳/弹簧 | 箭头末端 |

| **洛伦兹力 qvB** | 品红 #9B59B6 | 虚线箭头⊥v | 箭头末端 |

| **动能 Ek** | 红色条 | 条形图柱 | 柱顶数值 |
| **势能 Ep** | 蓝色条 | 条形图柱 | 柱顶数值 |
| **总能 E总** | 灰色线 | 水平参考线 | 线右端数值 |
| **电场线** | 金色 #F39C12 | 带箭头曲线 | — |
| **磁感线** | 品红 #9B59B6 | ×/·符号阵列 | B值标注 |
| **等势面** | 金色虚线 | 虚线等高线 | 电势值 |

**区分规则**：
- 力学力用暖色系（红/橙/绿），电磁力用冷色系（金/品红）
- 同类力用同色系不同明度区分
- 每个力箭头旁标注名称缩写
- 多物体用不同底色区分（物体A=蓝底，物体B=橙底）

### 3.2 视角层交互

```
┌──────────────────────────────────────────────────────────────┐
│  视角：[受力] [运动] [能量] [动量] [场] [电路]               │
│  叠加：☑受力 ☐运动 ☐能量 ☐动量 ☐场 ☐电路                   │
│  密度：[简洁] [标准] [详细]                                  │
└──────────────────────────────────────────────────────────────┘
```

| 操作 | 效果 |
|------|------|
| 点击视角按钮 | 切换主视角，只显示该视角的物理量，其余全关 |
| 勾选叠加复选框 | 在当前主视角基础上叠加显示其他物理量（降低透明度0.3） |
| 切换信息密度 | 简洁=只有箭头；标准=箭头+数值；详细=箭头+数值+公式 |

**自动降密规则**：
- 切换主视角时，叠加层自动清除（回到干净状态）
- 场景中实体超过3个时，自动切换到"简洁"密度
- 场区域自动用半透明底色标识范围，不需要单独开启

### 3.4 2D vs 3D 决策

**本阶段全部2D**。水平圆周等需要空间感的场景用"主视图+辅助小图"补充。理由：
- 零学习成本（不需要教老师旋转3D视角）
- 技术复杂度可控（不引入Three.js）
- 与物理教材/试卷的示意图风格一致

## 维度四：场景预设与搭建体系

### 4.1 预设库设计

**预设 = 一组预配置的实体+关系+参数的场景快照。** 加载一个预设，就是往Entity系统中注入这组数据。

#### 三级预设组织

| 层级 | 维度 | 示例 |
|------|------|------|
| **L1 物理模型** | 按力学模型分类 | 斜面、圆周、碰撞、电场偏转、洛伦兹力… |
| **L2 典型场景** | 常见条件组合 | 斜面-匀速上滑、洛伦兹力-直线边界… |
| **L3 经典例题** | 精确参数的题目 | "2kg物体在30°斜面μ=0.3从静止下滑" |

#### 场景可穷举性

高中物理题目的场景高度模式化。高考物理的本质是模型识别——每道题都是把文字条件映射到有限的物理模型上。L1+L2预设能覆盖约85%的高中物理教学场景。覆盖不到的15%主要是多物体复合场景，可通过后续"手动搭建"能力扩展。

### 4.2 场景搭建：两种模式

#### 模式A：预设加载 + 参数微调（覆盖85%场景，推荐）

| 步骤 | 操作 | 耗时 |
|------|------|------|
| ① 选预设 | 从预设库选择最接近的场景 | 3秒 |
| ② 调参数 | 点击场景中的实体→左侧属性面板修改参数 | 10秒 |
| ③ 开始 | 点击播放或查看受力 | 1秒 |

**总耗时约15秒。**

#### 模式B：手动搭建（覆盖剩余15%，进阶，Phase 2提供）

| 步骤 | 操作 | 耗时 |
|------|------|------|
| ① 添加环境 | 工具栏点击"添加"→选择表面类型（斜面/水平面/轨道） | 3秒 |
| ② 添加物体 | 工具栏点击"添加"→选择物体类型（物块/小球/带电粒子） | 3秒 |
| ③ 建立关系 | 将物体拖放到表面上→系统自动识别为"接触"关系 | 3秒 |
| ④ 添加场（可选） | 工具栏点击"添加"→选择场类型→拖拽定义作用区域 | 5秒 |
| ⑤ 设参数 | 点击各实体→设置属性 | 10秒 |
| ⑥ 开始 | 点击播放 | 1秒 |

**总耗时约25秒。** 关键是步骤③——不需要老师手动定义"法向约束+摩擦"，只需把物块拖到斜面上，系统自动识别接触关系。

### 4.3 快捷操作设计

| 快捷设计 | 说明 |
|---------|------|
| **智能默认值** | 添加斜面→自动θ=30°、μ=0；添加物块→自动m=1kg |
| **拖放自动关联** | 物体拖到表面上→自动建立接触关系；物体拖到场区域内→自动受场作用 |
| **条件一键切换** | 光滑⇄粗糙、有初速⇄从静止、有电荷⇄中性 |
| **参数联动推断** | 设"匀速"→自动反算所需摩擦因数 |
| **最近参数记忆** | 上次使用的参数自动填充 |
| **数值直接输入** | 滑块旁可键入精确值 |

### 4.4 场景保存与复用

| 功能 | 设计 |
|------|------|
| **保存** | 点击工具栏"保存场景"→输入名称→保存所有实体+关系+参数+显示状态+当前时刻 |
| **存储位置** | 浏览器 localStorage（单机使用），后续可扩展到云端 |
| **组织方式** | 按保存时间倒序 + 支持文件夹分组（老师可建"高三一轮复习"等文件夹） |
| **检索** | 名称搜索 + 按模型类型筛选 |
| **分享** | 导出为JSON文件→另一台电脑导入（不依赖账户系统） |
| **与预设的关系** | 保存的场景数据结构与系统预设完全一致——个人保存就是"个人预设" |

## 维度六：分阶段实施策略

### 实施路线

```
Phase 1: 受限模拟器 + 预设覆盖
  ├── 实体系统基础架构（Entity/Relation/Property）
  ├── 受力分析引擎（支持已知模型的力计算）
  ├── 运动求解器（解析解为主，RK4为辅）
  ├── 渲染引擎 + 6个视角层
  ├── 预设库（力学≈50个 + 电磁≈30个 + 其他≈20个预设）
  └── 产品形态：老师从预设加载+调参数，不能手动搭建

Phase 2: 结构化搭建能力
  ├── "添加实体"工具栏
  ├── 拖放自动关联
  ├── 新增实体类型（小车、滑轮、更多连接件）
  └── 产品形态：老师可以从菜单选择组件组合搭建场景

Phase 3: 自由编辑器
  ├── 通用约束求解器
  ├── 任意实体组合的碰撞检测
  ├── 自定义场区域
  └── 产品形态：接近Algodoo的自由度，但保留精确计算和预设入口
```

### Phase 1 的范围边界

**做什么**：
- 统一的实体数据模型（Phase 1只支持预设中出现的实体类型和关系组合）
- 受力引擎、运动求解器、渲染引擎
- 6个视角层
- ≈100个预设场景
- 场景保存/加载

**不做什么**：
- 不做手动添加实体（Phase 2）
- 不做自由拖放建关系（Phase 2）
- 不做通用约束求解器（Phase 3）
- 不做通用碰撞检测（Phase 3）

**关键架构约束**：虽然Phase 1老师只能用预设，但底层架构必须是Entity系统。预设的加载就是往Entity系统中注入一组预配置的实体和关系。这样Phase 2/3是扩展，不是重构。

### 各Phase的预设覆盖

| Phase | 新增实体类型 | 新增预设 | 累计覆盖 |
|-------|------------|---------|---------|
| Phase 1 | 物块、小球、带电粒子、斜面、水平面、弧形轨道、弹簧、轻绳、重力场、匀强电场、匀强磁场、电阻、导体棒 | ≈100个（力学+电磁+振动+光学+热力学+天体） | ≈85%教学场景 |
| Phase 2 | 小车、滑轮、轻杆、传送带、非匀强场 | ≈40个（复合场景+跨域） | ≈95%教学场景 |
| Phase 3 | 用户自定义 | 用户自建 | ≈100% |

## 维度七：工程组织与并行开发策略

### 7.1 仓库策略：单一 Monorepo

**决策：所有模块在同一个代码仓库中，通过目录结构隔离边界。**

理由：

1. **统一模拟器的设计要求共享实体系统**。P-01/P-02/P-08/P-12/P-13/P-14 六个 P0 模块共享 Entity System、物理求解器、渲染引擎和视角层，拆成多仓库会破坏统一架构。
2. **共享代码量大**。类型定义、数学工具函数、Canvas 渲染基础设施、UI 组件（参数面板、时间轴控制栏）在所有模块间复用，多仓库会造成重复代码和同步负担。
3. **接口变更在单仓库内可原子提交**。跨仓库的接口变更需要协调版本发布，增加不必要的协作成本。

#### 各模块与统一模拟器的关系

| 模块 | 与统一模拟器的关系 | 独立仓库可能性 |
|------|-------------------|---------------|
| P-01 受力分析 | 核心，共享实体系统+物理引擎 | 不可独立 |
| P-02 运动模拟 | 核心，共享实体系统+运动求解器 | 不可独立 |
| P-08 电磁场 | 核心，场实体是实体系统的一部分 | 不可独立 |
| P-12 动量守恒 | 核心，碰撞是运动求解器的扩展 | 不可独立 |
| P-13 电磁感应 | 核心，综合力学+电路+场 | 不可独立 |
| P-14 机械能守恒 | 核心，能量是视角层的一种 | 不可独立 |
| **P-04 电路搭建器** | 交互模式完全不同（拖拽连线），计算引擎不同（基尔霍夫方程组） | **可独立**，但建议 monorepo 内独立目录 |
| P-03 光学 | 光线追迹，不与力学混合 | 可独立，P1 优先级不急 |
| P-05/06 振动波动 | 弹簧振子属于统一模拟器的预设场景 | 不可独立 |
| P-07 热力学 | 分子模拟独立，但气缸模型需力学计算 | 部分可独立 |
| P-09 天体 | 万有引力是场的一种，轨道运动是运动求解器的延伸 | 不可独立 |

#### P-04 电路搭建器的特殊处理

P-04 虽然交互模式和计算引擎与统一模拟器不同，但建议在 monorepo 内作为独立 app 目录存在（如 `apps/circuit-builder/`），原因：
- 共享 UI 组件库（参数面板样式、数值输入组件等）
- 共享类型定义基础设施
- 未来如需独立部署，可零成本拆出

**注意交叉点**：P-13 电磁感应的电路视角需要电路求解能力。统一模拟器内部需要一个轻量级电路求解模块（仅支持串并联+欧姆定律），与 P-04 的完整电路搭建器（支持拖拽连线、故障模拟、五大实验等）是不同层次的实现。

### 7.2 并行开发分工策略

**两位开发者各使用独立的 Claude Code 进行开发（vibe coding 模式），每个人必须能独立交付可运行的完整功能。**

#### 为什么按物理域纵切而非按技术层横切

按层横切（一人做引擎、一人做UI）在 vibe coding 场景下有三个致命问题：

1. **每个功能都需要两人对接才能跑起来**——vibe coding 的核心是"写完就跑，从功能角度看对不对"，横切让每个人都无法独立验证。
2. **接口契约在 vibe coding 中一定会频繁变动**——边做边发现"这里应该这样"，横切时接口一变两人都要改，互相阻塞。
3. **两人进度不同步时产生等待**——A做完引擎等B做UI，或B用Mock做完UI等A的真实引擎。

**因此采用按物理域纵切：每人负责完整的功能域，包含该域的引擎+渲染+UI+预设。**

#### 分工方案

**P0 模块（Phase 1 首批交付）：**

| | 开发者 A：力学域 | 开发者 B：电磁域 |
|--|-----------------|-----------------|
| **负责模块** | P-01 受力分析器、P-02 运动模拟器、P-12 动量守恒、P-14 机械能守恒 | P-08 电磁场可视化器、P-13 电磁感应、P-04 电路搭建器 |
| **交付形态** | 每个预设场景 = 完整可用的功能（引擎+渲染+UI+预设） | 同左 |
| **独立验证** | 写完一个预设就能在浏览器里跑起来看效果 | 同左 |
| **互不阻塞** | 力学域和电磁域在 Phase 1 中没有交叉依赖 | 同左 |

**P1 模块（第二批交付）：**

| 模块 | 归属 | 理由 |
|------|------|------|
| P-05 简谐运动 | 开发者 A（力学域） | 弹簧振子是力学的 F=-kx，复用 A 已有的弹簧实体和运动求解器 |
| P-06 波动 | 开发者 A（力学域） | 本质是多个振子排列，是 P-05 的延伸 |
| P-09 天体运动 | 开发者 A（力学域） | F=GMm/r² 就是力，轨道运动复用运动求解器 |
| P-07 热力学 | 开发者 B | 分子模拟是独立范式，与力学框架关系不大；气缸模型的力学部分（活塞受力平衡）很简单，不需要复用 A 的求解器 |

P1 工作量均衡：A 拿 3 个（都是力学延伸，复用度高），B 拿 1 个（独立范式，体量小，B 在 P0 阶段模块数更多）。P-07 热力学在目录上可以是 `domains/em/` 下的 `thermo/` 子目录，或独立为 `domains/thermo/`，由开发者 B 按实际开发体验决定。

#### 目录结构：插件式注册，各自目录


**核心设计原则：`core/` 和 `shell/` 提供的是机制（注册、渲染循环、布局框架），不包含任何具体物理域的知识。具体物理域的一切（实体定义、求解逻辑、渲染方式、UI参数schema）都在 `domains/` 各自目录下。**

#### 防冲突机制：只加不改

框架基础设施在并行开发中一定会被扩展（"发现基础设施不够用，得加东西"），但通过以下设计避免文件级冲突：

| 扩展需求 | 做法 | 冲突风险 |
|---------|------|---------|
| 新增实体类型 | 在自己的 `domains/xxx/entities/` 下加文件，通过注册机制挂载 | 无 |
| 新增渲染能力 | 在自己的 `domains/xxx/renderers/` 下加文件 | 无 |
| 需要新的绘制 primitive（如区域高亮） | 在 `renderer/primitives/` 下加新文件，不改已有文件 | 无 |
| 需要新的 UI 组件（如颜色选择器） | 在 `shell/components/` 下加新文件 | 无 |
| `core/types.ts` 需要加字段 | 简单沟通后加字段（加字段向后兼容，不算breaking change） | 极低 |
| 注册机制本身需要修改 | 极少发生，走沟通 | 极低 |

**原则：只加文件不改已有文件、各自目录、通过注册机制集成。Git 合并时几乎不会有文件级冲突。**

#### 协作机制

1. **Git 分支隔离**：开发者 A 在 `feat/mechanics` 分支，开发者 B 在 `feat/em` 分支，定期合并到主分支。
2. **CLAUDE.md 写明归属**：在项目 CLAUDE.md 中声明 `domains/mechanics/` 归开发者A、`domains/em/` 归开发者B，Claude Code 不会越界修改他人负责的目录。
3. **共享层沟通机制**：`core/`、`shell/`、`renderer/primitives/` 的修改需简单沟通确认（因为采用只加不改策略，实际冲突极少）。

#### 启动顺序

```
第一步：架构设计
   产出：技术栈选型 + 目录结构 + core/types.ts 基础接口
        + Preset JSON Schema + 注册机制设计 + 参考实现规范

第二步：框架初始化（一个人完成，或一起完成）
   ├── 项目骨架（技术栈、构建工具、目录结构）
   ├── core/ 注册机制实现
   ├── shell/ UI骨架实现
   ├── renderer/primitives/ 基础绘制能力
   └── 一个最简参考实现（如"水平面上的物块受力"）作为开发范例

第三步：并行开发（各自独立推进，互不等待）
   ├── 开发者 A：从"斜面下滑"预设开始，逐步扩展力学域
   └── 开发者 B：从"电场可视化"预设开始，逐步扩展电磁域
   每做完一个预设 = 一个完整可用的功能，立刻可在浏览器验证

第四步：整合跨域场景
   └── "斜面+磁场"等跨域综合预设，两人合作完成
```

#### 功能点的开发节奏（以开发者A力学域为例）

每个模块有多种模型/场景，开发节奏是"先纵深打通一个，再横向扩展"：

```
1. 先做 P-01 的"斜面·下滑"这一个预设
   → 驱动实现：重力+支持力+摩擦力求解 + 物块和斜面渲染 + 受力视角
   → 浏览器可跑 ✓

2. 扩展 P-01 其他模型（悬挂、连接体、圆周…）
   → 复用受力求解框架，新增各模型的力计算 + 渲染
   → 每做完一个模型，浏览器可跑 ✓

3. 转做 P-02 运动模拟
   → 复用 P-01 的受力引擎，新增运动求解器 + 运动视角 + v-t图
   → 每做完一个场景，浏览器可跑 ✓

4. 转做 P-12 动量守恒
   → 复用运动求解器，新增碰撞求解 + 动量视角

5. 最后 P-14 机械能守恒
   → 几乎不需新求解器，主要是能量视角 + 能量条形图渲染
```

### 7.3 求解器架构：数据模型通用，求解逻辑枚举注册

Entity System 的数据模型按通用方式设计（实体、关系、属性结构化定义），但求解逻辑按"实体+关系组合"枚举注册。这不是两种对立的架构，而是**同一架构在不同阶段的自然演进**：

```
求解器注册表（开发阶段逐步充实）：
  "block + slope + contact"      → slope-contact-solver    （斜面受力求解）
  "charged-particle + bfield"    → lorentz-force-solver    （洛伦兹力圆周）
  "conductor-bar + rail + bfield"→ em-induction-solver     （电磁感应单棒）
  ...

Phase 1 运行时：
  预设 JSON 加载 → 实体和关系注入 Entity System → 匹配注册表中的求解器 → 执行
  （匹配结果是确定的，因为预设内容是预先定义的）

Phase 2 运行时：
  老师搭建场景 → 实体和关系动态注入 Entity System → 同样匹配注册表 → 执行
  （匹配逻辑不变，只是数据来源从 JSON 文件变成了用户操作）
```

**不存在"通用Entity系统"与"预设硬编码"的矛盾**——数据模型始终是通用的，求解器始终是按组合枚举注册的，匹配机制始终是查表。Phase 1 和 Phase 2 的区别仅在于"谁来提供场景数据"。

### 7.4 共享物理工具库（core/physics/）

`core/physics/` 只放**两个域都会用到的纯数学工具函数**，不放特定域的物理公式。

#### 收录标准

| 放 core/physics/ | 不放（留在各域 solvers/） |
|-----------------|----------------------|
| `rotateVector(v, θ)` — 两个域都用 | `lorentzRadius(m,v,q,B)` — 只有电磁域用 |
| `pointInRect(x,y,rect)` — 碰撞检测通用 | `slopeAccel(θ,μ)` — 只有力学域用 |
| `rk4Step(fn,state,dt)` — 任何ODE都用 | `elasticCollision(m1,v1,m2,v2)` — 只有P-12用 |
| `solveQuadratic(a,b,c)` — 纯数学 | `inducedEMF(B,L,v)` — 只有P-13用 |
| `g = 9.8` — 到处用 | `energyBalance(...)` — 只有P-14用 |

开发过程中如果发现某个函数两边都需要，再提到 core/ 里即可，不需要提前穷举。

#### 文件组织：一个函数一个文件

`vector/` 和 `geometry/` 是两人都会在早期往里加函数的高频目录。为避免冲突，采用"一个函数一个文件 + index.ts 统一导出"的组织方式：

- 加新函数 = 加新文件 + 在 index.ts 补一行 export
- index.ts 的冲突即使发生也是 trivial 的（各加一行 export），git auto-merge 就能处理
- `integrators.ts`、`algebra.ts`、`constants.ts` 函数较少，不需要拆目录

#### 放在 core/ 的理由

- 这些函数是**纯数学工具**，不含任何物理域知识——`pointInRect(x, y, rect)` 不关心是物块还是粒子
- 两个域都需要：力学要检测"物块离开斜面"，电磁要检测"粒子进入磁场区域"，用的是同一个几何判断
- "只加不改"原则仍然适用——每个函数是独立文件，加新函数不影响已有函数

### 7.5 预设 Schema 设计（架构设计阶段细化）

预设 JSON 是功能点的最终载体。每个预设文件完整描述一个场景需要的一切：包含哪些实体、实体间关系、可调参数范围、支持的视角、特殊功能开关。

预设 Schema 的具体字段定义属于架构设计阶段的产出，此处仅说明设计原则：

- **预设 = 实体 + 关系 + 参数 + 视角 + 功能开关**，完整自描述
- **参数面板由 schema 驱动**：UI 组件读取预设中的参数定义（min/max/step/default）自动渲染，不需要为每个预设写 UI 代码
- **加一个新功能点 = 加一个新 JSON 文件**，不需要改已有代码
- 求解器通过实体类型和关系类型自动匹配（查注册表），预设文件只需描述"这个场景有什么实体和关系"

### 7.6 Phase 2 的项目组织展望

Phase 1 老师只能选预设调参，Phase 2 让老师从菜单选组件自己搭建场景。两者的底层是同一套 Entity System，区别只是数据来源不同：

```
Phase 1：开发者写好 preset JSON → 系统加载 → 老师调参数 → 运行模拟
Phase 2：老师从工具栏选实体 → 拖到画布建关系 → 系统在内存中动态生成场景数据 → 运行模拟
```

Phase 2 在项目中新增的部分：

```
src/
├── core/               # 不变
├── shell/              # 不变
│   └── builder/        # Phase 2 新增：搭建工具
│       ├── Toolbar.tsx  #   "添加实体"工具栏
│       ├── DragDrop.tsx #   拖放交互 + 自动关联逻辑
│       └── SceneBuilder.ts  # 将用户操作转化为 Entity+Relation 数据
├── renderer/           # 不变
├── domains/
│   ├── mechanics/      # 求解器和渲染器不变，可能扩展实体类型（小车、传送带等）
│   ├── em/             # 同上
│   └── cross-domain/   # Phase 2 新增：跨域组合场景
│       ├── solvers/    #   多阶段求解（斜面阶段→磁场阶段→…）
│       └── presets/    #   跨域预设
└── app/                # 不变
```

Phase 1 的架构保证 Phase 2 是**扩展而非重构**：
- Entity System 是通用的 → Phase 2 只是动态创建实体，不改数据模型
- 求解器按注册机制工作 → Phase 2 不需要改已有求解器，只需新增组合求解模板
- 渲染器按实体类型注册 → Phase 2 新增实体类型时注册新渲染器即可
- Preset JSON schema 就是内存数据模型 → Phase 2 搭建器生成的数据和预设格式完全一致
