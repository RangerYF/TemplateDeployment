> 日期：2026-03-13
> 背景：基于《P物理沙盒_产品方案.md》的设计评审讨论

---

## 一、预设模型能覆盖多少场景？

### 问题

S1（选模型）和 S2（选预设）的设计，能覆盖大多数老师需要的场景吗？物理题目的场景是好分类、好做预设的，还是五花八门的？

### 结论

**能覆盖约85%的高中物理教学场景。** 高中物理题目的场景高度模式化，不是五花八门的。

高考物理的本质是模型识别——每道题都是把文字条件映射到有限的物理模型上。以力学为例，场景可穷举度很高：

| 模型 | 下级场景 | 穷举性 |
|------|---------|--------|
| 水平面 | 匀速/加速/减速 × 光滑/粗糙 × 单体/连接体 | 约12种组合，**可穷举** |
| 斜面 | 上滑/下滑/静止 × 光滑/粗糙 × 有无初速 | 约12种，**可穷举** |
| 悬挂 | 单绳/双绳/绳+杆 × 静止/圆周 | 约8种，**可穷举** |
| 圆周运动 | 水平(转盘/锥摆/管道) / 竖直(绳/杆/轨道) | 约10种，**可穷举** |
| 碰撞 | 完全弹性/非弹性/完全非弹性 × 追及/对碰 | 约6种，**可穷举** |
| 平抛 | 水平抛出/斜面上抛出 | 约4种，**可穷举** |

**真正五花八门的不是场景本身，而是参数组合和问法。** 场景（几何结构+约束类型）是有限的，参数（质量、角度、摩擦因数等）是连续的——所以"选模型/预设 + 调参数"的模式是对的。

**覆盖不到的15%** 主要是多物体复合场景（如"传送带上的物体飞出后做平抛落到斜面上"——跨了3个模型），以及非标几何（如"两个斜面背靠背"、"弧形轨道接平面"）。这些可通过后续"场景拼接"能力扩展。

## 二、需求文档中是否提到了"对比教学"和"逐步推理"

### 对比教学

**有，PRD多处明确提到：**

| 模块 | PRD原文 |
|------|--------|
| P-02 运动模拟器 | "多物体对比模式：最多3组不同参数，不同颜色区分" |
| P-14 机械能守恒 | "有/无摩擦对比：任意模型切换光滑/有摩擦，有摩擦时总能量减少=摩擦力做功" |
| P-12 动量守恒 | 恢复系数滑块0→1，本质就是对比不同碰撞类型的结果 |

对比功能不是新增需求，是PRD已有需求的交互方案设计。

### 逐步推理

**部分有，部分是新增建议：**

PRD中已有的逐步展示：

| 模块 | PRD原文 |
|------|--------|
| P-01 受力分析 | "正交分解动画"——本身就是逐步过程 |
| P-13 电磁感应 | **明确提到**："暂停后逐步展示速度→EMF→电流→安培力" |
| P-08 电磁场 | 粒子偏转的过程演示（播放/暂停/单步控制） |

**新增的部分**是把P-01受力分析从"一次性显示所有力"扩展为"每步揭示一个力+推理文字"。PRD原文中P-01的交互是"选模型→设条件→自动显示受力图"（结果导向），逐步推理是过程导向的，属于超出PRD的新建议。

### 处理决定

产品方案中已去掉"对比教学(S3)"和"逐步推理模式"两个功能，保持与PRD一致。

## 三、是否需要做成自由场景编辑器？

### 问题背景

当前方案固定的是"选模型→调参数"，但实际物理题目可能出现复合场景（如斜面上有个小车，小车里有个球）。按现在的设计不太好应对。

立体几何产品有过类似经历：V0.1是"功能堆叠式"设计（选几何体→调参数→看结果），后来发现老师实际需要的是自由的3D编辑器，于是V0.2做了架构重构（Entity/Command/Tool/Selection七系统）。

**物理产品会不会也存在这种情况？老师需要的是否是一个自由的场景编辑工具？**

### 结论

**方向是对的，但复杂度差异巨大，需要分阶段。**

#### 与立体几何的相似性

| 维度 | 立体几何 | 物理沙盒 |
|------|---------|---------|
| PRD初始设计 | 选几何体→调参数→看结果 | 选模型→调参数→看结果 |
| 老师真实需求 | 自由放点、画线、建坐标系 | 自由放物体、加力、设约束 |
| "模板"的角色 | 快捷入口，不是唯一入口 | 应该也是快捷入口 |

#### 与立体几何的关键差异

| 维度 | 立体几何编辑器 | 物理模拟器 |
|------|-------------|-----------|
| **计算复杂度** | 纯几何计算（坐标、距离、面积），解析公式直接算 | 物理仿真（微分方程、碰撞检测、场计算），需要数值积分引擎 |
| **实体间关系** | 简单引用（点在棱上、线段连两点） | 复杂物理约束（接触、铰链、弹簧、绳、场中受力） |
| **"放上去就能跑"的难度** | 放一个点就在那里，不需要"模拟" | 放一个物体要受力、运动、碰撞——背后是完整的物理引擎 |
| **正确性要求** | 几何计算要么对要么错，容易验证 | 物理仿真可能"看起来像但数值不对"，高考题要求精确 |

#### 推荐的渐进路径

```
第一阶段（当前）: 预设模型 + 参数调节
  → 快速出MVP，验证老师真的在用
  → 但架构上按"实体系统"设计，不按"功能堆叠"

第二阶段: 模型组合能力
  → 同一场景放置多个物体
  → 定义物体间关系（A在B上面、A和B用绳连接）
  → 仍然是"结构化选择"而非"自由绘制"

第三阶段: 自由场景编辑器
  → 拖拽放置任意物体
  → 自由定义约束
  → 预设模板变成"场景快照"
```

#### 架构影响

| 现有方案 | 应该改为 |
|---------|---------|
| "13种受力模型"是顶层概念 | "模型"是预设场景，底层是实体+约束 |
| 参数面板硬编码每种模型的参数 | 参数面板由选中实体的属性驱动 |
| 模块内部用专属数据结构 | 统一的Entity系统（物体、表面、连接件、场都是实体） |
| "切Tab"切换受力/运动/能量视角 | 保留——它是渲染层的视角切换，不影响数据层 |

**关键原则：第一阶段就要用Entity架构，不要功能堆叠。这样第二、三阶段是扩展，不是重构。**

## 四、业界产品调研：Algodoo 和 PhET

### 产品概况

| 维度 | PhET | Algodoo |
|------|------|---------|
| **创始** | 2002年，诺奖得主Carl Wieman创办 | 2009年（前身Phun 2008年） |
| **团队** | 科罗拉多大学Boulder专业团队 | 瑞典Algoryx Simulation AB |
| **定位** | 单主题交互式模拟（125+个模拟程序） | 2D自由物理沙盒 |
| **使用方式** | **浏览器在线运行**（HTML5），也可下载HTML离线用 | **客户端安装**（Windows/Mac桌面程序），iPad有独立App($7.99) |
| **价格** | 完全免费 | 免费（iPad版$7.99） |
| **开源** | **完全开源**（GPLv3/MIT），代码在 [github.com/phetsims](https://github.com/phetsims) | **不开源**，免费但闭源 |
| **语言支持** | 121种语言，含中文 | 有中文界面 |
| **成熟度** | 极其成熟，运营20+年 | 成熟，2025年7月仍在更新(v2.2.4) |
| **社区规模** | 美国1200+高中、300+高校使用 | 5万+用户生成场景 |

### 在中国的使用情况

#### PhET

- 网站支持中文，完全免费
- 国内有一定推广，但不算主流教学工具
- 面临与NOBOOK等本土化工具的竞争
- 参数设置较基础，适合概念演示，不够精确用于高考题核对

#### Algodoo

- **小众**。中国知网仅40篇相关文献（截至2022年），一线教师了解甚少
- 已有实践的老师评价正面："直观"、"节约实验准备时间"、"提高课堂效率"
- **核心障碍："设计出符合物理实验要求的仿真实验，需要一定的操作能力"**——上手成本太高
- 大多数研究停留在教师演示层面，缺乏系统性教学效果评估

### 两款产品的优缺点分析

#### Algodoo

| 做对的 | 做错的（对中国物理老师） |
|--------|----------------------|
| 自由度极高，可画任意形状、设任意物理属性 | 太自由——老师要"3分钟复现一道斜面题"，不是"从零画一个斜面" |
| 视觉效果好，模拟直观 | 数值精度不够——视觉模拟，不是精确计算，高考题要求精确到小数点 |
| 社区内容丰富(5万+场景) | 没有中国教材对齐——没有按人教版章节组织，没有高考题型预设 |
| | 没有受力分析——模拟运动，但不展示"这个物体受了哪些力"的分析过程 |

#### PhET

| 做对的 | 不够的（对中国高中物理老师） |
|--------|--------------------------|
| 极低门槛——拖拽式交互，3秒上手 | 每个模拟是独立的——不能在同一场景中从受力→运动→能量串联 |
| 聚焦概念理解 | 参数设置基础——不能精确输入题目条件(如μ=0.3, θ=30°) |
| 免费+浏览器——零部署成本 | 没有对齐中国教材和考点 |
| | 没有精确数值输出供核对 |

### 对我们产品的定位启示

```
PhET                          我们的产品                    Algodoo
(单主题小工具)                 (应该在这里)                  (自由物理沙盒)
│                             │                            │
极低门槛                       中等门槛                      高门槛
极低自由度                     结构化自由度                   完全自由
概念理解                       精确计算+概念理解              视觉模拟
125个独立模拟                  沙盒+预设                     一个通用画布
```

| 维度 | PhET的教训 | Algodoo的教训 | 我们应该怎么做 |
|------|-----------|-------------|--------------|
| **自由度** | 太低，每个模拟是封闭的 | 太高，老师不会用 | 结构化的自由——从菜单选择组件组合，不是从零绘制 |
| **精度** | 视觉模拟，无精确数值 | 视觉模拟，无精确数值 | 精确计算——数值与解析解一致，老师能核对答案 |
| **教材对齐** | 按概念组织(美国) | 无组织 | 按中国教材章节+物理模型双维度组织 |
| **串联能力** | 没有 | 有(同一场景) | 沙盒内Tab串联——受力→运动→能量 |
| **上手成本** | 极低(3秒) | 高(需要学习) | 目标30秒——选模板→调参数→看结果 |

## 五、PhET源码的参考价值

### PhET的架构模式

每个模拟都遵循统一的 Model-View 架构：

```
每个模拟 =
  ├── model/          ← 纯物理计算（状态+方程+积分）
  │   ├── XXXModel.ts     ← 主模型（物理状态+step()方法）
  │   └── 子模型...        ← 如 Projectile, Trajectory 等
  ├── view/           ← 渲染+交互
  │   ├── XXXScreenView.ts ← 主视图
  │   └── 子视图...        ← 如 ForceArrowNode, TrajectoryNode
  └── XXXScreen.ts    ← Screen入口（连接Model和View）
```

### 与我们模块直接对应的PhET仓库

| PhET模拟 | GitHub仓库 | 对应我们的模块 |
|---------|-----------|--------------|
| Forces and Motion: Basics | [forces-and-motion-basics](https://github.com/phetsims/forces-and-motion-basics) | P-01 + P-02 |
| Projectile Motion | [projectile-motion](https://github.com/phetsims/projectile-motion) | P-02 平抛 |
| Energy Skate Park | [energy-skate-park](https://github.com/phetsims/energy-skate-park) | P-14 机械能 |
| Faraday's Law | [faradays-law](https://github.com/phetsims/faradays-law) | P-13 电磁感应 |
| Geometric Optics | [geometric-optics](https://github.com/phetsims/geometric-optics) | P-03 光学 |
| Collision Lab | [collision-lab](https://github.com/phetsims/collision-lab) | P-12 动量 |
| Charges and Fields | [charges-and-fields](https://github.com/phetsims/charges-and-fields) | P-08 电磁场 |

### PhET共享核心库对照

| PhET库 | 功能 | 对应我们需要的 |
|--------|------|--------------|
| `axon` | Observable属性、事件系统 | Zustand store + 事件 |
| `dot` | 向量、矩阵、数学工具 | 物理计算工具函数 |
| `scenery` | 场景图渲染 | 我们用React + Canvas |
| `scenery-phet` | 物理相关UI组件（箭头、滑块、时钟） | 向量箭头、参数面板 |
| `bamboo` | 图表组件 | v-t图、能量条 |

### Algodoo的技术实现（闭源，来自公开论文）

| 技术 | 说明 |
|------|------|
| SPOOK约束求解器 | Claude Lacoursière的论文公开发表，基于拉格朗日力学的变分离散化方法 |
| SPH流体 | 改进的光滑粒子流体动力学，有专利（非公开） |
| 时间步 | 默认1/60秒，使用近似数值解保证交互实时性 |
| 碰撞检测 | 基于AGX Dynamics商业引擎的简化版 |

Algodoo追求的是"看起来对的视觉模拟"，而我们需要"数值精确的教学计算"，它的近似求解方法是我们要避免的。

### 实际参考建议

| 参考源 | 能借鉴什么 | 不能照搬什么 |
|--------|-----------|-------------|
| **PhET源码** | Model-View分离架构；物理模型文档写法；共享组件设计模式；具体物理方程和积分实现 | 渲染框架Scenery（我们用React+Canvas）；"每个模拟完全独立"的产品架构（我们要沙盒联动） |
| **Algodoo论文** | 通用约束求解器的理论思路（未来自由编辑器阶段参考） | 近似求解方式（我们要精确解） |

**最有价值的行动**：直接读PhET中 `forces-and-motion-basics`、`projectile-motion`、`energy-skate-park`、`collision-lab` 这几个仓库的 `model/` 目录，学习物理模型代码的组织方式。
