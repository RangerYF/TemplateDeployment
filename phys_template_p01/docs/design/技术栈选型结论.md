# 技术栈选型结论

> 阶段1产出文档。记录每个技术决策及理由，供后续阶段直接引用。

---

## 1. 前端框架：React + Zustand

**选择**：React 18 + Zustand

**理由**：
- **状态管理轻量**：项目状态不复杂（当前选中实体、当前视角、模拟状态、参数值），Zustand 的 store 模式足够，无需 Redux 的样板代码
- **Canvas 分离**：React 只管 UI 面板（参数面板、信息面板、时间轴），Canvas 渲染独立于 React 虚拟 DOM，框架不是性能瓶颈
- **AI 辅助友好**：React 生态资料最丰富，Claude Code 对 React 的辅助能力最成熟
- **设计系统已对齐**：`design_guid/` 中的组件（Button、Slider、Select 等）基于 React + Radix UI 实现，直接复用

**淘汰方案**：
| 方案 | 淘汰理由 |
|------|---------|
| React + Redux | 状态不复杂，Redux 样板代码多，过重 |
| Vue 3 + Pinia | 设计系统组件已用 React 写好，切 Vue 需要重写；团队 AI 对 React 更熟 |
| Solid.js | 生态小，遇问题参考资料少，AI 辅助能力弱 |

## 2. Canvas 渲染方案：原生 Canvas 2D API

**选择**：原生 HTML5 Canvas 2D Context API

**理由**：
- **场景简单**：上限不超过 10 个实体 + 若干箭头/标注/轨迹线，原生 Canvas 性能绰绰有余
- **精确控制**：物理箭头、力标注、坐标变换（物理坐标 ↔ 画布像素）需要像素级控制，原生 API 反而最灵活
- **hit test 可控**：产品讨论记录已列出所有实体的命中检测方式（pointInRect、pointInCircle 等基础几何判断），实现不复杂
- **零依赖**：不引入额外渲染库，包体积最小
- **学习曲线低**：Canvas 2D API 是基础 Web 技术，AI 辅助资料充足

**淘汰方案**：
| 方案 | 淘汰理由 |
|------|---------|
| Konva.js | 内置场景图是优势，但对物理渲染（箭头缩放、坐标变换、帧动画）的 API 抽象层反而增加适配成本 |
| Pixi.js | WebGL 加速对本项目场景完全不必要，杀鸡用牛刀 |
| Fabric.js | 偏静态图形编辑器，不适合物理动画 |

## 3. 图表方案：Canvas 自绘

**选择**：在独立 Canvas 元素上用原生 2D API 自绘图表

**理由**：
- **帧同步需求**：图表需要与模拟引擎每帧同步更新（追加数据点 + 移动"当前时刻"标线），外部图表库的更新机制（destroy → rebuild 或 data push → update）在 60fps 下效率不如直接操作 Canvas
- **技术统一**：主画布已用原生 Canvas，图表也用 Canvas 可复用坐标变换、绘制工具等基础设施，开发者只需掌握一套渲染技术
- **完全可控**：需要在图表上叠加"当前时刻"竖线、高亮区间等自定义层，自绘最灵活
- **图表需求有限**：只需折线图（v-t、s-t、a-t、x-t）和柱状图（能量条形图），坐标轴/刻度/标签的绘制代码量可控
- **300px 宽度约束**：紧凑空间内图表样式需要精确控制，自绘避免与图表库的默认样式/间距做对抗

**自绘需实现的基础模块**（在 renderer/primitives/ 或 shell/ 中）：
- 坐标轴绘制（带刻度和标签）
- 折线绘制（数据点数组 → polyline）
- 柱状图绘制（值数组 → 矩形条）
- 当前时刻标线（垂直虚线 + 时间标注）
- 自适应刻度计算（根据数据范围和画布尺寸计算合理的刻度间隔）

**淘汰方案**：
| 方案 | 淘汰理由 |
|------|---------|
| Chart.js | 60fps 逐帧更新性能存疑；自定义叠加层（当前时刻标线）需要插件，API 不直观 |
| D3.js | 学习曲线陡峭，API 偏底层，对简单的折线图/柱状图过于复杂 |
| uPlot | 轻量高性能，但只擅长折线图，能量柱状图支持弱；引入外部依赖的价值不大 |

## 4. 构建工具与项目配置

### 4.1 构建工具：Vite

**选择**：Vite 6

**理由**：
- 开发服务器启动快（ESM 原生模块）
- 对 React + TypeScript 支持开箱即用
- HMR 体验优秀，适合 Canvas 应用的频繁调试
- 社区活跃，是当前 React 纯客户端应用的标准选择

**淘汰方案**：
| 方案 | 淘汰理由 |
|------|---------|
| Next.js | 不需要 SSR/SSG，纯客户端应用用 Next.js 过重 |
| Create React App | 已停止维护 |

### 4.2 包管理器：pnpm

已在 CLAUDE.md 中约定，不再讨论。

### 4.3 TypeScript 配置

**选择**：TypeScript 严格模式（`strict: true`）

关键配置项：
- `target: "ES2020"` — 支持 `??.` `??` 等语法，兼容目标浏览器
- `strict: true` — 严格类型检查
- `noEmit: true` — 类型检查由 tsc 负责，构建由 Vite（esbuild）负责
- `moduleResolution: "bundler"` — 配合 Vite
- `jsx: "react-jsx"` — React 18 的 JSX transform

### 4.4 代码规范：ESLint + Prettier

- **ESLint**：`@typescript-eslint/eslint-plugin` + React hooks 规则
- **Prettier**：统一格式化，避免两人风格差异导致的 diff 噪音
- **Husky pre-commit hook**：`pnpm lint && pnpm tsc --noEmit`，保证提交代码质量

### 4.5 CSS 方案：Tailwind CSS

**选择**：Tailwind CSS v3

**理由**：
- `design_guid/` 中已有完整的 Tailwind 配置和 EduMind 色彩 token
- UI 组件（Button、Slider、Input 等）已基于 Tailwind + class-variance-authority 实现
- 参数面板由 schema 驱动渲染，Tailwind 的 utility-first 方式适合动态组合样式
- 两人各自开发 UI 时不会产生 CSS 命名冲突

**配套依赖**：
- `class-variance-authority` — 组件变体管理（design_guid 已使用）
- `clsx` + `tailwind-merge` — 类名合并工具（design_guid 已使用）
- `@radix-ui/*` — 无样式基础组件（Select、Slider、Dialog 等）
- `lucide-react` — 图标库（design_guid 已使用）

## 5. 数学计算：手写 + 参考成熟实现 + 单测守护

**选择**：在 `core/physics/` 中手写所有数学工具函数，参考业界成熟实现，配套单元测试

### 5.1 为什么手写而非引入第三方库

经调研市面上的轻量数学/几何/ODE 库后，结论是没有一个库完美匹配本项目需求：

| 候选库 | 淘汰理由 |
|--------|---------|
| Math.js（>600KB） | 只用其中 <1%，包体积严重浪费；内部模块耦合度高，截取单个函数会带出一串依赖，tree-shaking 效果差 |
| gl-matrix（~9KB gzip） | 偏 3D/WebGL，vec2 模块 API 操作 Float32Array，不符合 2D 物理直觉 |
| @math.gl/core | 为 WebGL/WebGPU 可视化设计，含大量 3D 抽象（Matrix4、Quaternion），对纯 2D 过重 |
| vecti（<2KB） | 不可变设计在 60fps 物理循环中产生大量 GC 压力；有 7 个依赖 |
| victor（~2KB gzip） | 无原生 TS 类型、不支持 tree-shaking、长期未更新 |
| @thi.ng/vectors | tree-shaking 优秀，但 API 偏函数式（操作数组），依赖链深 |
| ode-rk4 / integrate | 8-11 年前发布，无 TS 支持，已停止维护 |
| odex（原生 TS） | 自适应步长 Bulirsch-Stoer 法，对高中物理过重 |
| geometric（~5-8KB gzip） | API 简洁但非原生 TS |
| @flatten-js/core（~50KB） | 功能过于完备（含布尔运算等），体积偏大 |

**根本原因**：本项目所需的全部数学能力（向量、积分、方程、几何），手写总代码量预估在 **300-500 行 TypeScript** 以内。这个量级引入第三方库的维护成本和适配成本远高于手写收益。

### 5.2 积分器选型修正

调研中的重要发现：**RK4 不是物理模拟的最佳默认选择**。

| 积分器 | 特性 | 适用场景 |
|--------|------|---------|
| Semi-implicit Euler | 辛积分器，保能量，3-5 行代码 | 大多数高中物理场景（匀变速、圆周运动等） |
| Velocity Verlet | 辛积分器，保能量，二阶精度，~10 行代码 | 需要更高精度时（弹簧振子长时间模拟等） |
| RK4 | **非辛积分器，不保能量**，长时间模拟有能量漂移 | 非守恒系统（电磁感应单棒 `m·dv/dt = -B²L²v/R`） |

**修正方案**：
- **默认积分器**：Semi-implicit Euler（够用且保能量）
- **高精度场景**：Velocity Verlet（弹簧振子、天体轨道等周期运动）
- **非守恒 ODE**：RK4（仅用于电磁感应等阻尼系统，这类场景本身不需要保能量）

### 5.3 参考实现来源

不是凭空写，而是参考成熟实现后用 TypeScript 重写：

| 模块 | 参考来源 | 说明 |
|------|---------|------|
| Vec2 向量运算 | matter.js `src/geometry/Vector.js` | 简洁高效的可变 2D 向量实现，API 直观 |
| 积分器 | [Gaffer on Games - Integration Basics](https://gafferongames.com/post/integration_basics/) | 游戏物理积分的经典教程，涵盖 Euler/Verlet/RK4 对比 |
| 几何算法 | geometric 库源码 | 纯函数风格，易移植为 TypeScript |
| 方程求解 | 直接手写 | 求根公式、克拉默法则，代码量极小（5-10 行/个） |

### 5.4 可靠性保障：单元测试守护

每个数学函数配套单元测试，用**手算已知结果**做断言：

| 模块 | 测试策略 |
|------|---------|
| 向量运算 | 用特殊角度（0°/30°/45°/90°）验证旋转、分解；验证单位向量模长为 1 |
| 积分器 | 用有解析解的场景对比：简谐运动 x=Acos(ωt) 对比 Verlet 数值解，误差 < 0.1% |
| 方程求解 | 覆盖正常情况 + 边界情况（判别式=0、a≈0 退化、无实根） |
| 几何判断 | 覆盖内部/外部/恰好在边界三种情况；平行线/重合线等退化情况 |

### 5.5 将要实现的模块

| 模块 | 内容 | 预估代码量 |
|------|------|-----------|
| `vector/` | magnitude、rotate、add、subtract、dot、decompose、scale、normalize | ~80 行 |
| `geometry/` | pointInRect、pointInCircle、pointToLine、lineIntersect | ~80 行 |
| `integrators.ts` | semiImplicitEuler、velocityVerlet、rk4Step | ~60 行 |
| `algebra.ts` | solveQuadratic、solveLinear2x2 | ~30 行 |
| `constants.ts` | g=9.8、k(库仑常数)、μ₀ 等物理常数 | ~20 行 |

### 5.6 后备方案

如果后期遇到复杂自适应步长 ODE 需求（混沌系统、刚度问题），可引入 `odex`（原生 TS、自适应 Bulirsch-Stoer 法、~10-15KB gzip）。但高中物理阶段大概率用不到。

## 6. 公式渲染：Phase 1 暂不引入

**选择**：Phase 1 不引入 KaTeX，用纯文本 + Canvas fillText 标注

**理由**：
- Phase 1 的公式展示需求很弱：力箭头旁的 "G=19.6N" 用 Canvas fillText 即可
- 信息面板的实时数值（v=3.2m/s、a=9.8m/s²）用 HTML + Unicode 下标足够
- KaTeX 主要用于"详细"信息密度下的公式推导（如 v²=v₀²+2ax），Phase 1 优先级不高
- 后续需要时可无缝引入 KaTeX（只影响信息面板的渲染组件，不影响架构）
