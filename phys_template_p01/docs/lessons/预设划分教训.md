# 预设划分教训：条件变体不等于独立预设

> **背景**：力学域阶段1~5 开发（2026-03-19）
> **类型**：设计偏差复盘，供后续开发参考

---

## 发生了什么

力学域开发时，将模型数据文档中 FM-001 的 8 个条件变体（FM-001-A~H）直接映射为多个独立预设和 qualifier 变体：

```
FM-001-A 光滑·静止·无外力     → horizontal-block.json
FM-001-C 粗糙·静止·水平推力   → horizontal-with-force.json
FM-002   减速                 → friction-deceleration.json
FM-002   加速                 → friction-acceleration.json
```

结果：4 个预设做的是**同一个物理模型**（水平面上的物块），老师想从"静止"演示到"加速"需要切换预设。

而需求文档的交互描述是："选模型 → 设条件 → 自动分析"——应该是 1 个统一场景，条件用参数切换。

## 根因

1. **计划阶段基于模型数据文档的穷举表拆分任务**，而没有回溯产品需求文档的交互描述
2. **CLAUDE.md 的"枚举注册"策略被误读**——本意是"不做通用物理引擎"，但被 AI 理解为"每种条件组合都需要独立注册"
3. **求解器代码实际上是统一的**（`block-on-slope.ts` 内部一个函数处理 4 种场景），但预设层面仍然做了碎片化拆分

## 修复方向

将同一物理模型的多个条件变体预设合并为统一场景，用 `select`/`toggle` 参数替代独立预设。合并后单个场景覆盖的条件组合反而更多（参数自由组合 vs 预定义的有限变体）。

### 力学域已实现预设的具体合并方案

**水平面 4 → 1**：

| 原预设 | 合并后 |
|--------|--------|
| `horizontal-block.json`（光滑·静止） | 合并为 1 个「水平面受力」预设 |
| `horizontal-with-force.json`（有外力） | 条件参数：接触面（toggle：光滑/粗糙） |
| `friction-deceleration.json`（减速） | 条件参数：外力（toggle：有/无） |
| `friction-acceleration.json`（加速） | 物理参数：μ、F、初速度等 |

求解器根据 `paramValues` 逐力判断：μ>0 → 有摩擦力，F>0 → 有外力，v₀>0 → 有初速度。

**斜面 4 → 1**：

| 原预设 | 合并后 |
|--------|--------|
| `slope-static.json`（静止） | 合并为 1 个「斜面受力」预设 |
| `slope-sliding-down.json`（下滑） | 条件参数：接触面（toggle：光滑/粗糙） |
| `slope-sliding-up.json`（上冲） | 条件参数：运动状态（select：静止/下滑/上冲） |
| `slope-smooth.json`（光滑） | 物理参数：θ、μ、m、初速度等 |

**悬挂 3 → 保持 3（用 group 分组）**：

| 原预设 | 处理 |
|--------|------|
| `single-rope-suspension.json` | 保留，加 `group: "suspension"` |
| `double-rope-symmetric.json` + `double-rope-asymmetric.json` | 合并为 1 个「双绳悬挂」，对称/不对称用角度参数自然覆盖 |
| `rope-rod-suspension.json` | 保留，加 `group: "suspension"` |

## 教训

- **始终以产品需求文档的交互描述为准**，模型数据文档的穷举表是物理完备性罗列，不是产品场景划分
- **给 AI 的规则要精确**：写"枚举注册"时需明确"按实体拓扑枚举，不按条件组合枚举"
- **做预设前先问：老师在课堂上会不会来回切换这些条件？** 如果会，就应该是同一个场景

## 预设划分的两个维度

### 同一拓扑内的条件变体 → 合并为 1 个预设

实体组成（类型和数量）不变，只是参数/条件不同。用 `select`/`toggle` 参数切换。

例：水平面上的物块——光滑/粗糙、有无外力、静止/运动，都是 block + surface，应该是 1 个预设。

### 不同拓扑但同一模型类别 → 独立预设 + `group` 分组

实体组成变了（需要增删实体），但属于同一个物理模型类别。用独立预设文件，通过 `group` 字段让 UI 呈现为同一入口下的子类型选择。

例：悬挂模型——单绳（pivot×1+rope×1）vs 双绳（pivot×2+rope×2）vs 绳+杆（pivot+rope+rod），拓扑不同、求解器不同，但需求文档中属于同一个「悬挂模型」入口，用 group 分组。

```
┌──────────────────────────────┐
│ 悬挂模型                      │
│  ○单绳  ●双绳  ○绳+杆         │  ← 子类型切换（实际是切换预设）
│  ─────────────────────        │
│  质量 m  [====●===] 2kg       │  ← 同名参数跨切换保持
│  左绳角度 α [==●===] 30°      │
│  右绳角度 β [===●==] 45°      │
└──────────────────────────────┘
```

## 电磁域的情况

电磁域天然没踩这个坑——`cyclotron-motion` 通过参数调整 charge/mass/v₀/B 覆盖多种物理情况，没有按条件拆分。但后续扩展时仍需注意：

- P08 粒子在磁场中的边界模型：需求文档描述为 1 个统一场景（边界类型用参数切换），不要按边界类型拆成独立预设
- P13 单棒/双棒模型：拓扑确实不同（纯电阻 vs 含电容需要增加实体），拆分是合理的
