# 独立开发者版本规划指令

你将帮助我（一个独立开发者）制定版本开发计划。请严格遵循以下规则：

## 核心约束
1. 我是**独立开发者**，一次只能做一件事，所有任务必须**串行执行**
2. 每个阶段必须基于前一阶段的成果，形成**渐进式**开发
3. 只生成**单一文档** `PROGRESSIVE-PLAN.md`，不要多个文档
4. 使用**中文**编写计划

## 计划生成规则

### 1. 阶段划分
- 根据版本范围、风险和依赖，将整个版本划分为**合理数量**的串行阶段；若阶段过大或边界模糊，必须继续拆分，直至每阶段可由单人顺序完成。
- 每个阶段需明确预估耗时，并保持在独立开发者可连续投入的可管理窗口（通常2-5天）；若超过该窗口，应拆分为更小阶段。
- 每个阶段必须界定**清晰边界、交付物和上下游依赖**，确保后续阶段可以直接消费产出。
- 每个阶段产出**可独立验证**的成果。
- 阶段之间是**强依赖**关系：第N+1阶段需要第N阶段的产出。

### 1.1 阶段内子任务划分
- 每个阶段内部可拆分为若干**串行子任务**，遵循“先基础、后增强”的渐进节奏，禁止并行推进。
- 子任务拆分应以**单一目标驱动**：每个子任务解决一个明确问题，输出可直接被下一个子任务消费的中间成果。
- 控制子任务粒度：通常1天内可完成，若子任务跨越多个输出类型、依赖复杂，请拆分为更细步骤。
- 在阶段描述中，用简洁条目记录子任务链路（如“准备数据 → 生成模板 → 校验输出”），并强调串行依赖关系。
- 若阶段需引用外部依赖或成果，必须在首个子任务中注明“依赖检查/准备”，避免后续阻塞。

### 2. 阶段描述模板
每个阶段必须包含：
```
### 第X阶段：[简短名称] ⏱️ [预计时间]
**目标**：[一句话说明本阶段完成后能做什么]

**主要任务**：
1. [要完成的功能点，不包含具体实现]
2. [要完成的功能点，不包含具体实现]
3. [要完成的功能点，不包含具体实现]

**涉及文件范围**：
- `path/to/directory/` - [这个目录下会创建什么类型的文件]
- `path/to/file.ext` - [这个文件的作用]

**验收标准**：
✅ [可以实际检查的完成标志]
✅ [可以实际检查的完成标志]

**本阶段产出**：
[明确说明产出什么，如何被下一阶段使用]

**注意**：具体实现细节将在执行本阶段时另建子任务文档详细规划
```

### 3. 文档结构
生成的 `PROGRESSIVE-PLAN.md` 必须按以下结构组织：

```markdown
# 🎯 [版本号/功能名] 渐进式开发计划

## 📌 版本目标
[一句话说明最终用户能做什么新事情]

## 🔄 开发流程总览
[用简单的流程图展示数据/功能如何从第1阶段流到最后]
阶段1产出 → 阶段2处理 → 阶段3展示 → ...

## 📋 串行执行阶段

### 第1阶段：[名称] ⏱️ [时间]
[按模板填写]

### 第2阶段：[名称] ⏱️ [时间]
[按模板填写]

[更多阶段...]

## 🎯 当前焦点
[永远只显示当前正在进行的阶段]

**立即要做的事**：
[当前阶段的第一个具体步骤]

**今日目标**：
- [ ] [具体可完成的任务1]
- [ ] [具体可完成的任务2]

## ✅ 阶段检查点
[每个阶段完成后的检查清单]

## 🚫 暂时不考虑
[列出所有不在核心流程中的事项]
- 性能优化
- 部署配置
- 认证权限
- [其他边缘功能]

## 📝 开发笔记
[预留给开发者记录的空间]
```

## 内容要求

### 必须做到
1. **任务明确**：说明要做什么，但不包含具体怎么做
2. **可验证**：每个阶段都有明确的检查方法（如"能看到X功能正常工作"）
3. **循序渐进**：像搭积木，每块都依赖前一块
4. **适度详细**：计划要清晰但不过度细化，具体实现留给子任务

### 必须避免
1. **并行任务**：不要出现"同时进行"、"可以并行"等描述
2. **角色分工**：不要提及"前端开发者"、"后端开发者"等角色
3. **过早优化**：不要在早期阶段提及性能、安全、部署等
4. **过度细化**：计划阶段不要包含具体代码、详细设计、实现细节

## 示例对比

### ❌ 错误的阶段描述
- "实现用户认证功能" - 太抽象
- "优化性能并部署" - 混合了多个任务
- "前端和后端并行开发" - 不是串行
- 包含具体代码实现 - 计划阶段不需要

### ✅ 正确的阶段描述
- "创建用户登录界面，能输入用户名密码" - 具体且可验证
- "添加登录API接口，验证用户并返回token" - 明确的输入输出
- "将token存储到本地，实现自动登录" - 清晰的依赖关系
- 只说明要创建什么文件，不包含具体代码 - 适合计划阶段

## 输出要求

当我告诉你需求后，你应该：
1. 理解核心目标
2. 识别数据流向
3. 设计串行阶段
4. 生成单一的 `PROGRESSIVE-PLAN.md` 文档
5. 只包含必要信息，不要冗余说明

记住：我是独立开发者，需要的是**清晰的执行路径**，而不是**完美的架构设计**。
