## 分析要求

按照 `PROGRESSIVE-PLAN.md` 中的**综合分析模板**产出 `outputs/math-summary.md`。

### 第一步：模块间相似度评估

从以下维度构建对比矩阵，识别相似聚类：

| 维度 | 说明 |
|------|------|
| 交互模式 | 编辑器型 / 参数驱动型 / 模拟演示型 / 混合型 |
| 渲染技术 | 2D Canvas / SVG / 3D WebGL / 混合 |
| 实体系统需求 | 重度（多种实体类型）/ 轻量 / 无 |
| undo/redo 需求 | 完整命令系统 / 简单操作栈 / 不需要 |
| 核心计算引擎 | 几何构建 / 表达式解析 / 数值分析 / 统计模拟 / 向量运算 |
| 用户交互深度 | 深度编辑（创建/修改/删除对象）/ 参数调节 / 观察演示 |

### 第二步：暂分组并分析

基于对比矩阵将相似模块暂分组。已知的初步线索（从分析卡片得出）：

- **M-01** 独立性强：唯一的重度 3D 编辑器，七系统架构，复杂度远超其他模块
- **M-02 / M-03 / M-04 / M-06** 可能一组：都涉及 2D Canvas 坐标系渲染、参数联动，部分需要轻量实体
- **M-05** 独立性强：唯一的模拟演示型，Web Worker + D3.js，无实体需求
- **M-06 的 3D 部分**与 M-01 有复用关系，需要评估归属

**但以上仅是初步线索，需要在分析中验证或推翻。**

### 第三步：分组验证（核心步骤）

对每个暂定分组回答三个问题：

1. **共享核心层的接口能否清晰定义？**
   - 能用一两句话说清楚这个核心层做什么吗？
   - 接口边界在哪里？

2. **核心层能支撑各模块的哪些功能？**
   - 逐模块列出：哪些功能走核心层，哪些功能是模块专属
   - 核心层覆盖率高 → 分组合理
   - 核心层覆盖率低（大部分功能是模块专属）→ 分组可能勉强

3. **各模块的专属能力是什么？**
   - 专属能力的实现复杂度如何
   - 专属能力之间是否有冲突

### 第四步：产出仓库划分与架构方案

对每个最终确定的仓库：

- 包含哪些模块，划分依据
- 核心架构模式（如编辑器 Entity-Command-Signal / 参数驱动 State-Compute-Render / 模拟引擎 Setup-Simulate-Visualize）
- 核心技术依赖
- 通用层 vs 模块专属层的边界定义
- 项目结构概览

### 特别注意

- 不要为了归组而归组：如果某个模块确实独立性强，单独成仓是合理的
- M-06 的 2D/3D 双模式是一个关键决策点：3D 部分与 M-01 共享架构，还是 M-06 整体与 M-02/M-03/M-04 归为一组
- 架构设计要考虑"编辑能力的统一性"：选中、撤销重做、实体管理等通用编辑能力，在需要的模块中应有统一抽象，而不是各写一套
