## 1. 模块间相似度评估

### 1.1 七维对比矩阵

| 维度 | C-02 分子查看器 | C-03 方程式配平 | C-04 元素周期表 | C-05 晶体查看器 | C-06 电化学演示 | C-07 反应速率平衡 | C-08 酸碱滴定pH | C-09 有机路径图 |
|------|----------------|----------------|----------------|----------------|----------------|------------------|----------------|----------------|

### 1.2 关键观察

**全线共同特征（与数学/物理产品线的根本差异）：**
- **8 个模块全部不需要实体系统和 undo/redo** — 全部为查看器/工具/演示器，无一是编辑器型
- **全部为预置数据 + 前端计算** — 无用户自定义实体创建
- **React + TypeScript** 为统一基础栈

**渲染技术高度异构：**
- Three.js WebGL：仅 C-02、C-05（2个）
- Canvas 2D：C-06(部分)、C-07（2个）
- SVG：C-06(部分)、C-09（2个）
- DOM/CSS/Chart：C-03、C-04、C-08（3个）
- 无任何一种渲染技术覆盖超过 3 个模块

**算法引擎完全独立：** 每个模块的核心算法引擎互不相关，无法共享。

### 1.3 相似度聚类

基于矩阵，识别出以下相似聚类：

| 聚类 | 模块 | 相似依据 | 相似强度 |
|------|------|----------|----------|
| A: 3D 查看器 | C-02 + C-05 | Three.js 3D渲染管线、OrbitControls、原子球体/键渲染、Raycasting、CPK配色 | **强** |
| B: 动画模拟器 | C-06 + C-07 | 粒子动画、播放/暂停/调速控制、模型配置驱动 | **中** |

## 2. 分组验证

### 2.1 聚类 A：3D 化学查看器（C-02 分子 + C-05 晶体）

**Q1：共享核心层的接口能否清晰定义？**

> **能。** 核心层 = "化学 3D 场景引擎"：管理 Three.js 场景生命周期、相机控制(OrbitControls)、原子/键的球棍渲染管线、Raycasting 拾取交互、CPK 配色体系、显示模式切换（球棍/空间填充/线框）、标注系统（键长/键角/配位数文本标签）。
>
> **接口边界：** 核心层提供"给定原子坐标和键连接关系，渲染 3D 分子/晶体场景并处理交互"的能力。坐标如何计算（Z-matrix vs 分数坐标转换）属于模块专属层。

**Q2：核心层能支撑各模块的哪些功能？**

| 功能 | C-02 分子查看器 | C-05 晶体查看器 |
|------|:-:|:-:|
| Three.js 场景初始化/销毁 | ✅ 核心层 | ✅ 核心层 |
| OrbitControls 相机控制 | ✅ 核心层 | ✅ 核心层 |
| 原子球体渲染(InstancedMesh) | ✅ 核心层 | ✅ 核心层 |
| 化学键圆柱渲染 | ✅ 核心层 | ✅ 核心层 |
| CPK 配色系统 | ✅ 核心层 | ✅ 核心层 |
| Raycasting 拾取 | ✅ 核心层 | ✅ 核心层 |
| 显示模式切换 | ✅ 核心层 | ✅ 核心层 |
| 文本标注(键长/键角/配位数) | ✅ 核心层 | ✅ 核心层 |

| 空隙位置计算 | — | 🔶 专属 |

**核心层覆盖率：高。** 8/8 项渲染和交互功能可共享，专属部分集中在坐标计算和业务数据。

**Q3：各模块专属能力**

- **C-02 专属**：Z-matrix 坐标计算、VSEPR 几何推断、80+ 分子数据管理、分子对比多视口。实现复杂度中等，与 C-05 专属能力无冲突。
- **C-05 专属**：分数坐标转换、晶胞扩展算法、配位多面体、堆积动画、空隙计算。实现复杂度中高，与 C-02 专属能力无冲突。

**验证结论：✅ 分组合理。** 3D 渲染管线共享度极高，专属能力互不冲突，核心层接口清晰。这是化学产品线中最有价值的共享机会。

### 2.2 聚类 B：动画模拟器（C-06 电化学 + C-07 反应速率平衡）

**Q1：共享核心层的接口能否清晰定义？**

> **可以定义，但边界模糊。** 潜在核心层 = "粒子动画引擎 + 播放控制器"：管理动画时间轴（播放/暂停/逐步/调速）、粒子生命周期（创建/运动/销毁）、Canvas 2D 渲染循环。
>
> **但存在本质差异：**
> - C-06 的粒子是**确定性路径动画**（电子沿导线、离子沿电解质，路径由模型配置 JSON 预定义）
> - C-07 的粒子是**随机物理模拟**（分子随机运动、碰撞检测、基于活化能的有效碰撞判定）
>
> 两者的粒子运动模型完全不同：一个是路径插值，一个是物理引擎。

**Q2：核心层能支撑各模块的哪些功能？**

| 功能 | C-06 电化学演示 | C-07 反应速率平衡 |
|------|:-:|:-:|
| 播放/暂停/逐步/调速控制 | ✅ 可共享 | ✅ 可共享 |
| Canvas 2D 渲染循环(rAF) | ✅ 可共享 | ✅ 可共享 |
| 粒子渲染(圆形/图标绘制) | ✅ 可共享 | ✅ 可共享 |
| SVG 装置图渲染 | 🔶 专属 | — |
| 确定性路径动画引擎 | 🔶 专属 | — |
| 电极反应方程式同步 | 🔶 专属 | — |
| 13个电池模型配置JSON | 🔶 专属 | — |
| 随机粒子运动物理引擎 | — | 🔶 专属 |
| 碰撞检测+有效碰撞判定 | — | 🔶 专属 |
| 化学动力学方程求解 | — | 🔶 专属 |
| 宏观图表实时联动 | — | 🔶 专属 |
| 条件改变交互(温度/浓度/压强) | — | 🔶 专属 |

**核心层覆盖率：低。** 仅 3 项基础设施可共享（播放控制、渲染循环、粒子绘制），核心业务逻辑（动画引擎本身）完全不同。

**Q3：各模块专属能力**

- **C-06 专属**：SVG 装置图层、路径动画引擎、电极反应同步、模型配置系统。复杂度集中在动画编排。
- **C-07 专属**：物理粒子引擎、碰撞检测、动力学方程、实时图表。复杂度集中在模拟计算。
- 两者专属能力无冲突，但**共享层太薄**（播放控制器本身极其轻量，约 50-100 行代码）。

**验证结论：❌ 分组勉强。** 共享核心层太薄，不足以支撑独立仓库的共享层。播放控制器作为一个通用 React hook 即可，不构成仓库级共享依据。C-06 和 C-07 业务逻辑差异大，强行合仓会导致共享层形同虚设。

### 2.3 聚类 C 验证：DOM 工具类（C-03 + C-04 + C-08）

**Q1：共享核心层的接口能否清晰定义？**

> **不能。** 三者唯一共同点是"用 DOM 渲染"，但：
> - C-03 是**文本处理型**（化学式解析 → 配平算法 → 纠错引擎）
> - C-04 是**数据密集型**（118 元素数据库 → CSS Grid 表格 → 属性可视化）
> - C-08 是**参数计算型**（滴定参数 → pH 公式计算 → 图表渲染）
>
> 无法用一两句话定义一个有意义的共享核心层。

**验证结论：❌ 不构成分组。** 仅因渲染层相同而归组无意义，三者应各自独立评估。

### 2.4 跨模块共享组件分析：化学式渲染

任务文档提到"C-03、C-06、C-07 都需要化学式/方程式渲染"。逐模块验证：

| 模块 | 化学式渲染需求 | 复杂度 |
|------|---------------|--------|
| C-03 方程式配平 | **核心功能** — 化学式解析、上下标渲染、配平系数标注、逐步高亮 | 高（需交互式编辑） |
| C-06 电化学 | 电极反应方程式展示 | 低（静态展示） |
| C-07 反应速率 | 反应方程式展示 | 低（静态展示） |
| C-04 周期表 | 元素符号展示 | 极低 |
| C-02 分子 | 分子式标题展示 | 极低 |

**结论：** C-03 的化学式渲染是完整的解析+交互系统，远超其他模块"静态展示化学式"的简单需求。抽取一个通用"化学式渲染组件"的投入产出比不高 — C-06/C-07 只需一个简单的 `<ChemFormula>` React 组件（将 `H2SO4` 渲染为 H₂SO₄），这可以作为一个独立 npm 包或仓库内 shared 工具，无需因此影响仓库划分。

## 3. 仓库划分结论

### 最终方案：7 个仓库 + 1 个共享包

```
仓库 1: chem-3d-viewer        → C-02(分子) + C-05(晶体)
仓库 2: chem-equation-balance  → C-03(方程式配平)
仓库 3: chem-periodic-table    → C-04(元素周期表)
仓库 4: chem-electrochem       → C-06(电化学演示)
仓库 5: chem-kinetics          → C-07(反应速率与平衡)
仓库 6: chem-titration         → C-08(酸碱滴定)
仓库 7: chem-organic-pathways  → C-09(有机路径图)

共享包: @chem/formula-render   → 化学式渲染组件(轻量)
```

### 划分依据

| 决策 | 依据 |
|------|------|
| **C-02 + C-05 合仓** | 3D 渲染管线共享度极高（8项核心功能可共享），专属能力互不冲突，是唯一通过分组验证的聚类 |
| **C-06 独立** | 虽与 C-07 都有粒子动画，但动画引擎本质不同（确定性路径 vs 随机物理），共享层太薄，独立更清晰 |
| **C-07 独立** | 物理模拟引擎 + 实时图表联动是其核心，与任何其他模块无有效共享 |
| **C-03 独立** | 纯 DOM 文本处理型，核心是算法引擎（解析+配平+纠错），与所有其他模块技术栈完全不同 |
| **C-04 独立** | 大规模数据管理型（118元素），DOM+CSS Grid 为主，无技术交集 |
| **C-08 独立** | P3 级轻量小品模块，DOM+Chart 库，体量小但业务独立 |
| **C-09 独立** | 唯一的网络图可视化（SVG+D3.js），交互模式独特 |

### 为什么是 7 仓库而非更少？

化学产品线的**技术异构性远高于物理产品线**（物理 12 个模块全用 2D Canvas，化学 8 个模块横跨 5 种渲染技术）。除 C-02 + C-05 外，其余模块在渲染技术、算法引擎、交互模式上均无有效共享面。强行合仓只会制造"名义共享、实际独立"的虚假架构。

**每个独立仓库的体量评估：**

| 仓库 | 预估体量 | 是否适合独立 |
|------|----------|-------------|
| chem-3d-viewer (C-02+C-05) | 中大（共享3D引擎+两套业务逻辑） | ✅ 共享价值高 |
| chem-equation-balance (C-03) | 中（算法引擎复杂） | ✅ 算法独立性强 |
| chem-periodic-table (C-04) | 中（数据管理复杂） | ✅ 数据独立性强 |
| chem-electrochem (C-06) | 中（13模型+动画引擎） | ✅ 动画系统独立 |
| chem-kinetics (C-07) | 中大（物理模拟+图表联动） | ✅ 模拟引擎独立 |
| chem-titration (C-08) | 小 | ✅ 虽小但业务完整独立 |
| chem-organic-pathways (C-09) | 小中 | ✅ D3.js 技术栈独特 |

## 4. 每个仓库的架构与技术方案

### 4.1 chem-3d-viewer（C-02 分子 + C-05 晶体）

**核心架构模式：** 共享 3D 引擎 + 模块化业务插件

**核心技术依赖：**
- React + TypeScript
- Three.js (3D WebGL 渲染)
- @react-three/fiber 或原生 Three.js（根据开发偏好选择）
- @react-three/drei（OrbitControls 等工具）

**通用层 vs 专属层边界：**

```
通用层 (packages/core-3d/)
├── scene/          — Three.js 场景初始化、相机、灯光、销毁
├── controls/       — OrbitControls 封装、自动旋转、重置视角
├── rendering/      — 原子球体(InstancedMesh)、化学键圆柱、显示模式切换
├── interaction/    — Raycasting 拾取、hover 高亮、点击选中
├── styling/        — CPK 配色方案、原子半径数据
├── annotation/     — 3D 文本标签（键长/键角/配位数标注）
└── types/          — 共享类型定义(Atom, Bond, SceneConfig)

专属层 - 分子模块 (packages/molecule-viewer/)
├── data/           — 80+分子 JSON 数据库
├── compute/        — Z-matrix→笛卡尔坐标、VSEPR 几何推断
├── features/       — 分子对比多视口、分子信息面板
└── ui/             — 分子选择器、搜索、参数控制面板

专属层 - 晶体模块 (packages/crystal-viewer/)
├── data/           — 15 晶体模型数据
├── compute/        — 分数坐标转换、晶胞扩展、配位计算、空隙计算
├── features/       — 配位多面体渲染、堆积动画(播放控制)
└── ui/             — 晶体选择器、显示选项面板
```

**项目结构概览：**
```
chem-3d-viewer/
├── packages/
│   ├── core-3d/          — 共享 3D 引擎（仓库核心价值）
│   ├── molecule-viewer/  — C-02 分子查看器应用
│   └── crystal-viewer/   — C-05 晶体查看器应用
├── apps/
│   ├── molecule/         — 分子查看器入口（Vite dev server）
│   └── crystal/          — 晶体查看器入口
├── package.json          — monorepo 根配置
└── pnpm-workspace.yaml
```

### 4.2 chem-equation-balance（C-03 方程式配平）

**核心架构模式：** 算法引擎 + 数据驱动 UI

**核心技术依赖：**
- React + TypeScript
- 无额外渲染库（纯 DOM）

**架构分层：**

```

├── parser/         — 化学式词法/语法解析器（支持嵌套括号、水合物、离子）
├── balancer/       — 配平算法（待定系数法、观察法等 7 种方法）
├── validator/      — 纠错规则引擎（7 大类错误识别）
├── ionic/          — 离子方程式拆分规则引擎

UI 层 (src/ui/)

├── step-display/    — 配平步骤逐步演示
├── error-display/   — 纠错标注展示
├── exercise/        — 练习模式（题库+评分）
└── common/          — 化学式渲染组件（上下标）

数据层 (src/data/)
├── exercises/       — 配平练习题库
├── error-rules/     — 纠错知识库
└── ion-rules/       — 强弱电解质拆分规则
```

**项目结构概览：**
```
chem-equation-balance/
├── src/
│   ├── engine/       — 算法引擎（核心价值）
│   ├── ui/           — React UI 组件
│   └── data/         — 规则库与题库
├── tests/
│   └── engine/       — 算法单元测试（重点覆盖）

### 4.3 chem-periodic-table（C-04 元素周期表）

**核心架构模式：** 数据驱动型交互应用

**核心技术依赖：**
- React + TypeScript
- CSS Grid（周期表布局）
- SVG（电子排布动画）

**架构分层：**

```
数据层 (src/data/)

├── schema.json        — JSON Schema 校验（数据质量保障）
├── electron-config/   — 电子排布规则数据

### 4.8 共享包：@chem/formula-render

**用途：** 跨仓库共享的化学式渲染 React 组件

**功能：**
- 将化学式文本（如 `H2SO4`、`Ca(OH)2`、`Fe3+`）渲染为正确的上下标格式
- 支持化学方程式渲染（系数 + 反应物 → 生成物 + 条件）
- 纯展示型组件，不含解析算法（C-03 的完整解析器是其专属能力）

**发布为独立 npm 包**，各仓库按需引用。体量极小（预估 200-400 行），维护成本低。

**使用方：** C-03（部分复用）、C-06、C-07，以及未来可能的物理/数学模块。
