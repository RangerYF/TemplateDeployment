## 1. 模块间相似度评估

### 1.2 暂分组与依据

基于对比矩阵，识别出三个相似聚类：

**聚类 A — M-01 独立**：3D 编辑器
- 唯一的重度 3D WebGL 模块，七系统编辑器架构
- 实体系统复杂度（9 种类型）远超其他模块
- 计算引擎（几何体 Builder/Calculator/Unfolding/Projection）完全专属
- 与其他任何模块没有核心层可共享（仅有 KaTeX、设计规范等边缘共享）

**聚类 B — M-02 / M-03 / M-04 / M-06(2D)**：2D 坐标系 + 函数/曲线
- 共同特征：2D Canvas 坐标系渲染、参数滑块联动、曲线/图形绘制
- M-02 是核心（表达式解析 + 曲线引擎），M-03/M-04 是其特化场景
- M-06 的 2D 部分与此组共享坐标系和交互模式

**聚类 C — M-05 独立**：模拟演示型
- 唯一的模拟演示型模块，架构范式完全不同
- Web Worker 大规模模拟 + D3.js SVG 图表渲染
- 不需要坐标系曲线引擎、不需要表达式解析器
- 与聚类 B 的共享仅限于基础 UI 组件（滑块、KaTeX）

**M-06 的归属问题**：
- M-06 有 2D/3D 双模式——2D 部分与聚类 B 相似（共享坐标系），3D 部分与 M-01 有组件复用关系
- 需要在分组验证中重点讨论

## 2. 分组验证

### 2.1 聚类 A 验证：M-01 独立

#### 共享核心层接口定义

**不适用**——M-01 作为独立模块，不存在"与谁共享核心层"的问题。

#### M-01 独立的理由

1. **架构差异极大**：M-01 是唯一需要完整 Entity-Command-Signal-Tool-Selection-Inspector-Renderer 七系统架构的模块。其他模块最多需要轻量实体管理 + 简单操作栈，架构复杂度差一个数量级。

2. **渲染技术独占**：Three.js + @react-three/fiber 是 M-01 的核心依赖，其他模块均以 2D Canvas/SVG 为主。将 Three.js 打入共享核心层会给不需要 3D 的模块带来巨大的包体积负担（Three.js 核心约 600KB min+gzip）。

3. **计算引擎完全专属**：13 种几何体 Builder、13 种 Calculator、11 种展开图算法、13 种三视图投影——这些全部是立体几何专属，零复用价值。

4. **代码量已经很大**：V0.3 阶段已有相当体量的代码，单独维护更清晰。

5. **M-06 3D 部分的复用**：M-06 需要 3D 渲染，但仅复用 M-01 的少量基础设施（Scene3D、OrbitControls、坐标系渲染器），不构成"合并仓库"的理由。可通过 npm 包或代码拷贝实现组件级复用。

**结论**：M-01 独立成仓，理由充分。✅

#### 需要讨论的问题

**M-05 是否应该加入此组？**

不应该。理由：
1. M-05 的核心范式是"模拟演示型"——运行模拟 → 观察动态过程，与聚类 B 的"参数驱动型"完全不同
2. M-05 不需要坐标系曲线引擎——它需要的是统计图表（柱状图/直方图/散点图），这是完全不同的渲染需求
3. M-05 不需要表达式解析器
4. M-05 的核心是 Web Worker 大规模模拟引擎，这在聚类 B 中完全没有对应
5. 如果强行合并，核心层只能退化为"KaTeX + 滑块"这种几乎无实质价值的公共层

**M-06 的 3D 部分怎么处理？**

M-06 有 2D/3D 双模式。两种处理方式：

- **方案 A：M-06 整体放入聚类 B，3D 部分通过组件复用 M-01**
  - 优点：M-06 的运算引擎 2D/3D 统一，放在一起开发更自然
  - 优点：M-06 的 2D 部分与聚类 B 共享坐标系引擎
  - 缺点：需要引入 Three.js 依赖（但可通过构建时 tree-shaking 和按需加载缓解）

- **方案 B：M-06 拆分——2D 放聚类 B，3D 放 M-01 仓库**
  - 优点：依赖关系更干净
  - 缺点：一个模块拆两个仓库，开发和维护成本高；运算引擎需要两仓复制

**推荐方案 A**——M-06 整体放入聚类 B。Three.js 仅在 3D 模式下按需加载，不影响 2D 模块的包体积。M-01 的 3D 基础组件（Scene3D、OrbitControls、坐标系渲染器）可通过内部 npm 包或直接复制引入。

**结论**：M-02 / M-03 / M-04 / M-06 同仓，分组合理。✅

### 2.3 聚类 C 验证：M-05 独立

#### M-05 独立 vs 并入聚类 B

**并入聚类 B 的论点**：
- 同属数学产品线，学科归属相同
- 有少量共享 UI 组件（KaTeX、滑块）
- 减少仓库数量

**独立的论点**：
1. **架构范式完全不同**：模拟演示型（Setup → Simulate → Visualize）vs 参数驱动型（State → Compute → Render）
2. **渲染技术栈差异大**：D3.js SVG 统计图表 vs Canvas 坐标系曲线绘制
3. **核心技术独有**：Web Worker 大规模模拟引擎、随机数生成器、统计量计算器——这些在聚类 B 中完全不需要
4. **不使用聚类 B 的核心能力**：不需要坐标系渲染引擎、不需要表达式解析器、不需要曲线绘制引擎
5. **共享层退化**：如果合并，"核心层"只剩 KaTeX + Tailwind + React DOM，这不是有实质价值的共享

**但是**——M-05 的代码量和复杂度相对有限（12 个功能模块，多数是"选实验 → 看结果"的简单交互），独立成仓是否值得？

**权衡结论**：M-05 并入聚类 B 仓库（edu-math-2d），但作为**独立模块目录**存在，不强制使用核心层的坐标系/曲线引擎。理由：
1. 减少仓库总数（1 个数学 2D 仓库比 2 个更易管理）
2. M-05 的统计图表（直方图、散点图）中的坐标轴渲染与核心层的坐标系渲染在底层有部分共性
3. 共享 UI 组件（参数滑块、KaTeX 公式、Canvas 容器、动画帧控制 Hook）仍有实际价值
4. 最终输出是独立 HTML 文件，构建时不会把不需要的模块打包进去
5. M-05 的体量不足以独立成仓

## 3. 仓库划分结论

### 最终方案：2 个仓库

```
数学产品线/
├── edu-math-3d/          # 仓库1：M-01 立体几何展示台
└── edu-math-2d/          # 仓库2：M-02 ~ M-06（2D 数学可视化）
```

| 仓库 | 包含模块 | 划分依据 |
|------|---------|---------|
| **edu-math-3d** | M-01 立体几何展示台 | 唯一重度 3D 编辑器；Seven-System 架构；Three.js 核心依赖；计算引擎完全专属；代码量大、复杂度高 |
| **edu-math-2d** | M-02 函数图像实验室、M-03 解析几何画板、M-04 三角函数演示台、M-05 概率统计模拟器、M-06 向量运算演示台 | 共享 2D Canvas 坐标系引擎 + 参数滑块联动 + KaTeX 公式渲染；M-02/M-03/M-04 强烈共享曲线绘制引擎和表达式解析；M-06 的 2D 共享坐标系，3D 按需加载；M-05 作为独立模块目录，共享基础 UI |

### 与已有方案的对比

已有的 `docs/仓库规划与技术方案.md` 提出了相同的 2 仓方案（edu-math-3d + edu-math-2d），本综合分析通过系统化的分组验证确认了该方案的合理性。关键一致点：
- M-01 独立（理由一致：3D 编辑器架构差异）
- M-02/M-03/M-04 共享 2D 坐标系引擎（理由一致：曲线绘制 + 参数滑块）
- M-06 整体归入 2D 仓库（理由一致：运算引擎统一、3D 部分按需加载）
- M-05 并入 2D 仓库但保持独立模块目录（理由一致：体量不足以独立成仓）

## 4. 每个仓库的架构与技术方案

### 4.1 edu-math-3d（M-01 立体几何展示台）

#### 核心架构模式

**Entity-Command-Signal 编辑器架构**（已有，成熟验证）

```
用户操作 → Tool 捕获 → Command 封装 → EntityStore 变更 → Signal 广播 → Renderer 更新 / Inspector 刷新
```

七系统架构：
| 系统 | 设计模式 | 已实现 |
|------|---------|--------|
| Entity（9 种类型） | Registry + 联合类型 | ✅ |
| Command（11 种 + Batch） | Command 模式 + History | ✅ |
| Signal（7 个信号） | 发布-订阅 | ✅ |
| Tool（7 种工具） | 策略模式 | ✅ |
| Selection | 单例 Store | ✅ |
| Inspector（9 种） | Registry + 自注册 | ✅ |
| Renderer（9 种） | Registry + 自注册 | ✅ |

#### 核心技术依赖

- React 19 + TypeScript + Vite 7
- Three.js + @react-three/fiber + @react-three/drei（3D 渲染）
- Zustand（5 个 Store）
- Tailwind CSS + KaTeX
- 构建输出：vite-plugin-singlefile → 单 HTML 文件

#### 通用层 vs 模块专属层

**全部为模块专属**——M-01 是单模块仓库，不存在通用层/专属层的区分。

但 M-01 有几个**可被外部复用的架构资产**（供 M-06 3D 部分参考）：
| 资产 | 可复用方式 |
|------|-----------|
| Scene3D 场景容器 + 灯光 + OrbitControls | 代码拷贝或 npm 包 |
| CoordSystemRenderer（坐标轴/刻度/标签） | 代码拷贝（需微调参数范围） |
| ToolEventDispatcher（拖拽事件分发） | 代码拷贝 |
| Signal 事件系统 | 设计模式参考 |
| Command + HistoryStore 模式 | 设计模式参考 |

#### 项目结构

```
edu-math-3d/                          # 即现有 visual_template 项目
├── src/
│   ├── editor/                       # 编辑器核心
│   │   ├── entities/                 # 9 种实体类型定义
│   │   ├── commands/                 # 11 种 Command + BatchCommand
│   │   ├── signals/                  # Signal 事件系统
│   │   ├── tools/                    # 7 种交互工具
│   │   └── stores/                   # 5 个 Zustand Store
│   ├── engine/                       # 计算引擎（纯函数，无 UI 依赖）
│   │   ├── builders/                 # 13 种几何体 Builder
│   │   ├── math/calculators/         # 体积/表面积/角度/距离计算
│   │   ├── unfolding/                # 11 种展开图算法
│   │   └── projection/               # 三视图投影
│   ├── components/
│   │   ├── scene/                    # 3D 场景（Renderer/ToolBar/EventDispatcher）
│   │   ├── panels/                   # Inspector/Parameter/EntityList
│   │   ├── views/                    # 展开图/三视图面板
│   │   ├── info/                     # 度量/计算步骤
│   │   ├── layout/                   # 布局

### 4.2 edu-math-2d（M-02 ~ M-06 数学 2D 可视化）

#### 核心架构模式

**参数驱动型架构**（State → Compute → Render），辅以轻量实体管理和简单 undo/redo。

```
┌───────────────────────────────────────────────────────┐
│                    UI 层 (React DOM)                   │
│   表达式输入 / 参数滑块 / 工具栏 / 模式切换 / 面板    │
├───────────────────────────────────────────────────────┤
│               各模块专属逻辑                           │
│   M-02: 多函数管理/变换/导数                          │
│   M-03: 圆锥曲线/焦点弦/离心率                       │
│   M-04: 单位圆/五点法/解三角形                        │
│   M-05: 模拟引擎/统计图表                             │

#### 构建策略

每个模块独立入口，构建时分别打包为独立单 HTML 文件：

```
dist/
├── function-lab.html      # M-02
├── analytic-geo.html      # M-03
├── trigonometry.html      # M-04
├── probability.html       # M-05
└── vector.html            # M-06
```

使用 Vite 多入口 + vite-plugin-singlefile 实现。各模块仅打包自身依赖 + 实际使用的核心层代码（tree-shaking 生效）。M-06 的 Three.js 依赖不会被打入其他模块。

#### 开发优先级建议

1. **Phase 1：M-02 函数图像实验室**（优先级最高）
   - 理由：M-02 是核心引擎的"试金石"——坐标系、曲线绘制、表达式解析、参数滑块、数值分析全部在 M-02 中验证
   - 产出：核心引擎 + M-02 完整功能
   - 表达式解析器是最大技术风险，需先攻克

2. **Phase 2：M-03 解析几何画板 + M-04 三角函数演示台**
   - 理由：复用 M-02 验证过的核心引擎，增量开发各自专属功能
   - M-03 新增：圆锥曲线计算 + 专题模式 + 直线工具
   - M-04 新增：单位圆引擎 + 解三角形 + 公式库

3. **Phase 3：M-05 概率统计模拟器 + M-06 向量运算演示台**
   - M-05 独立性强，主要工作在模拟引擎 + D3.js 图表
   - M-06 2D 部分复用坐标系引擎，3D 部分复用 M-01 组件

## 附录：关键技术风险汇总

| 风险 | 影响模块 | 等级 | 应对策略 |
|------|---------|------|---------|
| 表达式解析器的健壮性 | M-02/M-03 | 高 | 使用 math.js 成熟库，不自研；覆盖边界用例（嵌套括号/隐乘/特殊函数） |
| 滑块拖动 60fps 实时渲染 | 全部 | 中 | Canvas 立即模式渲染性能充裕；编译后表达式求值微秒级；必要时 debounce |
| 离心率连续演变的拓扑过渡 | M-03 | 中 | 使用焦点-准线统一定义参数化；e=1 附近做特殊处理 |
| M-06 Three.js 按需加载 | M-06 | 中 | React.lazy + Suspense；dynamic import 确保 2D 模式不加载 Three.js |
| 不连续函数渲染（tan/分段） | M-02/M-04 | 低 | 采样时检测不连续跳跃，断开路径；渐近线虚线处理 |
| Web Worker 通信延迟 | M-05 | 低 | 批量推送中间结果而非逐次；前 100 次动画后切快速模式 |
| 单 HTML 包体积 | 全部 | 中 | tree-shaking + 按模块独立打包；Three.js 仅 M-06 打入；D3.js 仅 M-05 打入 |
| 精确值输出（分数/根号化简） | M-02/M-03/M-04 | 低 | 优先数值近似 + 已知精确值查表；不做通用符号计算 |
