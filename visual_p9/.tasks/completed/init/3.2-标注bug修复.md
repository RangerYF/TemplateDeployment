## Bug 清单

### BUG-1：线段名字标签遮挡中点

**现象**：一条棱线同时有名字和中点时，名字标签挡住中点标签，两者视觉重叠。

**根因**：`EdgeLabels` 用 3D 空间的叉积偏移（`up × edge` 方向 0.15~0.2 单位）来避让，但 `<Html distanceFactor={8}>` 的缩放特性导致 3D 偏移量在不同视角/缩放下视觉效果不一致。某些角度下偏移不足以避免遮挡。

**修复方案**：
- 将 `EdgeLabels` 的位置改回精确中点（不做 3D 偏移）
- 改用 CSS `transform: translateY(-18px)` 做固定像素级偏移，使名字标签始终显示在中点标签上方
- 这样无论视角如何旋转，名字始终在中点上方固定距离处
- 自定义点标签也检查：确保 `CustomPointLabel` 的 `translateY(-14px)` 不与棱线名字冲突

**涉及文件**：`src/components/scene/GeometryRenderer.tsx`（`EdgeLabels` 组件）

### BUG-2：顶点拖拽行为不稳定

**现象**：点击顶点时，有时能正常选中，有时意外进入拖拽状态。交互行为不可预测。

**根因**：`DragHandles` 的不可见球体（Three.js mesh，半径 0.12）和 `VertexLabels`（HTML overlay）在同一位置竞争事件：
- 若鼠标点击命中 3D 球体 → `onPointerDown` 被球体捕获 + `stopPropagation` → 进入拖拽预备状态。虽然加了 4px 移动阈值，但球体已经吞掉了事件，VertexLabel 的 `onClick` 不会触发。
- 若鼠标点击命中 HTML 标签（标签遮挡了球体）→ DOM 事件走 HTML 路径 → 正常触发选中/编辑，不触发拖拽。
- 两个事件系统（Three.js pointer events vs DOM HTML events）在同一位置互相竞争，命中哪个取决于浏览器事件分发顺序和精确的鼠标位置。

**修复方案**：统一事件入口，消除两个系统的竞争。两种可选路线：

**方案 A（推荐）：标签驱动拖拽**
- 移除 `DragHandles` 组件（不可见球体）
- 将拖拽逻辑集成到 `VertexLabel` 的 HTML 元素上：
  - `onPointerDown` → 记录起始位置
  - `onPointerMove` → 超过阈值后激活拖拽（需要将屏幕坐标映射到 3D）
  - 单击 vs 拖拽通过移动阈值区分
  - 双击仍然进入编辑
- 优点：只有一个事件入口，行为完全确定
- 缺点：HTML 元素的 pointer 坐标需要额外转换到 3D 空间

**方案 B：球体驱动一切**
- 保留 `DragHandles` 不可见球体，但不 `stopPropagation`
- 球体 `onPointerDown` 仅记录拖拽预备状态
- 将选中/编辑逻辑也迁移到球体的 `onClick` / `onDoubleClick`
- 移除 VertexLabel 的点击事件，标签变为纯展示
- 缺点：标签不可点击，必须精确点击球体区域

**涉及文件**：
- `src/components/scene/GeometryRenderer.tsx`（DragHandles）
- `src/components/scene/VertexLabels.tsx`（VertexLabel 事件处理）
