# Knowledge 维护系统安装指南

> 将此文档输入给大模型（如 Claude Code），它将为你的项目自动安装一套完整的知识库维护系统。

---

## 系统概述

这是一套**分层、规范化的项目知识库系统**，用于在 AI 辅助开发过程中持续沉淀经过验证的项目知识。

**核心价值**：
- 新对话/新开发者能快速获取项目的架构决策、设计规范、已知陷阱
- 知识质量有准入标准和格式规范，避免垃圾信息堆积
- 通过 skill 命令一键触发知识评估和写入，融入开发工作流

**系统组成**：
1. `.knowledge/` 目录 — 知识库本体（6 个核心文档 + playbooks）
2. `.claude/commands/update-knowledge/` — 知识更新 skill
3. `.claude/commands/checkpoint/` — 检查点 skill（含知识更新）
4. `CLAUDE.md` 中的路由规则 — 告诉 AI 何时查阅哪个文档

## 第二部分：创建 Skills（Claude Code Commands）

### 2.1 update-knowledge skill

创建文件：`.claude/commands/update-knowledge/SKILL.md`

```markdown
---
name: update-knowledge
description: 评估当前任务是否产生了可复用的项目知识（架构模式、设计决策、踩坑记录、实施流程、产品边界），并沉淀到 .knowledge/ 对应文档。在任务审查完成后或用户显式调用时触发。
---

# 知识库更新

评估本次任务产出，将经过验证的新知识沉淀到 `.knowledge/` 知识库。

## 执行流程

### Step 1：回顾变更

回顾本次任务的完整变更范围：代码改动、设计决策、bug 修复、新建模式、产品讨论。

### Step 2：逐维度评估

对照以下 5 个维度判断是否有新知识产生：

| 维度 | 判断标准 | 目标文档 |
|------|---------|---------|
| 架构模式 | 建立了新的代码模式、接口设计、模块组织方式 | `architecture.md` |
| 实施流程 | 确立了可复用的标准操作步骤序列（≥3步） | `playbooks/` 新建或更新 |
| 经验教训 | 发现新陷阱、修复了需要根因分析的 bug | `pitfalls.md` |
| 产品决策 | 做出了"做/不做"的边界决策 | `product.md` |
| 设计规范 | 确立了新的交互模式、UI 样式、视觉编码 | `design.md` |

**判断原则**（详见 `references/quality-rules.md`）：
- 只记录**经过验证的**知识，不记录猜测或临时方案
- 只记录**可复用的**知识，一次性的修改不需要记录
- 只记录**非显而易见的**知识，能从代码直接看出来的不需要记录

### Step 3：读取目标文档

对每个命中的维度，先读取目标文档的当前内容，理解现有结构和内容。

### Step 4：写入新知识

按目标文档的现有格式追加或修改内容。

**格式要求**（详见 `examples/good-vs-bad.md`）：
- `pitfalls.md`：每条必须包含 现象 → 原因 → 修复 → 涉及文件
- `architecture.md`：描述模式的 what + why + 关键接口 + 扩展指导
- `playbooks/`：设计阶段 → 实现阶段 → 验证清单
- `product.md`：决策 + 理由 + 影响范围
- `design.md`：规范 + 适用场景 + 具体值（颜色/尺寸/间距）

### Step 5：验证

- 检查更新后的文档没有重复条目
- 检查新增内容与已有内容不矛盾（如矛盾，更新旧条目而非追加新条目）
- 检查引用的文件路径仍然存在

### Step 6：输出摘要

```
## 知识库更新
- architecture.md: [更新/无变化] — 简述
- playbooks/: [新建 xxx.md / 更新 xxx.md / 无变化] — 简述
- pitfalls.md: [更新/无变化] — 简述
- product.md: [更新/无变化] — 简述
- design.md: [更新/无变化] — 简述
```

## 不做什么

- **不记录代码实现细节**：具体的代码写法能从源码看到，不需要重复记录
- **不记录任务进度**：当前做到哪一步是任务管理的职责
- **不记录 git 历史**：谁改了什么、什么时候改的用 `git log` 查
- **不修改 CLAUDE.md**：流程规则不在此 skill 的职责范围内
- **不创造虚假知识**：如果本次任务没有产生新知识，直接输出"本次无新知识需要沉淀"并结束
```

### 2.2 update-knowledge 质量规则

创建文件：`.claude/commands/update-knowledge/references/quality-rules.md`

```markdown
# 知识条目质量规则

## 准入标准（必须同时满足）

1. **经过验证**：在实际代码中已经实现并验证过，不是设想或计划
2. **可复用**：未来遇到类似场景时能指导决策，不是一次性的特定修复
3. **非显而易见**：不能直接从代码或 git history 推导出来
4. **有因果关系**：不只记录"是什么"，更要记录"为什么"

## 排除标准（命中任一则不记录）

- 能通过 `grep` 或 `git log` 直接获取的信息
- 只在当前任务上下文中有意义的临时决策
- 尚未验证的假设或推测
- 纯粹的代码注释级别的说明（应写在代码里而非知识库）

## 各维度的具体质量要求

### pitfalls.md 条目

**必须包含 4 个字段**：
- **现象**：用户/开发者能观察到的外部表现
- **原因**：根因分析，解释为什么会发生
- **修复**：具体的修复方法
- **涉及文件**：受影响的代码文件

**额外要求**：
- 如果是通用教训（不限于特定模块），标注适用范围
- 如果有前置条件（只在特定配置下触发），标注触发条件

### architecture.md 条目

**必须包含**：
- 模式名称和一句话描述
- 为什么选择这个模式（设计动机）
- 关键接口或 API（类型签名级别，不是完整代码）
- 扩展指导（未来如何基于此模式添加新功能）

**禁止**：
- 复制粘贴完整代码块（指向文件路径即可）
- 记录实现细节（只记录接口契约）

### playbooks/ 条目

**必须包含 3 个阶段**：
- 设计阶段：需要做出的决策清单
- 实现阶段：按序操作步骤，精确到文件路径
- 验证清单：可逐项勾选的检查点

**质量标志**：
- 一个新人（或新对话中的 AI）能否仅凭此 playbook 完成操作？
- 如果不能，缺少的信息是什么？

### product.md 条目

**必须包含**：
- 决策内容（做什么 / 不做什么）
- 决策理由（为什么这样定）
- 影响范围（哪些功能受此约束）

### design.md 条目

**必须包含具体值**：
- 颜色：hex 值
- 尺寸：像素值或对应单位值
- 交互：事件序列描述
- 光标/图标：类型名称

**禁止**：
- 模糊描述如"合适的颜色"、"适当的间距"

## 更新规则

### 追加 vs 修改
- 新发现 → 追加到对应章节末尾
- 与已有条目矛盾 → **修改已有条目**（不是追加一个新的矛盾条目）
- 已有条目的补充说明 → 在已有条目下追加子点

### 去重
- 写入前先搜索目标文档中是否已有类似内容
- 如果已有 ≥80% 重叠的条目，合并而非新增

### 文件路径引用
- 写入的文件路径必须验证当前存在
- 使用相对于项目根目录的路径
```

### 2.3 update-knowledge 正反示例

创建文件：`.claude/commands/update-knowledge/examples/good-vs-bad.md`

```markdown
# 正反示例对照

## pitfalls.md

### 好的条目

```markdown
### 旋转方向计算
- **现象**：鼠标顺时针移动，物体逆时针旋转
- **原因**：`atan2(dx, dy)` 计算的是从 Y+ 轴到鼠标向量的顺时针角度，但物理引擎 angle 为 CCW 正方向
- **修复**：`Math.atan2(-dx, dy)` 而非 `Math.atan2(dx, dy)`
- **文件**：SelectTool.ts
```

有根因分析，有具体的修复方法，有涉及文件。下次遇到类似开发时能直接避坑。

### 差的条目

```markdown
### 旋转 bug
- 旋转方向不对，改了 atan2 的参数就好了
```

缺少根因分析，修复描述模糊，没有文件引用。

---

## architecture.md

### 好的条目

```markdown
### Registry 模式
**原则**：新增类型 = 1 个描述文件 + 1 行 import，删除 = 删 1 个文件。
**实现**：`src/models/registry.ts` 维护 `Map<Type, Descriptor>`。
**扩展指导**：未来的新系统应复用此模式，避免 switch/case 分散耦合。
```

清晰的原则、精确的文件路径、对未来的扩展指导。

### 差的条目

```markdown
### Registry 模式
所有类型都注册在 registry.ts 里，用 Map 存储。
```

只描述了"是什么"（可以从代码看到），缺少"为什么这样设计"和"未来怎么扩展"。

---

## playbooks/

### 好的条目

```markdown
### 2.1 创建描述符文件
在 `src/models/` 创建 `yourType.tsx`。
实现必需接口：
- toConfig(item) — 返回 Config
- render(ctx, item, scale) — Canvas 2D 渲染
```

精确到文件路径和具体操作，新对话中的 AI 能直接执行。

### 差的条目

```markdown
### 2. 实现
创建描述符文件并注册。
```

没有文件路径，没有具体接口要求，无法直接执行。

---

## design.md

### 好的条目

```markdown
- 选中态：蓝色 `#3b82f6`
- 对齐辅助线：蓝色虚线 `#3b82f6`，lineWidth=1，setLineDash([4,4])
```

有具体的颜色值、线宽、虚线参数。

### 差的条目

```markdown
- 选中时显示蓝色高亮
- 对齐时显示虚线
```

"蓝色"是哪个蓝？"虚线"多宽多长？无法直接实现。

## 不应该记录的例子

以下内容**不应写入知识库**：

- 修复拼写错误 → 一次性修改，不可复用，git log 能看到
- 接口字段列表 → 能从代码直接看到，不需要记录
- "考虑用 X 替代 Y" → 未验证的想法，不是已确立的知识
```

### 2.4 checkpoint skill

创建文件：`.claude/commands/checkpoint/SKILL.md`

```markdown
---
name: checkpoint
description: 同步任务文档链、归档用户原始输入、更新知识库。在验收修复完成后或上下文接近上限时手动调用。
---

# Checkpoint

一次性完成文档同步与知识沉淀。

## 执行流程

### Step 1：提取用户原始输入

回顾本轮对话，提取用户的**关键输入**：

| 标签 | 含义 |
|------|------|
| [决策] | 产品/设计/架构的"做/不做"选择 |
| [Bug] | 用户验收时发现的问题 |
| [调整] | 对计划、优先级、方案的修改 |
| [反馈] | 对实现效果的评价和改进要求 |
| [设计] | 交互/视觉/体验的具体偏好 |

保留用户原话（引用格式），附简要上下文。

**不记录以下内容**（即使用户说了也不归档）：
- 确认型提问（"可以验收吗？""这样对吗？"）— 正常沟通，不是决策
- 操作指令（"开始执行""继续""确认"）— 已由流程处理
- 进度询问（"做到哪了？""还剩什么？"）— 临时状态，不需持久化
- 对 AI 工作方式的纠正 — 应记录到 memory/feedback，不是 user-inputs

**提取结果为空是正常的**。如果本轮对话只有执行和确认，Step 1 输出"无关键输入"，Step 2 跳过。

### Step 2：写入原始层

如果 Step 1 结果为"无关键输入"，跳过本步骤。

追加到 `.knowledge/user-inputs.md`，格式：

```
### {日期} · {阶段/任务名}

> 关联任务：[{任务文档名}]({任务文档路径})

- [{标签}] "{用户原话}" — {必要的补充上下文}
```

### Step 3：更新相关文档

更新本轮工作涉及的所有文档，包括但不限于：
- **任务文档链**（如有）：当前任务文档 → 父级阶段文档 → 总计划文档，逐级更新状态和进展
- **其他相关文档**：任务执行过程中创建的中间文档、参考文档、设计文档等

仅更新有变化的部分，不重写整个文档。

### Step 4：更新知识库

对照以下维度，将经过验证的新知识写入对应文档：

| 维度 | 目标文档 |
|------|---------|
| 架构模式 | `architecture.md` |
| 实施流程 | `playbooks/` |
| 经验教训 | `pitfalls.md` |
| 产品决策 | `product.md` |
| 设计规范 | `design.md` |

判断原则：只记录经过验证的、可复用的、非显而易见的知识。

**不写入以下内容**：
- 显而易见的常识
- 对已有模式的简单重复应用
- 仅在当前对话有意义的临时结论

**零写入是合法的**。如果本轮没有产生新知识，每个文档输出"无变化"即可，不要硬凑。

写入时尽量保留用户原话作为决策理由。

### Step 5：输出摘要

```
## Checkpoint 完成
- user-inputs.md: +{N} 条输入归档
- 任务文档：{更新了哪些文档}
- 知识库：
  - architecture.md: [更新/无变化]
  - pitfalls.md: [更新/无变化]
  - product.md: [更新/无变化]
  - design.md: [更新/无变化]
  - playbooks/: [更新/无变化]
```
```

## 第三部分：配置 CLAUDE.md 路由规则

在项目的 `CLAUDE.md` 中添加以下配置块（与你的现有规则合并即可）：

```markdown
## 知识库（.knowledge/）

项目知识库位于 `.knowledge/`，包含经过验证的规范和经验教训。

- 开始任务前读取 `.knowledge/INDEX.md` 确定需查阅的文档
- 涉及产品决策时查阅 `product.md`
- 涉及交互设计时查阅 `design.md`
- 涉及代码实现时查阅 `architecture.md`
- 执行标准操作时查阅 `playbooks/`
- 编写或审查代码时查阅 `pitfalls.md`
```

**可选**：如果你有任务结束/审查流程，在对应阶段加入提示：

```markdown
- 任务结束时，建议执行 `/checkpoint` 完成知识沉淀
- 或手动执行 `/update-knowledge` 仅做知识更新
```

## 第四部分：使用方式

### 日常使用

1. **开始任务前** — AI 自动读取 `.knowledge/INDEX.md`，按任务类型查阅对应文档
2. **执行过程中** — AI 参考 `pitfalls.md` 避免已知陷阱，参考 `architecture.md` 遵循既有模式
3. **任务完成后** — 用户执行 `/checkpoint` 或 `/update-knowledge`，AI 评估并沉淀新知识

### 触发命令

| 命令 | 用途 | 时机 |
|------|------|------|
| `/update-knowledge` | 仅评估+写入知识库 | 任务审查完成后 |
| `/checkpoint` | 归档用户输入 + 更新文档链 + 更新知识库 | 验收完成后 / 上下文接近上限时 |

### 知识库的生长过程

初始状态下各文档几乎为空。随着项目推进：
- 每次 `/checkpoint` 后，`pitfalls.md` 积累踩坑记录
- 每次做出产品决策后，`product.md` 记录边界
- 每次建立新模式后，`architecture.md` 记录设计
- 每次确立标准流程后，`playbooks/` 新增手册
