# 物理引擎误差与机械能损耗问题记录

更新时间：2026-04-18

## 背景
- 用户在统一测试阶段反馈：
  - 当静摩擦力与重力沿斜面分力理论上恰好平衡时，滑块起始阶段仍会出现很小的速度毛刺。
  - 绳约束圆周、弹簧振子等场景会出现机械能下降，用户希望区分“真实耗散”与“数值误差”。

## 当前结论

### 1. 起步毛刺不是单纯图表假象
- 当前毛刺主要来自“初始接触建立”与“接触求解近似”。
- 其中最直接的代码原因是：
  - `src/core/snap/SnapEngine.ts` 中存在真实物理层的 `VISUAL_GAP = 0.008`
  - 非球体吸附到斜面/支撑面时，会把这个 `0.008m` 直接写入物体真实位置
  - 结果是物体进入仿真时并非“刚好贴住”，而是先有一个极小缝隙，再由求解器建立接触

### 2. 静摩擦平衡目前不是“解析锁定”，而是“求解器近似平衡”
- 当前场景层只有单一 `friction`，没有独立 `μs / μk`。
- 教学层虽然会显示：
  - `G∥`
  - `G⊥`
  - `f静`
  - `f滑`
- 但这些属于教学派生显示，不是物理层硬约束。
- 因此“理论上应静止”的场景，在数值上仍可能出现极小 creep / settle。

### 3. 机械能不守恒的主体来源是数值误差
- 当前项目使用 Planck.js 迭代求解：
  - 接触
  - rope / rod / spring / pulley 约束
- 这类系统天然会带来数值耗散。
- 项目代码已经通过“子步进 + 迭代数”在部分场景做补偿：
  - rope / spring / rod 场景已有更高精度档
- 之前“提高 spring 场景精度后损耗下降”的现象，说明主要问题来自数值误差，而不是隐藏的真实阻尼。

## 已定位到的关键代码点

### 1. 吸附阶段的真实物理间隙
- 文件：`src/core/snap/SnapEngine.ts`
- 关键信号：
  - `VISUAL_GAP = 0.008`
  - 非球体斜面吸附时 `offsetDist += 0.008`
  - `snapPosX / snapPosY` 直接使用带 gap 的结果

### 2. 静摩擦教学层与动力学层脱耦
- 文件：`src/engine/teachingForces.ts`
- 当前 `static_friction / kinetic_friction / gravity_parallel / gravity_perpendicular` 是教学派生力，不是物理引擎约束。

### 3. 约束场景的数值误差来源
- 文件：`src/engine/PhysicsBridge.ts`
- 关键信号：
  - `runSubsteps(...)`
  - `getRecommendedSubsteps()`
  - `getRecommendedSolverIterations()`
  - `world.step(stepDt, iterations.velocity, iterations.position)`

### 4. 分析层只是把误差记录出来
- 文件：`src/engine/AnalysisRecorder.ts`
- 当前 `eMech` 是按实时状态直接计算，不是凭空造出损耗。

## 后续优化建议（暂不在本轮执行）

### P0：去掉物理层 snap gap
- 目标：
  - 将 `0.008m` 从“真实物理位姿”移出
  - 只保留到渲染补偿层
- 预期收益：
  - 显著减少“理论静止却开头先抖一下”的毛刺

### P1：增加静摩擦稳定器
- 目标：
  - 在斜面 / 倾斜传送带等教学场景下
  - 若切向速度极小且理论上应静止，则主动钳制切向速度为 0
- 预期收益：
  - 降低“刚好平衡时仍偷偷滑动”的现象

### P1：扩大高精度求解档覆盖范围
- 目标：
  - 不只 rope / spring
  - 对斜面接触、倾斜传送带、木板-物块等 contact-heavy 场景也提高子步进和迭代次数

### P2：拆分 `μs / μk`
- 目标：
  - 将静摩擦系数与滑动摩擦系数分离
- 预期收益：
  - 让教学语义与引擎语义更一致

## 当前处理策略
- 本问题已单独归档。
- 在统一测试阶段继续先按用户原始需求逐条验收。
- 后续若进入“物理引擎精度专项优化”，优先从：
  1. 去掉物理层 snap gap
  2. 静摩擦稳定器
  3. contact-heavy 场景高精度档
  这三个方向开始。
