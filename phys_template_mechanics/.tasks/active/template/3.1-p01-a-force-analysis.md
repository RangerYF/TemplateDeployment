## 验收问题根因修复（2026-04-01）
- 用户反馈问题：
  1. 部分支撑体（斜面/球槽等）与地面重叠，落到地面以下。
  2. 模板打开后的受力显示与开始仿真后不一致。
  3. 杆长（及可能其他约束长度）开仿真后突变。
  4. 竖直圆周模板初速度不足，无法完成圆周运动。

### 根因分析（系统性）
1. **位姿根因**：模板中存在手工估值位置，未统一走 `snapTo` 同源吸附，导致支撑体未贴地。
2. **受力根因**：编辑态 `probeForces` 默认探测 5 步，采样时刻与仿真首帧不一致。
3. **约束根因**：执行器推导 `rope/rod/spring` 默认长度时使用了“物体中心距”，而非“锚点距”。
4. **动力学根因**：竖直圆周模板初速度未按最小闭环条件（`rope: sqrt(5gR)` / `rod: sqrt(4gR)`）设计。

### 根因级修复
- `src/templates/commands/executor.ts`
  - 约束默认长度改为基于**世界锚点距离**推导（并同步修正 pulley 默认参数推导）。
- `src/engine/PhysicsBridge.ts`
  - `probeForces` 默认步数由 `5 -> 1`，并在探测前重置 EMA，保证与仿真首帧一致。
- `src/templates/commands/examples.ts`
  - 为斜面/球槽/传送带/V 槽等支撑体补齐 `snapTo -> ground` 放置。
  - 清理多处手工 `maxLength/length`，改为执行器按锚点自动推导。
  - 竖直圆周模板引入速度计算函数，按最小速度条件加安全系数配置初速度。
- `src/templates/commands/validator.ts` + 生成/加载链路
  - 新增场景健全性校验（地面穿透、约束几何不一致、竖直圆周速度不足等）。
  - 生成脚本与 loader 接入该校验，防止后续模板重复引入同类问题。

- 用户确认意图：`执行吧`

### 低侵入验证结果
1. **仅提高 solver iterations：无效**
   - 该结论延续前置研究，`8/3 -> 16/6 -> 32/12` 几乎无差异。
2. **更小子步进：有效**
   - 临时验证脚本表明，`dt/N × N substeps` 明显优于单步 `dt`。
   - `2 substeps` 改善有限，且 rope 样本 slack 比例并不稳定，因此未采用。
   - 继续比较后确认：
     - `rope` 场景用 `8 substeps` 优于 `4/6 substeps`
     - `rod` 场景 `6/8 substeps` 虽继续改善，但收益相对 `4 substeps` 较小，因此本轮保留 `4 substeps` 作为更稳妥的性能折中。
3. **强制投影到理想圆周：未采用**
   - 虽然几乎消除 joint error，但会反向加能、抹掉 rope 应有的 slack，破坏物理语义。

### 本轮正式方案
1. `src/engine/PhysicsBridge.ts`
   - `step(dt)` 改为：
     - 若场景中存在 `rope`，则自动使用 `8` 个子步进；
     - 若场景中存在 `rod(sceneType='rod')` 且无 rope，则自动使用 `4` 个子步进；
     - 否则保持单步。
   - `Canvas` 与 `probeForces` 改为走 `stepFrame(scene, dt)`，保证子步进下连续外力在每个子步前都重复施加。
   - 新增 `lastJointReactionInvDt` 与 `getJointReactionInvDt()`，让约束反力读取与真实内部步长一致。
2. `src/engine/ForceCollector.ts`
   - `collectJointForces()` 改用 `bridge.getJointReactionInvDt()` 读取约束反力，避免子步进后约束力被错误缩小。
