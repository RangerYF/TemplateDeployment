## 重要约束
- 每次回复用户前，必须以 `Sir` 开头（任务名可取自任务文档文件名简述）。
- 每次回复必须在开头声明当前模式：`[MODE: 模式名称]`。
- Git 提交必须使用作者 `cjn <1229412289@qq.com>`，否则 Vercel 自动构建会失败。如当前 git config 不匹配，需用 `--author="cjn <1229412289@qq.com>"` 指定。
- **禁止使用 `--no-verify`**：Git 提交时严禁使用 `--no-verify` 跳过 pre-commit hook。如果 hook 检查失败，必须修复问题后再提交。
- **禁止自动更新 `handoff.md`**：`handoff.md` 仅在用户显式调用 `/handoff.build` 命令时才允许写入/覆盖。对话过程中不得以任何理由自动修改该文件。

## 第一层：核心规则

### 核心规则A：规则优先级
- 触发条件：出现规则冲突或路径选择不确定。
- 动作：路径裁决按 `用户显式指令 > 风险分级自动流转 > 当前模式默认路径` 执行；安全与验证门禁仍受核心规则约束。
- 输出：明确说明当前采用的规则来源。
- 验收：不存在同轮反复索要模式口令的机械往返。

### 核心规则B：文档优先与任务归并
- 触发条件：开始新工作，或用户提出追加需求/bug反馈。
- 动作：先读取当前任务文档；同一任务续作默认并入当前任务，除非用户明确要求新任务。
- 输出：在任务文档记录关键用户原文与决策。
- 验收：无无依据新建任务；关键决策可追溯。

### 核心规则C：按计划执行与偏离处理
- 触发条件：进入执行模式后发现计划外改动需求。
- 动作：停止执行并返回计划模式补齐计划，不允许直接偏离实现。
- 输出：记录偏离原因与补充计划条目。
- 验收：执行内容与计划逐项可映射。

### 核心规则D：修改后检查门禁
- 触发条件：对项目文件产生修改。
- 动作：按受影响目录命中"附录F·最小回归门禁矩阵"中配置的检查命令执行回归检查；若命中数据链路，追加受影响模块的 dry-run 验证。
- 输出：在回复中给出检查命令与结果。
- 验收：本轮改动均有对应验证证据。

### 核心规则E：数据任务校验
- 触发条件：涉及数据读取、转换、映射、ID处理。
- 动作：先验证真实数据样本的字段名、类型、嵌套层级；禁止字段名推测。
- 输出：在任务文档记录数据结构样例与关键字段映射。
- 验收：出现数据结构不一致时立即停止执行并回到计划模式。

### 核心规则F：数据库安全
- 触发条件：涉及数据库清理、重建或破坏性语句。
- 动作：先备份，再二次确认清除范围；优先非破坏性方案。
- 输出：在任务文档记录备份位置与确认记录。
- 验收：无未备份的 `DROP/TRUNCATE` 等破坏性操作。

### 核心规则G：防重复确认
- 触发条件：用户已表达执行确认意图（如“确认执行/按计划修复/开始改/继续/执行”）。
- 动作：直接进入执行，不重复要求“进入执行模式”口令。
- 输出：简要确认将按既定计划执行。
- 验收：同一阶段不出现重复确认循环。

### 核心规则H：风险分级驱动流程
- 触发条件：MODE 0 任务评估完成后。
- 动作：必须先判定风险等级（L0-L3），再按等级映射的最小流程路径流转。
- 输出：在任务文档记录“风险等级 + 流程路径 + 强制门禁”。
- 验收：所有执行任务均可追溯到风险等级与对应路由。

### 核心规则I：交付DoD模板化
- 触发条件：进入 MODE 6，或需要对本轮改动给出交付结论。
- 动作：按统一DoD模板输出“变更摘要、验证结果、风险与回滚、任务完成质量模块”。
- 输出：给出可追溯命令证据与质量结论；L2-L3补充高风险检查结论。
- 验收：DoD字段完整，且与执行证据一致。

## 第二层：流程规则（RIPER-7）

### MODE 0：任务确认模式
- 用途：任务生命周期入口（创建、切换、暂停、恢复）。
- 必须行动：
  1. 生成任务ID（`月-日-时-分-任务关键词`）。
  2. 在 `.tasks/active/` 创建任务文档（复杂任务完整文档，简单任务轻量记录）。
  3. 读取 `.tasks/task_evaluation_framework.md` 完成"任务类型 + 风险等级（L0-L3）"评估。
  4. 简单任务先输出"任务理解+执行计划"，待用户确认后执行。
  5. 按风险等级路由到后续模式（L0可直达执行，L1-L3按矩阵执行）。
  6. 续作优先补充到现有任务文档。

### MODE 1：研究模式
- 用途：仅信息收集。
- 允许：读取文件、澄清问题、理解结构。
- 禁止：提出实现建议、写代码、做最终规划。
- 数据强制：查看真实样本、验证完整结构、记录字段和嵌套关系。

### MODE 2：创新模式
- 用途：头脑风暴可选方案。
- 允许：讨论想法与优缺点。
- 禁止：定案、具体实现、代码编写。

### MODE 3：计划模式
- 用途：输出可执行技术计划。
- 要求：编号清单；精确到文件路径与改动点。
- 数据前置：必须先分析真实数据结构，禁止基于命名假设。

### MODE 4：执行模式
- 用途：严格实现MODE 3计划。
- 进入条件：用户明确要求执行，或出现执行确认意图，或风险路由命中执行阶段。
- 禁止：计划外创造性添加。
- 数据前置：执行前再次核对数据字段名/类型/嵌套层级。

### MODE 5：审查模式
- 用途：验证实现是否与计划一致。
- 输出格式：
  - 偏离：`:warning: DEVIATION DETECTED: [描述]`
  - 结论：`:white_check_mark: 实现完全匹配计划` 或 `:cross_mark: 实现偏离计划`

### MODE 6：任务结束模式
- 用途：任务收尾与归档。
- 必须行动：
  1. 仅在对应项目文件改动时执行格式化。
  2. 仅用户明确要求时执行Git提交。
  3. 检查临时文件清理。
  4. 提供下一步建议。
  5. 按DoD模板输出交付结论（变更摘要、验证结果、风险回滚、质量模块）。
  6. 如上下文接近上限或验收修复完成，建议执行 `/checkpoint`。

## 风险分级与最小流程路径（L0-L3）

| 等级        | 判定条件（命中任一即升级）                                | 最小流程路径                                             | 必须产物                         | 强制门禁                                           |
| ----------- | --------------------------------------------------------- | -------------------------------------------------------- | -------------------------------- | -------------------------------------------------- |
| L0 低风险   | 单文件微调；不改业务逻辑；不涉数据/数据库/部署            | MODE 0 -> MODE 4 -> MODE 6                               | 轻量任务记录、变更摘要           | 改代码时执行命中目录最小回归；纯文档可记录跳过理由 |
| L1 常规风险 | 1-3文件改动；常规功能/配置修复；无高风险数据链路          | MODE 0 -> MODE 3 -> MODE 4 -> MODE 5 -> MODE 6           | 执行计划、验证证据、审查结论     | 命中目录最小回归 + DoD模板化交付                   |
| L2 高风险   | 涉及数据读取/转换/映射/ID；跨模块联动；外部接口变更       | MODE 0 -> MODE 1 -> MODE 3 -> MODE 4 -> MODE 5 -> MODE 6 | 数据结构样本、字段映射、风险说明 | L1全部 + 数据契约校验 + 受影响模块dry-run验证      |
| L3 关键风险 | 涉及数据库破坏性操作、生产路径/环境切换、系统级高影响改动 | MODE 0 -> MODE 1 -> MODE 3 -> MODE 4 -> MODE 5 -> MODE 6 | 分阶段计划、回退方案、确认记录   | L2全部 + 数据库安全强制 + 交付自检强制             |

## 自动流转规则（风险路由版）
1. 新任务且目标明确：完成 MODE 0 后，按风险等级自动路由到下一模式。
2. L0 默认可直接执行；L1 默认进入计划；L2-L3 默认先研究再计划。
3. 研究完成后自动进入计划；计划确认后进入执行；执行完成后进入审查。
4. 续作/bug反馈默认并入当前任务，先给“问题理解 + 处理计划 + 执行确认请求”。
5. 用户出现执行确认意图（`确认执行/按计划修复/开始改/继续/执行`）时，直接进入执行，不重复索要口令。
6. 用户显式模式指令始终优先；若与自动流转冲突，以显式指令为准并记录原因。
7. 若显式指令试图绕过高风险门禁（数据校验/数据库安全），必须保留门禁并说明依据。
8. 禁止机械化往返：同一阶段已确认后，不回退索要模式切换口令。

## 第三层：模板附录

### A. 任务文档目录结构
```text
.tasks/
├── active/
├── completed/
├── suspended/
└── template.md
```

### C. 模式转换信号
- 显式模式信号：
  - `进入任务确认模式` / `新任务` / `开始任务`
  - `进入研究模式`
  - `进入创新模式`
  - `进入计划模式`
  - `进入执行模式`
  - `进入审查模式`
  - `进入任务结束模式` / `完成任务`
  - `简单任务直接执行`
- 执行确认意图（等价进入执行）：
  - `确认执行` / `按计划修复` / `开始改` / `继续` / `执行`
- 续作并入触发信号（默认并入当前任务）：
  - `顺手修一下` / `这个bug也一起处理` / `继续上个任务` / `再修复一个问题`

### D. 通用交付自检（路径/环境一致性）
1. 路径对齐：源码、部署、运行目录、数据目录、输出目录一致。
2. 环境一致：确认实际读取的 `.env` 与关键变量值。
3. 运行方式一致：启动命令与入口/模块路径匹配。
4. 产物可读：上线前只读验证目标文件存在且可读。
5. 先验证再自动化：先手动跑通，再启用自动流程。

### E. 数据库操作安全要求
- 严禁未备份执行 `DROP DATABASE`、`DROP TABLE`、`TRUNCATE`。
- 清空或重建数据库前，必须二次确认清除范围并取得明确同意。
- 优先非破坏性方案（如安全更新、迁移脚本）。

### F. 最小回归门禁矩阵（阶段4）

- `src/` 变更：执行 `pnpm lint && pnpm tsc --noEmit`（工程初始化后生效）
- `src/engine/` 变更且命中数学计算：追加对应几何体的手算验证
- 多目录命中：门禁命令叠加执行

### G. 统一交付DoD模板（阶段4）
```markdown
## 交付DoD
- 变更摘要
- 验证结果（命令 + 结果 + 跳过原因）
- 风险与回滚
- 任务完成质量模块（bug/文档/规则/根因/改进）
- 审查结论（匹配/偏离计划）
```

### H. 高风险任务强制检查清单（阶段4）
- L2-L3 数据任务：数据契约校验（字段名/类型/嵌套）强制。
- L2-L3 且命中数据链路：执行附录F中配置的质量检查与受影响模块 dry-run 验证。
- L3 数据库任务：备份、范围二次确认、回滚步骤强制。
- 所有跳过项必须记录"跳过原因 + 证据"。
