# 知识库更新

目标不是“每次都写知识库”，而是“只在值得时写”。

## Step 0：Preflight

先用最小成本判断本轮是否值得进入完整知识评估。

### 必查信号

只要没有命中任何稳定知识信号，就直接结束：
- **架构模式**：出现新的模块边界、接口契约、状态流、职责拆分
- **实施流程**：形成可重复执行的标准流程（至少 3 步）
- **经验教训**：发现有通用价值的根因、陷阱、修复规则
- **产品决策**：出现明确的做/不做边界
- **设计规范**：形成可复用的交互、视觉、布局规范

### 直接跳过条件

满足以下任一情况，可直接输出“本次无新知识需要沉淀”：
- 本轮只有代码实现或文档同步，没有新的稳定结论
- 本轮只有 `user-inputs` 归档，没有抽象出通用规则
- 本轮只修改 `.tasks/`、任务进度、状态描述
- 本轮只是重复应用已有模式，没有新增约束或例外
- 本轮结论尚未验证，仍处于猜测或临时方案阶段

## Step 1：锁定命中的维度

不要默认评估全部 5 个维度。先根据本轮变化，锁定真正命中的维度。

示例：
- 只有“新的 UI 约束” => 只看 `design.md`
- 只有“新根因和修复规则” => 只看 `pitfalls.md`
- 只有“新场景生成流程” => 只看 `playbooks/`

## Step 2：按需读取目标文档

只读取命中维度对应的目标文档，不要全量打开整个知识库。

目标映射：
- 架构模式 -> `architecture.md`
- 实施流程 -> `playbooks/`
- 经验教训 -> `pitfalls.md`
- 产品决策 -> `product.md`
- 设计规范 -> `design.md`

## Step 3：写入新知识

按目标文档的现有格式追加或修改内容。

### 判断原则
- 只记录**经过验证的**
- 只记录**可复用的**
- 只记录**非显而易见的**

### 格式要求
- `pitfalls.md`：现象 -> 原因 -> 修复 -> 涉及文件
- `architecture.md`：what + why + 关键接口 + 扩展指导
- `playbooks/`：设计阶段 -> 实现阶段 -> 验证清单
- `product.md`：决策 + 理由 + 影响范围
- `design.md`：规范 + 适用场景 + 具体值

## Step 4：验证

- 检查新增内容没有重复
- 检查是否与既有条目冲突；若冲突，优先更新旧条目
- 检查引用文件路径仍存在

## Step 5：输出摘要

```
## 知识库更新
- architecture.md: [更新/无变化] — 简述
- playbooks/: [新建 xxx.md / 更新 xxx.md / 无变化] — 简述
- pitfalls.md: [更新/无变化] — 简述
- product.md: [更新/无变化] — 简述
- design.md: [更新/无变化] — 简述
```

## 不做什么

- 不记录代码实现细节
- 不记录任务进度
- 不记录 git 历史
- 不修改 AGENTS.md
- 不因为“本轮调用了 update-knowledge”就强行写内容
