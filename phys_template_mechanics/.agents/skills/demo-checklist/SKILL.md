# 演示清单技能

用于生成或重写任意版本的演示清单、体验清单、交付说明。

这个技能是**跨版本**的。它不是为 `v0.1` 定制模板，而是指导如何根据当前版本类型，整理出适合该版本的用户文档。

## 适用场景

当用户出现以下意图时使用本技能：
- 生成某个版本的演示清单
- 重写已有演示文档
- 把研发过程整理成面向用户的版本说明
- 把“本版本做了什么”整理成对外可读文档
- 需要区分“底座能力、版本增量、场景案例、边界说明”

默认输出到 `docs/演示/演示清单-vX.Y.md`。若用户指定其他文件名或目录，以用户要求为准。

## 核心原则

### 1. 文档必须跟着版本类型走

不要假设每个版本都适合相同结构。先判断版本属于哪一类，再决定写法。

常见版本类型：

- **首版/里程碑版**
  - 例如：第一次把编辑器底座交付出来
  - 写法重点：整体能力、平台价值、代表性场景、版本边界

- **增量迭代版**
  - 例如：`v0.2` 只新增几个功能、修一些体验问题
  - 写法重点：相对上个版本新增了什么、优化了什么、体验有什么变化

- **专题增强版**
  - 例如：集中增强分析系统、模板体系、工作台体验
  - 写法重点：围绕一个主题展开，减少全局重复介绍

- **收尾/优化版**
  - 例如：主要是交互打磨、视觉统一、性能和稳定性收敛
  - 写法重点：用户可感知改善，而不是内部技术清单

### 2. 文档始终面向用户

整篇文档必须站在用户视角，而不是研发视角。

应写：
- 这个版本用户可以做什么
- 和上个版本相比体验变好了什么
- 哪些教学场景现在可以用了
- 哪些边界暂时还不在本版

不要写：
- 内部阶段编号
- 执行依据
- 研发闭环
- 运行态、catalog、scene JSON
- “建议怎么讲”“建议对外口径”“这一部分是主线”

### 3. 事实来自真实内容，不来自旧文档复述

先确认事实，再写文档。

优先数据来源：
1. `docs/需求文档md/`
2. `.tasks/active/` 下与当前版本相关的任务文档
3. 项目中的真实配置、目录、注册表、运行态数据
4. 旧演示文档，仅作写法参考，不作事实真源

若旧文档、计划文档、当前真实内容冲突，以当前真实内容为准。

## 工作流

### 第一步：判断当前版本类型

在写文档前，先回答这几个问题：

1. 这个版本是“首次交付底座”，还是“在上一版上继续迭代”？
2. 当前重点是平台能力、某个专题能力，还是若干零散增量？
3. 用户最应该先看到的是：
   - 整体能力
   - 新增功能
   - 体验优化
   - 场景覆盖
4. 哪些内容是本版新增，哪些只是上一版已有能力的延续？

这一步决定整篇文档结构。

### 第二步：确认真实交付范围

至少确认这些信息：
- 当前版本覆盖了哪些模块/功能
- 当前版本新增了哪些能力
- 当前版本优化了哪些体验
- 当前版本有哪些代表案例
- 当前版本的边界是什么

若项目有“模块/模板/场景”这类内容，区分：
- 已上线
- 本版新增
- 沿用上版
- 暂未纳入

### 第三步：为当前版本选择合适结构

不要固定套用同一套标题。根据版本类型选结构。

#### A. 首版/里程碑版推荐结构

适用于第一次交付编辑器、平台或较大体系能力：

```markdown
# Vx.y 演示清单

## 一、版本定位
## 二、核心能力
## 三、代表性场景
## 四、推荐体验顺序
## 五、版本价值
## 六、当前版本边界
## 七、附录（可选）
```

#### B. 增量迭代版推荐结构

适用于 `v0.2`、`v0.3` 这种在上一版基础上继续补功能或优化体验的版本：

```markdown
# Vx.y 演示清单

## 一、本版概览
## 二、相对上一版的新增内容
## 三、重点优化项
## 四、推荐体验顺序
## 五、当前可体验内容
## 六、版本边界
## 七、附录（可选）
```

#### C. 专题增强版推荐结构

适用于围绕某一主题集中增强的版本：

```markdown
# Vx.y 演示清单

## 一、版本主题
## 二、本版新增能力
## 三、代表性使用场景
## 四、推荐体验顺序
## 五、版本价值
## 六、边界与后续空间
## 七、附录（可选）
```

### 第四步：决定“能力”和“案例”的比例

这是最重要的判断之一。

- 如果当前版本主要价值在**底座能力**，就让“能力”做主文档主角，案例只做证明
- 如果当前版本主要价值在**新功能/新场景**，就让“新增内容”做主角
- 如果当前版本主要价值在**体验优化**，就突出“用户感知变化”，不要写成 bug 修复流水账

默认规则：
- 主文档优先写能力、增量、体验变化
- 案例只保留少量代表项
- 大量模板/场景/模块清单放附录

### 第五步：可选加入“主动设计补充”

如果用户明确希望体现研发中的主动思考和主动设计，可以加一个附录，名称可用：
- `附录：主动设计补充`
- `附录：体验与设计增强`
- `附录：本版补充优化`

这个附录是**可选项**，不是每个版本都必须有。

适合放进去的内容：
- 用户能直接感知到的设计优化
- 不依赖单一场景、作用于底座或整体体验的增强
- 本来不在最初需求里，但通过主动思考补上的体验设计

不适合放进去的内容：
- 纯内部实现细节
- 文件路径、类名、算法名
- 研发排障过程

## 写作约束

### 视角约束

整篇文档必须使用面向用户的客观陈述。

应写：
- `当前版本包含以下内容`
- `本版新增能力如下`
- `当前版本边界如下`

不要写：
- `这一部分是主线`
- `建议优先讲`
- `建议话术`
- `建议对外口径`
- `用于回答当前到底做了多少场景`

### 措辞约束

把这些表达改成用户语言：

| 不推荐 | 推荐 |
|---|---|
| ready 模板 | 已上线内容 / 可体验内容 |
| 研发闭环 | 可继续编辑 / 可继续使用 / 可继续扩展 |
| 运行态一致 | 与当前实际内容一致 |
| 推荐演示顺序 | 推荐体验顺序 |

### 内容约束

- 主文档不写内部阶段编号
- 主文档不写任务文件路径
- 主文档不写“为什么研发时这么做”
- 可以写版本边界，但不要写内部排障过程

## 何时写“编辑器核心能力”

如果当前版本的重点是平台、底座、编辑器、工作台、分析系统，或者用户明确希望强调“自己主动设计出来的能力”，应写“核心能力”章节。

如果当前版本只是小规模增量，如：
- 增加 2 个小功能
- 修若干体验问题
- 做一个专题增强

那就不必强行写“编辑器核心能力”，可以改成：
- `本版新增能力`
- `本版重点优化`
- `本版变化摘要`

## 何时写“场景覆盖清单”

仅在满足以下任一条件时加入附录清单：
- 当前版本场景数量较多，需要交代覆盖面
- 用户明确要求列出模块或模板
- 主文档为了避免重复，故意把清单下沉

若当前版本场景数量很少，或重点根本不在场景，不必强行列附录。

## 体验顺序写法

如果写“推荐体验顺序”，只写步骤，不写讲解话术。

推荐写法：
- `步骤 A：进入与切换`
- `步骤 B：编辑与分析`
- `步骤 C：对比体验`

每步只写用户操作：
- 打开
- 切换
- 点击
- 调整
- 播放
- 回看
- 对比

不要写：
- `先讲`
- `说明`
- `强调`
- `建议演示`

## 校验

完成后至少做以下检查：

1. 格式检查
```bash
git diff --check -- <目标文档路径>
```

2. 作者口吻与研发口吻检查
```bash
rg -n '这一部分|建议|不建议|主线|对产品经理和用户|执行依据|研发闭环|ready|运行态|scene JSON|catalog|模板研发|建议话术|推荐演示顺序' <目标文档路径>
```

3. 结构检查
```bash
rg -n '^## |^### ' <目标文档路径>
```

## 输出要求

- 输出中文
- 先判断版本类型，再决定结构
- 文档优先体现用户能感知到的价值
- 如果是迭代版，不要把首版结构原样硬套上去
- 如果用户没有要求详细附录，主文档尽量简洁
