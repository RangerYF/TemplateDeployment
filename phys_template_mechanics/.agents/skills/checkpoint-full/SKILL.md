# Checkpoint Full

用于阶段级或会话级完整收尾。

## 适用场景
- 阶段完成
- 验收闭环
- 上下文接近上限
- 明确要做知识沉淀

## 执行流程

### Step 1：提取关键用户输入

提取：
- `[决策]`
- `[Bug]`
- `[调整]`
- `[反馈]`
- `[设计]`

### Step 2：写入 `.knowledge/user-inputs.md`

若有关键输入则写入；否则跳过。

### Step 3：同步任务文档链

按需更新：
- 当前子任务文档
- 父级阶段文档
- 总计划文档

仅更新有变化的部分。

### Step 3.1：补后续承接信息

即使走 `full`，也先确保后续任务可接续：
- 新约束
- 依赖变更
- 必看文件
- 可复用结论
- 未决风险

### Step 4：Preflight 判断是否调用 `update-knowledge`

只有在本轮形成稳定知识信号时才调用。若只是完整收尾但没有稳定知识，可跳过。

稳定知识信号包括：
- 新架构模式或职责边界
- 新设计规范
- 新产品边界决策
- 通用踩坑/根因修复
- 可复用流程

### Step 5：输出摘要

```
## Checkpoint 完成
- mode: full
- user-inputs.md: +{N} 条输入归档
- 任务文档：{更新了哪些文档}
- update-knowledge: [已调用/跳过] — {原因或结果摘要}
- 知识库变更：{引用 update-knowledge 输出或写“无”}
```
