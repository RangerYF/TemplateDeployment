# 知识条目质量规则

## 准入标准（必须同时满足）

1. **经过验证**：在实际代码中已经实现并验证过，不是设想或计划
2. **可复用**：未来遇到类似场景时能指导决策，不是一次性的特定修复
3. **非显而易见**：不能直接从代码或 git history 推导出来
4. **有因果关系**：不只记录"是什么"，更要记录"为什么"

## 排除标准（命中任一则不记录）

- 能通过 `grep` 或 `git log` 直接获取的信息
- 只在当前任务上下文中有意义的临时决策
- 尚未验证的假设或推测
- 纯粹的代码注释级别的说明（应写在代码里而非知识库）

## 各维度的具体质量要求

### pitfalls.md 条目

**必须包含 4 个字段**：
- **现象**：用户/开发者能观察到的外部表现
- **原因**：根因分析，解释为什么会发生
- **修复**：具体的修复方法
- **涉及文件**：受影响的代码文件

**额外要求**：
- 如果是通用教训（不限于特定物体/模块），标注适用范围
- 如果有前置条件（只在特定配置下触发），标注触发条件

### architecture.md 条目

**必须包含**：
- 模式名称和一句话描述
- 为什么选择这个模式（设计动机）
- 关键接口或 API（类型签名级别，不是完整代码）
- 扩展指导（未来如何基于此模式添加新功能）

**禁止**：
- 复制粘贴完整代码块（指向文件路径即可）
- 记录实现细节（只记录接口契约）

### playbooks/ 条目

**必须包含 3 个阶段**：
- 设计阶段：需要做出的决策清单
- 实现阶段：按序操作步骤，精确到文件路径
- 验证清单：可逐项勾选的检查点

**质量标志**：
- 一个新人（或新对话中的 Claude）能否仅凭此 playbook 完成操作？
- 如果不能，缺少的信息是什么？

### product.md 条目

**必须包含**：
- 决策内容（做什么 / 不做什么）
- 决策理由（为什么这样定）
- 影响范围（哪些功能受此约束）

### design.md 条目

**必须包含具体值**：
- 颜色：hex 值
- 尺寸：像素值或米制值
- 交互：事件序列描述
- 光标/图标：类型名称

**禁止**：
- 模糊描述如"合适的颜色"、"适当的间距"

## 更新规则

### 追加 vs 修改
- 新发现 → 追加到对应章节末尾
- 与已有条目矛盾 → **修改已有条目**（不是追加一个新的矛盾条目）
- 已有条目的补充说明 → 在已有条目下追加子点

### 去重
- 写入前先搜索目标文档中是否已有类似内容
- 如果已有 ≥80% 重叠的条目，合并而非新增

### 文件路径引用
- 写入的文件路径必须验证当前存在
- 使用相对于项目根目录的路径（如 `src/core/tools/SelectTool.ts`）
