可视化模板平台架构设计稿

本设计稿结合当前模板仓库与主项目现状，给出面向“模板持续新增/升级、实例后端化存储、实例 CRUD、继续编辑、版本兼容”的整体平台架构方案。 目标不是单纯替换 iframe，而是建立一套长期可扩展的模板平台。

适用范围：模板中心 / 模板实例 / 工作台 / 模板升级 设计重点：后端注册中心 + 版本化运行时 + 实例快照 迁移原则：先数据后协议，再逐步替换嵌入方式

1. 背景与现状

当前系统由两个部分组成：模板仓库与主项目可视化中心。模板已经有多个独立 Vite/React 应用，主项目通过模板目录页、实例页和工作台页接入这些模板。

模板仓库现状
  • 多个模板以独立应用形式存在，主要是 Vite + React + TypeScript。
  • 模板大多可独立部署在 Vercel 上，部分物理模板共享同一运行时入口。
  • 模板之间技术栈相近，但版本并未完全统一，存在 React 18/19、Vite 6/7/8 并存情况。
  • 模板本身更多是“独立 SPA”而不是天然可嵌入组件。
主项目现状
  • 模板目录由前端常量 VISUAL_TEMPLATE_MANIFEST 维护。
  • 模板实例由前端 localStorage 维护，当前只保存标题、模板编号、更新时间等元信息。
  • 工作台进入后通过生成前端 projectId 并用 iframe 打开模板入口地址。
  • 目前实例 CRUD 不是模板内容级 CRUD，而是前端卡片记录 CRUD。
当前最大问题不是“iframe 是否优雅”，而是模板、运行时、实例三层概念混在一起：模板目录前端硬编码、实例没有后端持久化、模板运行时没有版本绑定，导致后续无法稳定支撑升级、回滚、继续编辑和多端同步。

2. 设计目标

目标 A：模板可长期扩展 新增模板、下线模板、灰度发布模板、统一管理模板元信息，不再依赖前端改常量。
目标 B：实例完整后端化 模板实例需要真正落库，支持创建、重命名、复制、删除、继续编辑、分享和统计。
目标 C：模板升级可控 模板升级后，老实例默认不被破坏；支持版本绑定、升级预览、复制升级、回滚。
目标 D：保持迁移可行 短期允许继续使用 iframe，但必须引入统一 bridge，避免未来重做。
目标 E：主项目职责清晰 主项目负责平台壳子、权限、实例管理与宿主交互，不再承担模板内部业务状态存储。
目标 F：支持多种运行时形态 统一协议下允许 iframe、远程模块、Web Component 并存，逐步演进而非一次推翻。

3. 核心设计原则

  1. 模板定义、模板运行时、模板实例必须拆开建模。
  2. 实例必须绑定运行时版本，不能默认跟随模板最新代码运行。
  3. 模板内容状态必须以快照 snapshot 的形式后端存储。
  4. 嵌入技术不是第一优先级，统一 bridge 协议才是第一优先级。
  5. 优先解决后端注册中心与实例数据模型，再逐步替换前端接入方式。

4. 目标总体架构

目标架构由 5 层组成：模板注册中心、模板运行时、模板实例中心、主项目宿主层、模板桥接层。

5. 核心领域模型

5.1 四个核心对象

对象 职责 示例
VisualTemplateRuntime 一套真正可执行的前端运行时，负责模板渲染与交互。 phys-template-mechanics
VisualTemplateRuntimeVersion 某个运行时的具体发布版本，带入口地址和桥接协议版本。 physics-runtime@1.4.2
VisualTemplateDefinition 用户在模板库中看到的模板卡片与业务定义。 P-01 受力分析器
VisualTemplateInstance 用户基于某个模板创建的实例，保存标题、快照、版本、缩略图等。 斜面受力分析课堂演示

5.2 为什么必须拆成四层

模板定义 ≠ 运行时入口 例如多个物理模板可能共享一个运行时入口，但属于不同模板定义。后端必须支持“多个模板定义指向同一个 Runtime”。
实例 ≠ 模板卡片 实例必须有自己的快照、缩略图、更新时间、版本绑定和权限，而不是模板卡片的简单拷贝。

6. 后端架构设计

6.1 后端职责

Template Registry
  • 维护模板目录、模板配置、模板分类、封面、标签、状态。
  • 管理模板与运行时的映射关系。
  • 决定模板库页面展示哪些模板。
Runtime Version Service
  • 记录运行时版本、入口地址、资产地址、发布时间。
  • 支持灰度、回滚、冻结旧版本、指定推荐版本。
  • 向实例服务提供升级检查与迁移能力。
Instance Service
  • 负责实例创建、读取、重命名、复制、删除、继续编辑。
  • 保存实例快照、缩略图、运行时版本绑定。
  • 提供实例搜索、筛选、排序、最近访问等能力。
Share / Assets / Analytics
  • 缩略图上传、分享链接、使用次数、行为埋点。
  • 支持模板使用统计与实例编辑统计。
  • 为后续模板运营和资源治理提供数据基础。

6.2 建议的数据表

表名 作用 关键字段
VisualTemplateRuntime 记录运行时的基础信息。 runtimeKey, entryType, currentVersionId, status
VisualTemplateRuntimeVersion 记录每次 runtime 发布版本。 version, entryUrl, bridgeVersion, snapshotSchemaVersion
VisualTemplateDefinition 记录模板目录定义。 templateKey, runtimeId, title, subject, defaultConfig
VisualTemplateInstance 记录用户实例。 templateId, runtimeVersionId, snapshot, thumbnailUrl, updatedAt
VisualTemplateMigration 记录快照迁移规则。 fromSchemaVersion, toSchemaVersion, handlerKey, isAutomatic
你们现在主项目里“模板实例”只保存元信息，没有保存模板内容快照。新的实例表必须把 snapshot 当作核心字段，否则“继续编辑”无法成立。

7. 前端主项目架构设计

7.1 主项目角色

  • 展示模板库与实例列表。
  • 负责模板实例 CRUD、搜索、筛选、排序。
  • 负责工作台壳子、顶部工具栏、保存状态、升级提醒。
  • 负责向后端读取模板目录、实例详情、版本状态。
  • 负责装载模板运行时，但不负责模板内部状态长期存储。

7.2 页面层职责

模板库页 调用 /templates 接口，展示模板目录与筛选条件。用户从这里选择模板进入创建流程。
实例列表页 调用 /instances 接口，展示用户实例。支持重命名、复制、删除、继续编辑。
工作台页 根据 instanceId 装载实例，并通过 bridge 与模板运行时通信，负责保存、升级提示和错误边界。

7.3 前端必须移除的临时实现

  • 不再以前端常量作为正式模板目录。
  • 不再以 localStorage 作为正式实例存储。
  • 不再把“生成前端 projectId 并写本地存储”当成实例创建流程。
localStorage 可以保留为临时缓存或未登录草稿缓存，但不能再作为正式实例的唯一来源。正式实例必须以服务端记录为准。

8. Template Bridge 协议设计

bridge 是整套方案的稳定内核。短期即使继续用 iframe，也必须按统一协议通信；长期替换为 remote module 时仍然沿用同一协议。

在当前团队现状下，不应试图统一所有模板的内部技术实现，也不应要求所有模板共享同一种业务数据结构。真正要统一的是 bridge 接口、snapshot 外层协议、版本迁移机制，而不是内部 store 细节。

8.1 宿主传入模板运行时的上下文

字段 说明
instanceId 当前实例 ID。
templateId 当前模板定义 ID。
runtimeVersion 当前实例绑定的运行时版本。
initialSnapshot 实例初始快照。
theme / locale / user 宿主级上下文。
assetBaseUrl / apiBaseUrl 模板运行时所需的资源与接口地址。

8.2 模板运行时对宿主暴露的能力

命令类接口
  • mount(container, context)
  • load(snapshot, config)
  • setReadonly(flag)
  • destroy()
读写类接口
  • getSnapshot()
  • export(payload)
  • validateSnapshot()

8.2.1 推荐的 Adapter 统一接口

推荐所有模板最终都适配为统一运行时接口 
interface TemplateSnapshotAdapter {
  loadSnapshot(snapshot: TemplateSnapshotDocument): Promise<void>;
  getSnapshot(): Promise<TemplateSnapshotDocument>;
  validateSnapshot(snapshot: unknown): Promise<ValidationResult>;
  getDirtyState(): Promise<{ dirty: boolean; reason?: string }>;
}

interface TemplateRuntimeBridge extends TemplateSnapshotAdapter {
  mount(container: HTMLElement, context: HostContext): Promise<void>;
  destroy(): Promise<void>;
}

8.3 模板运行时向宿主回传的事件

  • onDirtyChange：模板内部是否有未保存修改。
  • onSaveRequest：模板主动请求宿主保存。
  • onNavigate：模板内部页面变化或子路由变化。
  • onError：运行时错误、加载失败、导出失败。
  • onAnalyticsEvent：模板内部关键行为埋点。

9. 模板库配置与发布方式

9.1 模板库配置原则

  1. 模板目录由后端返回，前端只消费，不手写生产配置。
  2. 模板定义和运行时入口分开配置。
  3. 多个模板定义可以复用一个运行时。
  4. 模板卡片只展示模板定义，不直接暴露底层入口细节。

9.2 运行时发布流程

flowchart LR
    A[模板仓库代码变更]
    B[CI 构建运行时产物]
    C[发布到 CDN / 静态托管]
    D[后端登记 RuntimeVersion]
    E[更新 TemplateDefinition 推荐版本]
    F[主项目拉取模板目录]
    G[用户创建或继续编辑实例]

    A --> B --> C --> D --> E --> F --> G

9.3 接入形态建议

形态 适用阶段 适用对象 建议
iframe + bridge 短期主方案 现有绝大多数模板 先解决实例存储与协议问题，迁移成本最低。
remote ESM / 私有包 中期增强 适合深度集成的单页模板 在 bridge 稳定后逐步替换。
独立子应用 长期共存 像完整业务应用的复杂模板 保留独立部署能力，但纳入统一注册中心和实例系统。

10. 用户使用流程设计

10.1 从模板创建实例

flowchart TD
    U1[用户进入模板库]
    U2[选择模板]
    U3[前端请求模板详情]
    U4[后端返回模板定义 + 推荐运行时版本]
    U5[前端调用 POST /instances]
    U6[后端创建实例并绑定 runtimeVersion]
    U7[返回 instanceId]
    U8[跳转工作台]
    U9[工作台加载实例并装载模板]

    U1 --> U2 --> U3 --> U4 --> U5 --> U6 --> U7 --> U8 --> U9

10.2 继续编辑实例

  1. 用户进入“我的模板课件”实例列表。
  2. 主项目调用 GET /instances 获取实例列表。
  3. 用户点击“继续编辑”。
  4. 工作台根据 instanceId 拉取实例详情。
  5. 主项目按实例绑定的 runtimeVersionId 装载运行时。
  6. 主项目把实例快照传给运行时，恢复现场。

10.3 复制实例

  1. 用户在实例列表中点击“新建副本”。
  2. 前端调用 POST /instances/:id/duplicate。
  3. 后端复制实例元信息与快照，并生成新的实例 ID。
  4. 前端跳转到新实例工作台，用户继续编辑副本。

11. 实例数据存储设计

11.1 实例必须保存什么

类别 字段示例 说明
基础元信息 title, subject, thumbnailUrl 用于实例列表展示和搜索。
绑定信息 templateId, runtimeVersionId, snapshotSchemaVersion 保证继续编辑与版本回溯可用。
业务快照 snapshot 模板真实编辑状态，决定能否恢复现场。
审计/运营字段 createdAt, updatedAt, lastOpenedAt 用于排序、统计、历史与埋点。

11.2 快照设计建议

  • 优先使用 JSON 结构保存，避免依赖 DOM 或 HTML 字符串。
  • 每份快照必须携带 schemaVersion。
  • 模板内部应负责定义自己快照的最小可恢复结构。
  • 对于大文件或媒体资源，只在快照中保存引用，不直接塞入大体积二进制。
  • 对于随机模拟类模板，优先考虑保存 seed + params + engineVersion + resultSummary，而不是无条件保存全量试验结果。
实例快照不是“可选优化项”，而是“继续编辑”能力的基础。只存标题和模板编号并不构成实例系统。
Snapshot 不应等价于“内存 dump”。能通过参数、算法版本和随机种子重算出来的结果，应优先采用“可重放”设计，而不是把所有原始结果完整写入数据库。

11A. Snapshot 统一策略

当前模板由不同同学开发，进度、技术栈版本、功能复杂度差异很大。因此 snapshot 设计不能要求“所有模板共享同一业务数据结构”，而应采用 统一 envelope，放开 payload 的模式。

11A.1 统一什么，不统一什么

必须统一
  • snapshot 外层 envelope 结构
  • snapshot 生命周期接口
  • snapshot schema version 规则
  • migration 执行机制
  • dirty 状态与保存触发约定
可以不统一
  • 模板内部 store 形态
  • React 版本与局部依赖版本
  • 具体 payload 字段命名
  • 模板内部 UI 与交互细节
  • 模板内部模块组织方式

11A.2 推荐的 Snapshot 文档结构

平台统一 envelope，模板维护 payload 
{
  "envelope": {
    "instanceId": "ins_xxx",
    "templateKey": "p01",
    "runtimeKey": "physics-mechanics",
    "runtimeVersion": "1.4.2",
    "bridgeVersion": "1.0",
    "snapshotSchemaVersion": 3,
    "createdAt": "2026-04-09T10:00:00Z",
    "updatedAt": "2026-04-09T10:05:00Z",
    "persistMode": "full",
    "checksum": "sha256:xxx"
  },
  "payload": {},
  "uiState": {},
  "assets": []
}

11A.3 Snapshot 分层建议

层级 是否持久化 示例
domain state 必须 场景数据、参数配置、教学步骤、图元数据、自定义内容
uiState 按需 当前 tab、视角、缩放位置、当前激活面板
ephemeral state 不应 hover、loading、toast、拖拽中状态、临时请求 ID

11A.3.1 大体量结果的处理原则

Minimal 只保存最小编辑状态，不保存大结果。适合仅要求继续编辑、不要求完全一模一样的模板。
Deterministic 保存 seed + params + engineVersion，恢复时重新生成结果。适合随机模拟类模板，推荐优先使用。
Full 保存完整结果数组。仅在无法可靠重算、且必须精确逐点恢复时使用，不建议作为默认方案。
不建议直接把整个前端 store 全量序列化为 snapshot。不同模板代码质量和 store 设计差异较大，最稳妥的做法是由模板显式提取可持久化的 domain state。

11A.4 默认快照要求

  • 每个模板定义都必须提供 defaultSnapshot。
  • 实例创建时基于 defaultSnapshot 生成初始快照。
  • 默认快照也必须带 snapshotSchemaVersion。

11B. 异构模板团队协作策略

鉴于模板由不同同学开发，建议按接入能力分级，而不是要求所有模板一次性达到完全一致。

11B.1 模板接入等级

等级 能力 适用说明
L0 仅展示，不支持 snapshot 只能预览，不进入正式实例体系。
L1 支持 loadSnapshot/getSnapshot 进入正式实例体系的最低门槛。
L2 支持 dirty 检测与自动保存 可接入完整工作台体验。
L3 支持 schema version 与 migration 可安全参与模板升级与实例迁移。

11B.2 双模式要求

Standalone Mode 模板可独立运行，便于开发调试、演示部署、产品验收。
Embed Mode 模板接受宿主控制，使用宿主传入的实例、快照、权限和保存机制。
对复杂模板，例如已经带登录、工作台、路由和项目管理能力的模板，不应要求整个应用直接接入平台，而应拆出真正的编辑器核心，优先让 embed mode 支持 snapshot。

12. 模板升级与兼容策略

12.1 核心原则

老实例默认必须绑定旧运行时版本，不能在模板发布新版本后自动切到新代码。

12.2 三种升级策略

Pinned 实例永久绑定创建时的运行时版本。最稳，推荐默认策略。
Copy-on-upgrade 用户主动复制实例到新版本，保留老实例不动。最适合课件场景。
In-place migrate 仅在迁移脚本完全可靠时允许原地升级，需谨慎控制。

12.3 推荐默认策略

  1. 所有新实例默认使用创建时推荐的运行时版本。
  2. 模板新版本发布后，老实例保持旧版本。
  3. 工作台展示“有新版本可升级”提醒。
  4. 用户可选择“复制为新版本实例”。
  5. 只有通过迁移校验的模板，才允许“原地升级”。

12.4 Snapshot Migration 机制

推荐每个模板维护自己的 migration 链 
type SnapshotMigration = {
  templateKey: string;
  from: number;
  to: number;
  migrate: (oldPayload: unknown) => unknown;
};
  • 平台不理解模板 payload 的业务细节，只负责调度 migration。
  • 模板必须对自己的 snapshotSchemaVersion 负责。
  • 升级流程中可先跑 upgrade-preview，通过后再生成升级副本或执行原地升级。

    A[打开实例]
    B[检查实例 runtimeVersion]
    C[检查模板当前推荐版本]

13. API 设计草案

13.1 模板目录 API

  • GET /api/visual-center/templates：获取模板目录。
  • GET /api/visual-center/templates/:templateId：获取模板详情。
  • GET /api/visual-center/runtimes/:runtimeId：获取运行时详情。

13.2 实例 API

  • POST /api/visual-center/instances：创建实例。
  • GET /api/visual-center/instances：获取实例列表。
  • GET /api/visual-center/instances/:instanceId：获取实例详情。
  • PATCH /api/visual-center/instances/:instanceId：保存实例元信息或快照。
  • DELETE /api/visual-center/instances/:instanceId：删除实例。
  • POST /api/visual-center/instances/:instanceId/duplicate：复制实例。

13.3 版本与升级 API

  • GET /api/visual-center/instances/:instanceId/version-status：查询实例版本状态。
  • POST /api/visual-center/instances/:instanceId/upgrade-preview：查看升级预览。
  • POST /api/visual-center/instances/:instanceId/upgrade：执行升级。

14. 迁移路线

阶段 1：后端化模板目录与实例系统

  • 新增模板、运行时、实例相关数据表。
  • 把前端模板常量改为从 API 获取。
  • 把前端实例列表改为从 API 获取。
  • 保留 iframe，但引入 bridge 协议。

阶段 2：模板运行时支持快照协议

  • 给模板运行时补充 loadSnapshot/getSnapshot 能力。
  • 工作台支持手动保存、自动保存、离开提醒、继续编辑。
  • 实例真正从“卡片记录”升级为“可恢复作品”。
  • 定义模板接入等级，以 L1 作为进入正式实例系统的最低要求。

阶段 3：引入版本控制与升级能力

  • 实例创建时绑定 runtimeVersionId。
  • 模板发布新版本时，后端记录版本信息。
  • 工作台支持升级提醒和复制升级。
  • 对达到 L3 的模板引入 migration 机制。

阶段 4：按模板类型逐步升级嵌入方式

  • 保留复杂模板的独立子应用能力。
  • 把适合深度集成的模板逐步改为 remote ESM / 私有包。
  • 桥接协议保持不变，只替换运行时装载方式。
推荐迁移顺序是：先做后端实例与版本模型，再做 bridge，再逐步替换 iframe。不要反过来，否则很容易只是在“更换前端壳子”，而没有真正解决实例与升级问题。

15. 最终结论

对当前系统而言，最合理的宏观架构不是“把所有模板都直接改成组件库”或者“立即全量上模块联邦”，而是先把系统建设成一个 版本化模板平台：

  • 模板目录在后端注册和管理。
  • 实例在后端存储和 CRUD。
  • 实例保存真实编辑快照，支持继续编辑。
  • 实例绑定运行时版本，模板升级默认不破坏老实例。
  • 宿主与模板运行时通过统一 bridge 协议通信。
  • 短期继续兼容 iframe，长期逐步演进为更深度的嵌入方式。
这套架构既能兼容你们当前已有模板仓库和主项目现状，也能支撑后续模板越来越多、模板持续升级、实例持续沉淀的长期平台化目标。
