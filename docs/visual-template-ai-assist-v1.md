# 可视化模板 AI 辅助搭建方案（V1）

本文档用于收敛当前“老师进入模板后不易快速上手”的问题，并给出一版可落地的产品原型与最小技术方案。

本文档默认建立在以下现状之上：

- 模板通过 `iframe` 嵌入主平台
- 模板侧已逐步补齐最小 bridge / snapshot 能力
- 平台已经具备“进入模板入口”的能力
- 当前主要痛点不在“找模板”，而在“进入模板后如何快速搭到可用状态”

---

## 1. 问题定义

当前模板平台已经部署了较多模板，但教师上手成本仍然较高：

1. 不同模板复杂度差异很大
2. 有些模板适合快速点开即用，有些需要较多参数配置
3. 教师进入模板后，往往需要花时间理解按钮、模式、参数和场景
4. 模板虽然具备较强能力，但教师不一定能快速搭出自己想要的效果

一句话概括：

> 平台现在已经基本解决了“进入模板”的问题，但还没有解决“进入后快速获得想要效果”的问题。

---

## 2. 当前阶段判断

当前不建议继续只发散功能点，而建议先收敛两件事：

1. 产品原型
2. 最小技术方案

原因：

- 方向已经基本清晰
- 模板由不同同事开发，复杂度和可控性不统一
- 如果不先统一体验和技术边界，后面很容易出现平台、模板、AI 各做各的

因此，当前阶段的目标不是做“全功能 AI 助教”，而是先定义：

> 教师一句话需求，系统如何在现有模板能力范围内，快速生成一个可继续编辑的初始状态。

---

## 3. 本次方案的核心结论

### 3.1 不再重点投入“模板推荐型 AI”

平台侧已经打通“从模板库进入模板”的链路，因此：

- 关键词匹配 / 分词搜索已经能解决“怎么找到模板”
- 继续投入“只推荐模板”的能力，收益有限

真正值得解决的是：

> 进入模板后，如何减少教师从空白状态搭建到可用演示效果的时间。

### 3.2 V1 的主方向

V1 建议聚焦：

> 宿主侧 AI 根据教师自然语言需求，生成模板的结构化状态草稿，再通过模板 bridge 进行校验和加载。

也就是说：

- AI 不直接操纵 DOM
- AI 不直接模拟点击按钮
- AI 生成的是模板可理解的“状态草稿”
- 模板通过 `validateSnapshot` / `loadSnapshot` 接收这个状态

### 3.3 技术形态建议

优先采用：

- `iframe + bridge + snapshot`

而不是：

- 模板内直接接入复杂 Agent
- 各模板各自实现一套自然语言控制逻辑

原因：

- 和当前架构兼容
- 可控性更强
- 更容易逐模板渐进接入

---

## 4. 产品目标

V1 产品目标：

1. 教师输入自然语言需求
2. 系统基于现有模板能力生成一个“初始演示状态”
3. 模板加载这个状态
4. 教师在生成结果基础上继续编辑

非目标：

1. 不追求所有模板都支持同样强的 AI 能力
2. 不追求一步到位实现模板内多轮智能代理
3. 不追求替代教师全部操作

V1 重点是：

> 让教师从“空白进入模板”变成“带初始草稿进入模板”。

---

## 5. 用户体验原型（建议）

### 5.1 入口位置

建议在工作台壳子中提供一个统一的 AI 搭建入口，而不是每个模板各自定义入口。

可选位置：

1. 模板库页卡片上的“AI 帮我搭建”
2. 工作台进入后的右侧 AI 面板
3. 工作台顶部工具栏中的“快速生成”
4. 工作台右下角悬浮按钮 / 侧边悬浮小球

V1 推荐：

- 先放在工作台页
- 优先采用“悬浮按钮 / 小球 + 弹出对话框”的形态
- 因为很多 iframe 模板本身控件已经很多，固定面板容易挤占操作区
- 这种形态更容易承接“生成失败后继续手动编辑”的体验

### 5.2 典型流程

#### 流程 A：从模板库直接进入

1. 教师在模板库选择模板
2. 进入工作台
3. 点击“AI 帮我搭建”
4. 输入一句话需求
5. 系统生成初始状态
6. 模板加载
7. 教师继续编辑并保存

#### 流程 B：进入模板后再发起

1. 教师先进入某个模板
2. 发现功能较复杂
3. 点击工作台中的 AI 悬浮按钮 / 小球
4. 在弹出的对话框 / 抽屉中输入需求
5. 系统生成可加载状态
6. 模板恢复到目标草稿
7. 教师微调

### 5.2.1 推荐的原型形态

V1 推荐把 AI 助手设计成宿主层统一组件，而不是每个模板各自实现一套 AI 面板。

推荐形态：

1. 右下角悬浮按钮
2. 侧边悬浮小球
3. 点击后弹出对话框 / 抽屉

这样做的好处：

1. 不破坏模板本身布局
2. 对 iframe 模板最友好
3. 不需要每个模板重复适配一套 UI
4. 教师在不同模板中会形成统一使用习惯

### 5.3 关键交互状态

V1 原型至少应覆盖以下状态：

1. 空白待输入态
2. AI 生成中
3. 生成成功并已加载
4. 生成失败
5. 部分成功（只生成了部分效果）
6. 继续编辑
7. 保存成功

### 5.4 失败与降级体验

必须在原型中明确：

1. 未匹配到模板能力时怎么办
2. snapshot 校验失败时怎么办
3. 只生成部分内容时如何提示
4. 是否允许教师“应用部分草稿”

建议：

- 生成失败不阻断教师继续手动编辑
- 允许“先加载能通过校验的部分，再给出提示”
- 当 AI 生成结果在 `validateSnapshot` 阶段失败时，默认回滚到本次 AI 修改前状态
- AI 修改完成后默认只“应用”，不自动保存；保存动作仍由教师手动触发

## 6. 最小技术方案

### 6.1 总体调用链

建议链路如下：

1. 教师输入自然语言需求
2. 宿主读取当前模板能力描述
3. 宿主调用 iframe 内模板 `getSnapshot()` 获取当前状态
4. 宿主将“当前 snapshot + 模板能力描述 + 用户指令”发给 LLM
5. 宿主调用 LLM 生成结构化状态草稿
6. 宿主调用 iframe 内模板的 `validateSnapshot`
7. 校验通过后调用 `loadSnapshot`
8. 模板完成恢复
9. 教师继续编辑

### 6.2 为什么不直接让 AI 操作页面

不建议让 AI：

- 直接找按钮
- 直接点 DOM
- 直接按像素位置拖拽

原因：

1. 不稳定
2. 不可校验
3. 不利于继续编辑
4. 难以跨模板复用

V1 应坚持：

> AI 输出“状态”，模板负责“渲染状态”。

### 6.3 推荐的数据流

#### 宿主负责

- 悬浮按钮 / 小球 UI
- 对话框 / 抽屉交互
- 需求输入
- 模板能力收集
- 当前模板 snapshot 读取
- LLM 调用
- `validateSnapshot`
- `loadSnapshot`
- 错误与降级处理

#### 模板负责

- 状态结构定义
- `getSnapshot`
- `loadSnapshot`
- `validateSnapshot`
- 从状态恢复 UI

一句话：

> AI 助手的交互壳子由宿主统一开发，模板只复用统一 bridge / snapshot 协议。

### 6.4 推荐 AI 输出结构

V1 不建议直接让模型自由输出任意 JSON。  
建议模型输出一个宿主可控的结构，例如：

```json
{
  "templateKey": "c07",
  "intent": "build-demo",
  "confidence": 0.82,
  "missingInfo": [],
  "snapshotDraft": {
    "envelope": {},
    "payload": {}
  }
}
```

或者更保守的中间态：

```json
{
  "templateKey": "c07",
  "intent": "build-demo",
  "configDraft": {
    "reaction": "haber",
    "preset": "standard",
    "temperature": 725,
    "volume": 1.45,
    "catalyst": true
  }
}
```

然后由宿主将 `configDraft` 转换为 `snapshotDraft`。

### 6.4.1 当前状态参与生成

对于“进入模板后继续优化”的场景，LLM 不应只看老师最后一句话，还应看到：

1. 当前模板能力描述
2. 当前模板 snapshot
3. 用户新增指令

也就是说，AI 实际上更像是在：

> 基于当前状态修改 snapshot，而不是从零重新构造页面。

### 6.4.2 Snapshot 草稿 vs Snapshot Patch

V1 可以先支持“生成整份 snapshot 草稿”。

后续建议逐步演进到：

- 生成 `snapshot patch`

例如：

```json
{
  "templateKey": "c07",
  "patch": {
    "payload": {
      "controls": {
        "temperature": 725
      }
    }
  }
}
```

这样做的好处：

1. 更容易理解 AI 改了什么
2. 更利于做回滚与用户确认
3. 比每次重写整份 snapshot 更稳

### 6.4.3 当前建议：以 Patch 为主

结合当前讨论，V1 更倾向于：

> AI 输出 patch，宿主基于当前 snapshot 合成候选 snapshot，再校验与加载。

推荐流程：

1. 宿主调用 `getSnapshot()` 取得当前状态
2. 宿主把“当前 snapshot + 模板能力描述 + 用户指令”交给 LLM
3. LLM 输出 `snapshotPatch`
4. 宿主在前端将 patch 合并到当前 snapshot，得到候选 snapshot
5. 宿主调用 `validateSnapshot`
6. 若通过，则 `loadSnapshot`
7. 若失败，则回滚到 AI 修改前状态

### 6.5 V1 更推荐哪种

推荐采用折中方案：

> LLM 输出结构化配置草稿，宿主再转换成模板 snapshot。

原因：

1. 比直接输出 snapshot 更稳
2. 比只做模板推荐更有价值
3. 宿主可加入确定性转换逻辑
4. 降低模型胡乱生成无效字段的风险

### 6.5.1 Patch 合并层的职责

如果采用 patch 路线，宿主层需要承担一个明确职责：

> 将 AI 输出的 patch 应用到当前 snapshot，生成候选 snapshot。

这层建议放在宿主，而不是模板内部，原因：

1. 宿主更容易统一做日志、回滚、异常处理
2. 模板只需要负责“接受合法状态并恢复”
3. 不同模板可以共用统一 patch 合并与失败处理逻辑

### 6.6 不是每个模板都开发一套 AI 助手

这里需要明确：

- **不需要每个模板各自开发一个 AI 助手 UI**
- 需要统一开发的是宿主层 AI 助手
- 模板侧复用的是 bridge / snapshot 协议

也就是说：

#### 宿主统一复用

1. 悬浮按钮 / 小球
2. 对话框 / 抽屉
3. 指令输入
4. 当前 snapshot 读取
5. LLM 调用
6. `validateSnapshot + loadSnapshot`

#### 模板分别适配

1. `getSnapshot`
2. `validateSnapshot`
3. `loadSnapshot`
4. 模板能力描述

模板复用的是：

> 协议与接入方式

而不是：

> 各自再做一套 AI 面板

## 7. 模板能力分级

由于不同模板复杂度不统一，必须引入能力分级。

### L0

- 只能预览
- 不支持 AI 搭建

### L1

- 支持 `getSnapshot/loadSnapshot/validateSnapshot`
- 支持一次性 AI 生成初始状态

### L2

- 状态结构较完整
- 可支持生成后局部微调

### L3

- 暴露模板内动作级 API
- 可支持更强的模板内多轮助手

### 当前建议

V1 只要求：

- 进入正式能力范围的模板至少达到 `L1`

## 8. 模板能力描述层

建议为每个模板定义一份“能力描述”，供宿主和 LLM 使用。

最小字段建议：

```json
{
  "templateKey": "c07",
  "aiLevel": "L1",
  "supportsAiBuild": true,
  "supportsSnapshotDraft": true,
  "supportedIntents": [
    "build-demo",
    "restore-example"
  ],
  "capabilities": [
    "reaction-selection",
    "preset-selection",
    "temperature-control",
    "volume-control",
    "catalyst-toggle"
  ],
  "constraints": [
    "history is summarized",
    "animation frame is not restorable"
  ]
}
```

这层能力描述的作用：

1. 给 LLM 提供边界
2. 给宿主做调度
3. 给产品做模板 AI 能力展示

### 8.1 这本质上像每个模板的“私有知识说明书”

从产品角度可以把这层理解成：

> 每个模板都要有一份自己的“可操作说明书”给 AI 看。

如果不给 LLM 这层说明，就容易出现幻觉：

- 生成模板根本不支持的字段
- 要求切换不存在的模式
- 调整不存在的参数
- 误解模板适用场景

所以这层能力描述几乎是必须的。

### 8.2 V1 不建议一开始做成重型 RAG

当前更实际的做法是先做：

1. 结构化能力描述（机器可读）
2. 少量示例（few-shot）
3. 字段约束与不可恢复说明

因此 V1 更像：

- 模板能力清单
- 模板可控字段说明
- patch / snapshot 示例

而不是复杂检索式知识库。

### 8.3 是否要求所有模板统一结构

不建议强行要求所有模板的内部 payload 完全一致。  
但建议统一一层“AI-facing schema”，例如：

- 支持哪些意图
- 支持哪些字段改动
- 字段值类型与范围
- 哪些字段是高风险字段

## 9. 和当前 iframe 架构的关系

### 9.1 现有架构可继续沿用

当前平台已经是：

- 主平台壳子
- `iframe` 嵌入模板
- bridge 与 snapshot

这套模式完全可以承接 V1。

### 9.2 不需要现在重构掉 iframe

V1 的重点不是替换嵌入方式，而是：

- 把“自然语言 -> 模板状态”链路跑通

因此：

- `iframe` 仍然可继续使用
- bridge 仍然是核心
- snapshot 仍然是最重要的状态协议

### 9.3 当前方案在 iframe 场景下的优势

这套方案尤其适合你们当前的 iframe 架构：

1. 宿主统一承载 AI 入口，不需要侵入模板内部布局
2. 宿主可统一读取当前模板 snapshot
3. 宿主可统一处理校验、超时、失败提示和降级
4. 模板只负责“接受状态并恢复状态”，不负责复杂对话流程

因此，即使不同模板复杂度不统一，也可以先在宿主层提供一致的 AI 使用体验。

### 9.4 AI 助手属于工作台专属能力

当前建议把 AI 助手视为：

- 工作台页专属能力

而不是：

- 模板库全局能力
- 各模板内部自带组件

原因：

1. 它的目标是“进入模板后帮助搭建”
2. 它和当前 iframe、snapshot、继续编辑链路天然绑定
3. 工作台是最容易统一承接 AI 状态、失败提示和回滚逻辑的地方

## 10. 失败处理与降级策略

V1 必须内建以下降级能力：

### 情况 1：模板能力不支持

处理方式：

- 提示“当前模板暂不支持 AI 搭建”
- 允许继续手动编辑

### 情况 2：生成草稿不完整

处理方式：

- 加载已通过校验的部分
- 给出“部分内容需手动补充”的说明

### 情况 3：`validateSnapshot` 不通过

处理方式：

1. 宿主先尝试自动修正一次
2. 再校验
3. 仍失败则不加载，提示教师

### 情况 4：模板加载报错

处理方式：

- 保持当前模板不变
- 展示错误信息
- 不丢失教师当前已编辑内容

### 情况 5：AI 修改结果校验通过，但效果不符合预期

这种情况不一定是技术错误，而是生成结果和教师心智预期不一致。

建议处理方式：

1. 教师可以继续发出下一轮修改指令
2. AI 在当前 snapshot 基础上继续 patch
3. 必要时教师可以回退到 AI 修改前状态

## 10A. 当前讨论后的关键决策记录

以下内容用于记录当前已经收敛出的重要决策与待定项，便于后续技术方案继续细化。

### 10A.1 已明确

1. AI 助手入口优先放在工作台页
2. 入口形态优先采用可拖动的悬浮球 / 小球
3. V1 要支持连续微调，不是只支持一次性生成
4. AI 修改后默认只“应用”，不自动保存
5. 当 `validateSnapshot` 失败时，应回退到本次 AI 修改前状态
6. AI 助手由宿主统一开发，模板不重复开发 AI UI
7. 后续多轮优化应基于当前 snapshot 继续 patch

### 10A.2 当前倾向

1. AI 输出形式更倾向于 patch，而不是整份 snapshot
2. 模板需要一份自己的能力描述，相当于模板私有知识说明书
3. V1 的知识层更适合先做结构化能力描述，而不是重型 RAG
4. AI 助手更适合作为工作台专属能力

### 10A.3 尚待最终确定

1. LLM 调用放前端还是后端
2. 是否记录 AI 行为与生成历史
3. patch 合并层是否完全放在前端宿主

目前对第 1 点的初步看法是：

- 前端调用更轻、更快、交互更丝滑
- 后端代理更利于密钥安全、审计与行为分析

这一点建议后续结合真实前后端代码结构再定。

## 12. 推荐推进顺序

### 阶段 1：方案收敛

## 14. 基于现有前后端代码的接入建议

本节基于当前真实代码结构补充说明 AI 助手最适合接在哪一层，以及哪些现有能力可以直接复用。

### 14.1 前端现状判断

前端已经具备完整的模板平台骨架，关键位置如下：

- 模板库页：
  - `app/(workbench)/visualize/templates/page.tsx`
  - `components/visual-center/VisualTemplatesClient.tsx`
- 历史实例页：
  - `app/(workbench)/visualize/history/page.tsx`
- 工作台页：
  - `app/(workbench)/visualize/workbench/[templateId]/page.tsx`
- 前端 API 封装：
  - `lib/visual-center/api.ts`
- 宿主 bridge client：
  - `lib/visual-center/bridge.ts`

从现有实现可以确认：

1. 模板库页已经支持创建实例并跳转工作台
2. 工作台页已经拿到：
   - 当前模板信息
   - 当前实例信息
   - iframe 引用
3. 工作台页已经具备：
   - `getSnapshot`
   - `loadSnapshot`
   - 手动保存 snapshot

因此，AI 助手**最合适的宿主接入点**已经非常明确：

> `app/(workbench)/visualize/workbench/[templateId]/page.tsx`

### 14.2 后端现状判断

后端已经具备模板定义与实例能力，关键位置如下：

- 数据模型：
  - `app/models/visual_template.py`
- Schema：
  - `app/schemas/visual_template.py`
- 视觉模板接口：
  - `app/api/v1/visual_center/route.py`
- Migration / seed：
  - `alembic/versions/d4f8a1c2b3e4_add_visual_template_platform_tables.py`

后端当前已经支持：

1. 模板目录读取
2. 模板详情读取
3. 模板实例创建
4. 模板实例详情读取
5. 模板实例更新
6. 模板实例复制
7. 模板实例删除

而且模型里已经存在：

- `default_snapshot`
- `entry_url`
- `snapshot`
- `snapshot_schema_version`
- `capabilities`

这意味着：

> AI 助手不是要从零设计一套实例保存系统，而是建立在现有模板定义和实例 CRUD 之上。

### 14.3 当前最适合的分层

基于现有代码，推荐采用以下分层：

#### 宿主前端工作台负责

1. 悬浮球 / 小球 UI
2. 对话框 / 抽屉交互
3. 从 iframe 获取当前 snapshot
4. 调用后端 AI 接口
5. patch 合并
6. `validateSnapshot`
7. `loadSnapshot`
8. 失败回滚
9. 最终保存仍通过现有实例更新接口完成

#### 后端负责

1. 接收 AI 助手请求
2. 拼装模板能力描述
3. 调用 LLM
4. 返回：
   - `configDraft`
   - 或 `snapshotPatch`
   - 以及可选 explanation / warning

#### 模板负责

1. `getSnapshot`
2. `validateSnapshot`
3. `loadSnapshot`

### 14.4 为什么 AI 助手不建议直接放模板内部

虽然技术上可以让模板自己接一个对话框，但基于当前代码结构，不推荐这样做。

原因：

1. 工作台页已经是 iframe 宿主层，最适合统一挂载 UI
2. 模板内部布局复杂度差异很大
3. 如果每个模板自己做 AI 面板，后期维护成本会很高
4. 当前保存、继续编辑、实例更新都在宿主工作台链路上，统一处理更自然

一句话：

> AI 助手应属于工作台，而不是模板本体。

### 14.5 前端 BFF 现状与建议

前端当前已经有一层 `app/api/visual-center/...` 的 BFF / 代理路由。

这一层现在已经代理：

- `/api/visual-center/templates`
- `/api/visual-center/instances`

因此 V1 很适合继续沿用这个模式，新增一条 AI 助手接口，例如：

```text
/api/visual-center/assist
```

它的作用是：

1. 前端工作台调用统一入口
2. 可以复用当前鉴权方式
3. 可以灵活切换 mock / backend
4. 后续更容易埋点和限流

### 14.6 最推荐的 V1 请求链路

结合现有代码，推荐链路如下：

1. 教师在工作台页点击 AI 悬浮球
2. 输入自然语言指令
3. 工作台从 iframe 调 `getSnapshot`
4. 工作台调用 `/api/visual-center/assist`
5. BFF 转发到后端 `/api/v1/visual-center/...`
6. 后端结合模板能力描述调用 LLM
7. 后端返回：
   - `patch`
   - 或 `configDraft`
   - 以及 explanation / warning
8. 工作台本地合并 patch
9. 工作台调 `validateSnapshot`
10. 通过后调 `loadSnapshot`
11. 失败则回滚到 AI 修改前状态
12. 教师满意后，点击现有保存按钮保存实例

## 15. 仍需继续确认的实现细节

虽然接入位置已经比较清晰，但正式开发前，仍有几项细节需要继续收敛。

### 15.1 模板 key / entry_url / 能力描述统一

从后端 seed 代码看，当前数据库中的模板 key 和模板侧实际使用的 key 还没有完全统一。

例如：

- 模板侧可能使用 `c03`
- 后端 seed 仍是 `chem03`

这意味着在正式接入 AI 之前，必须先统一：

1. 模板 key
2. entry_url
3. snapshot schema version
4. capabilities 结构

否则 AI 助手链路会出现：

- 模板选对了
- key 不一致
- 结果调错模板或 entry_url

### 15.2 LLM 调用放前端还是后端

基于现有代码，我当前倾向于：

- **V1 放后端**

原因：

1. 你们后端已具备模板定义与实例模型
2. 更适合做模板能力描述拼装
3. 更适合做日志、审计和行为分析
4. 不暴露模型密钥

但这仍然需要后续结合你们的部署、鉴权和调用成本进一步确认。

### 15.3 patch 合并层放哪

基于现有代码，我当前倾向于：

- **放前端工作台宿主层**

原因：

1. 前端已经能拿到当前 snapshot
2. 前端已经有 `validateSnapshot/loadSnapshot`
3. 前端最容易做“本次 AI 修改前状态”的本地回滚

### 15.4 是否记录 AI 行为日志

当前虽然还没定，但我建议至少记录最小日志：

1. `templateKey`
2. `instanceId`
3. 指令摘要
4. 是否 validate 成功
5. 是否 load 成功
6. 是否最终被用户手动保存

这会对后续优化非常有帮助。
